# Feature 0024 — Habitica integration (unified external-tasks sidebar)

GitHub Issue #73. Add a read-only Habitica task integration (todos + dailies)
that lives in a **single, unified left-rail panel** alongside the existing
Todoist section. The new `habitica_sync` Django app mirrors the `todoist_sync`
stack (service-boundary decryption, Fernet at rest, versioned per-(user,date)
cache, system checks E001/W001). The frontend refactors the Todoist-specific
left sidebar into a source-agnostic `ExternalTasksSidebar` with per-source
sections, replaces the Todoist-only background poll with a unified
`useExternalTasksPoll`, and adds a Habitica connect card in Settings. Complete
(one-way, no un-complete) is supported for both Habitica todos and dailies via
Habitica's `score/up` endpoint, mirroring the Todoist optimistic-complete UX.

**No AI provider-egress**: Habitica task text is display-only, never fed into
draft/command/chat prompts (same stance as Todoist feature 0020).

Work lands in three phases (Phase 0, Phase A, Phase B) that **can ship as
separate PRs** but are documented together. Phase 0 (frontend refactor + poll
rename) is a prerequisite for cleanly slotting Habitica in; Phase A/B build the
`habitica_sync` app on top of it.

## User-locked decisions (honor verbatim)

1. This one plan doc covers all phases 0 + A + B; PRs may split per phase.
2. **Dailies** are shown ONLY when viewing **today** — where "today" is the
   **client's** today: `is_project_today(target_date) OR
   include_overdue_carryover`. This reuses the same client-today signal the
   todos filter already uses (`carry_overdue=1` bridges the browser-local vs
   server `localdate()` UTC gap), so a user ahead of UTC does **not** silently
   lose dailies on their browser-local today. Gate on Habitica's computed
   `isDue`; past/future dates show todos only.
3. **Todos filter**: dated todos match their `date` (due date) + overdue todos
   carried to today (mirror Todoist `carry_overdue=1`); **undated** todos show
   on **today only**.
4. `x-client` header value is built as `"{HABITICA_CLIENT_ID}-DayForge"` where
   `HABITICA_CLIENT_ID` is the maintainer's Habitica user ID, from env, **no
   default**; a startup check blocks boot when required (see § habitica.E001).
5. Phase B **complete** covers **todos AND dailies** via
   `POST /api/v3/tasks/{id}/score/up`, one-way (no un-complete); streak/XP side
   effects are accepted.
6. **Remove** `TODOIST_POLL_INTERVAL_SECONDS` + `useTodoistPoll` +
   `todoist_poll_interval` prop **outright** in Phase 0 (no deprecation
   fallback). Replaced by `EXTERNAL_TASKS_POLL_INTERVAL_SECONDS` (default `10`,
   `0` = disabled) + `external_tasks_poll_interval` prop + `useExternalTasksPoll`
   composable; update `deployment/` env in the same PR.

   > **AMENDED at implementation time — default is `60`, not `10`.** The poll
   > calls `refreshTasks`, which sends `refresh=1` and bypasses the server
   > cache on every tick, and Habitica costs **two** upstream calls per tick
   > (`type=todos` + `type=dailys`). At `10` that is 12 calls/min per open tab
   > against Habitica's ~30 req/min per-user limit — two tabs came close,
   > three exceeded it, and there is no client-side backoff to recover from
   > the resulting 429s. The rejected alternative was dropping `refresh=1`
   > from the poll, which would have capped freshness at the 300s cache TTL
   > and made polling largely pointless. Decision approved by the user during
   > the implementation review.
7. **NO collapsible sections** in V1 — static section headers with task counts.
8. Deploy updates for `dayforge.habitreward.org`: `HABITICA_ENCRYPTION_KEY` +
   `HABITICA_CLIENT_ID` in `deployment/` files + `.env.example` + Fernet
   key-generation instructions (the user runs server commands themselves — no
   SSH from tooling).

**Out of scope V1**: OAuth, task create/edit from Day Forge, habits/rewards, AI
coupling, webhooks, separate sidebars per app, collapsible sections, un-complete.

## Habitica API v3 facts

- Base URL `https://habitica.com/api/v3` — parameterise as `HABITICA_BASE_URL`.
  **No trailing slash** (matches the settings default at §Config and the
  `f"{HABITICA_BASE_URL}/tasks/..."` join in `service.py`; a trailing slash
  would produce `//tasks/...`).
- **Per-user auth headers**: `x-api-user` (Habitica user ID) + `x-api-key`
  (API token). **Plus** a required global `x-client` header
  (`"{HABITICA_CLIENT_ID}-DayForge"`).
- **List todos**: `GET /tasks/user?type=todos`.
- **List dailies**: `GET /tasks/user?type=dailys` — note Habitica spells the
  query value **`dailys`** (not `dailies`).
- **Complete**: `POST /tasks/{id}/score/up` with `Content-Type: application/json`.
  Returns **HTTP 200 with a JSON body** (delta, XP/HP/streak data) — **not** the
  `204 No Content` that Todoist's `/close` returns. The body is **ignored**
  (bare ack returned to the client). Re-scoring an already-completed
  todo/daily is a **benign 200** (Habitica applies another score, a
  streak/XP side-effect accepted under decision #5 — not a bug). **Scope
  bound**: V1 syncs only the authenticated user's **personal** todos + dailies
  (`GET /tasks/user`). Group/party/challenge **approval-gated** tasks
  (`approval.required`, which can yield a `202`/`requiresApproval` "not yet
  completed" response) are **out of scope** (habits/rewards/group already
  excluded), so the body-agnostic `_raise_for_status` acking every 2xx is
  correct here — a personal todo/daily never returns approval-pending. If a
  future group-task phase lands, revisit the ack to inspect the body.
- Response envelope is `{ "success": bool, "data": [...] }`; non-2xx also carries
  `{ "success": false, "error": ..., "message": ... }`.
- **Dailies** carry `isDue` + `completed` booleans computed by Habitica for
  today (relative to the user's custom day-start).
- **Todos** carry an optional `date` field (due date, ISO string or `null`) and
  a `completed` boolean.

## Phase A — habitica_sync backend (read-only) + Settings connect + section

### A.1 Data layer — `backend/habitica_sync/`

**`models.py` — `HabiticaAccount`** (mirror `TodoistAccount`, `OneToOne` per
user): fields `user` (OneToOne, `related_name="habitica_account"`),
`api_user_id` (`CharField`, the Habitica user ID for the `x-api-user` header —
**not** secret, stored plaintext), `api_token_encrypted` (`BinaryField`, Fernet
ciphertext — never serialised/logged/admin-rendered), `last_verified_at`,
`created_at` (`auto_now_add`), `updated_at` (`auto_now`). Methods
`get_token()` / `set_token()` copy `TodoistAccount` verbatim including the
`auto_now` footgun docstring. `set_token` uses `habitica_sync.crypto`.

**`crypto.py`** — copy `todoist_sync/crypto.py`, keyed by
`settings.HABITICA_ENCRYPTION_KEY`; `_fernet()` raises `ImproperlyConfigured`
when unset/malformed; `decrypt_token` maps `InvalidToken → ImproperlyConfigured`
(key-rotation surfaces as config error).

**`migrations/0001_initial.py`** — `CreateModel` for `HabiticaAccount` (generate
via `makemigrations habitica_sync`; the plan does not hand-write it).

**`apps.py`** — `HabiticaSyncConfig` with `ready()` importing `checks`.

**`admin.py`** — mirror `TodoistAccountAdmin`: `list_display`
(`user`, `api_user_id`, `last_verified_at`, `created_at`, `updated_at`),
readonly, no add/change; **omit `api_token_encrypted`** from `fields`.

**Register app**: add `"habitica_sync"` to `INSTALLED_APPS` in `settings.py`.

### A.2 `schemas.py`

`@dataclass(frozen=True) NormalizedHabiticaTask` with fields:
`id` (str), `title` (str, from Habitica `text`), `type` (str: `"todo"` |
`"daily"`), `due_date` (str ISO `YYYY-MM-DD` | `None`, todos only),
`completed` (bool). `normalized_task_to_dict(task)` serialiser.
`validate_account_payload(data)` returning `(cleaned, errors)` validating a
required `api_user_id` (str, non-empty, length cap) and `api_token` (str,
non-empty, length cap ~128) — mirrors Todoist's, but two fields.

### A.3 `service.py` — Habitica REST client (service-boundary owns the secret)

Typed exception hierarchy mirroring Todoist: `HabiticaError`,
`HabiticaAuthError` (401/403), `HabiticaTimeoutError`, `HabiticaProviderError`.

**Headers helper** `_headers(api_user_id, token)` → `{"x-api-user":
api_user_id, "x-api-key": token, "x-client": f"{settings.HABITICA_CLIENT_ID}-
DayForge", "Content-Type": "application/json"}`. **Never log the token.**
`_raise_for_status` maps 401/403 → auth, other non-2xx → provider.

**`is_project_today(target_date)`** — single source of truth
(`target_date == django_tz.localdate()`), shared with `cache.tasks_filter_scope`
(prevents cache/filter drift; same pattern as Todoist).

**`verify_credentials(api_user_id, token)`** — one cheap authenticated probe,
e.g. `GET /tasks/user?type=todos` (or `/user`); raise typed error on
401/403/timeout; return `None` on success. Never logs the token.

**`fetch_tasks_for_date(account, target_date, *, include_overdue_carryover=
False)`** — the **first** of two token-decryption call sites. Steps:

1. `token = account.get_token()`; build headers.
1b. **Unwrap the envelope** for every list response: Habitica wraps results as
   `{ "success": bool, "data": [...] }`. Read the task array from
   `payload["data"]` — **NOT** Todoist's `payload["results"]` key (blindly
   mirroring Todoist's `payload.get("results", [])` would silently yield an
   empty list in prod, `todoist_sync/service.py:210`). Validate the 2xx
   envelope before iterating: treat `success is False` or a non-list `data`
   as a malformed response → `HabiticaProviderError` (mapped to 502), so a
   `200`-with-`success:false` body is never cached as "zero tasks".
2. **Todos**: `GET /tasks/user?type=todos`. Normalise each raw todo →
   `NormalizedHabiticaTask(type="todo")` mapping `text → title`,
   `date → due_date` (see A.4 date normalisation — `due_date` is an ISO
   `str | None`), `completed`. **Filter** to the selected date per decision #3.
   All comparisons are **string-vs-string** and pinned to **`target_date`** (the
   client's today — NOT `django_tz.localdate()`): compare `due_date` (str)
   against `target_date.isoformat()`, matching the schema's `str` type, the
   cache-key `date_iso` component, and Todoist's `f"{target_date.isoformat()} |
   overdue"` server query (`todoist_sync/service.py:182`, which pins overdue to
   the requested date, not the server day):
   - `is_project_today(target_date)` OR `include_overdue_carryover` → include
     todos with `due_date == target_date.isoformat()`, PLUS **overdue** todos
     (`due_date is not None and due_date < target_date.isoformat()`, uncompleted
     — carryover; ISO `YYYY-MM-DD` strings sort lexicographically =
     chronologically), PLUS **undated** todos (`due_date is None`). **Both the
     exact-match and the overdue boundary use `target_date`** — using
     `localdate()` for the overdue boundary would drop a todo due on the server
     day when a non-UTC client sends `carry_overdue=1` with
     `target_date == localdate()+1` (the same TZ gap the dailies gate fixes).
   - else (past/future date) → include only todos with
     `due_date == target_date.isoformat()` (**no undated, no overdue carry**).
3. **Dailies**: fetched + shown **only when
   `is_project_today(target_date) or include_overdue_carryover`** (decision #2
   — the client-today signal, NOT bare `is_project_today`, so a non-UTC user's
   browser-local today still shows dailies; matches the `"with_overdue"` cache
   scope in A.5). `GET /tasks/user?type=dailys`; keep only dailies with
   `isDue == True`; normalise → `NormalizedHabiticaTask(type="daily",
   due_date=None, completed=<Habitica completed>)`. For a non-client-today date,
   skip the dailies fetch entirely.
4. Skip already-`completed` todos/dailies from the returned list (the panel
   shows outstanding work) — OR surface `completed` and let the UI filter; keep
   consistent with Todoist (Todoist list excludes closed tasks, so **exclude
   completed** here for parity).
5. Per-task normalisation wrapped in `try/except (KeyError, ValueError,
   TypeError)` → `logger.warning(...); continue` (one bad task must not fail the
   whole fetch; bug-class errors propagate).
6. Deterministic sort so the UI doesn't shuffle: e.g.
   `(-int(is_daily), due_date or "", title.casefold(), id)` (dailies first, then
   dated todos by date, then title). Pick one order and keep it stable.
7. Error wrapping mirrors Todoist: catch typed errors and re-raise;
   `requests.Timeout → HabiticaTimeoutError`; `requests.RequestException /
   Exception → HabiticaProviderError`; let `ImproperlyConfigured` (key rotation)
   **propagate** so the view maps it to a config-shaped 500; `finally: del token`.

**`complete_task(account, task_id)`** — the **second** token-decryption call
site. `POST {HABITICA_BASE_URL}/tasks/{task_id}/score/up` with the JSON
`Content-Type` header **and an explicit empty JSON body** (`requests.post(...,
json={})`) — the headers helper sets `Content-Type: application/json`, so a
bodyless POST can trip a JSON-parse `400`; send `{}`. Unlike Todoist's
`complete_task` (which POSTs no body to `/close`), Habitica needs the empty
`{}` because the `application/json` content-type is always present.
`_raise_for_status`. Same typed-error wrapping and
`del token` finally as Todoist's `complete_task`. Works for **both** todos and
dailies (decision #5). One-way (no un-complete). `task_id` is opaque (Habitica
UUID), built straight into the URL path (`<str:task_id>` converter is `[^/]+`).
Note `score/up` returns **200 with a JSON body** (ignored — see § Habitica API
v3 facts), so the body-agnostic `_raise_for_status` reused from Todoist is
correct; a test author must assert **200** here, **not** Todoist's `204`.

### A.4 Date normalisation (`service._normalize_due_date`)

Habitica todo `date` is an ISO string (possibly a full datetime with a `T`
component or `Z`/offset suffix) or `null`. Mirror Todoist's suffix-agnostic
`raw[:10]` slice **and its `.isoformat()` return so `due_date` is always an ISO
`str | None`** (matching §A.2's schema field and every downstream string
comparison — the A.3 filter, the A.5 cache scope, and `normalized_task_to_dict`
JSON serialisation all treat `due_date` as a `str`):
`null → None`; `"T" in raw → date.fromisoformat(raw[:10]).isoformat()`;
else `date.fromisoformat(raw).isoformat()`. The `date.fromisoformat(...)` parse
still validates the input (a malformed `date` raises `ValueError`, caught by the
per-task `try/except` in A.3 step 5); the `.isoformat()` normalises it back to a
canonical `YYYY-MM-DD` string. Drop the time component (date-only display).

### A.5 `cache.py` — versioned per-(user,date,scope) cache

Copy `todoist_sync/cache.py` verbatim with `_KEY_PREFIX = "habitica_tasks"`.
Key: `habitica_tasks:{user_id}:{account.updated_at.isoformat()}:{date_iso}:
{filter_scope}`. `tasks_filter_scope(target_date, *, include_overdue_carryover)`
returns `"with_overdue"` when `include_overdue_carryover or
is_project_today(target_date)` else `"exact"` (must match the filter the fetch
used). `get_cached_tasks` / `set_cached_tasks` (TTL =
`settings.HABITICA_CACHE_TTL_SECONDS`). `invalidate_tasks(account)` uses
`HabiticaAccount.objects.filter(pk=...).update(updated_at=now)` +
`account.updated_at = now` (the `auto_now`-bypass footgun applies — set
`updated_at` explicitly). Called after a successful complete.

**Cache-key unification / no cross-scope leakage** (parity note): the
two-scope key (`"with_overdue"` / `"exact"`) is retained **verbatim** from
Todoist so the scheme stays parity-correct. Divergence from Todoist:
Habitica fetches the FULL todo list (`GET /tasks/user?type=todos`) and
filters client-side, whereas Todoist pushes the date filter server-side —
but the cache stores the **already-filtered, normalised** task list under a
single `(date, scope)` key, so the client-side-vs-server-side fetch
difference never touches the key scheme. Both a `carry_overdue=1` read and a
plain today read resolve to `is_project_today == True → "with_overdue"` and
correctly share one key; a future-date `"exact"` read and a today
`"with_overdue"` read carry different `date_iso` components and can never
collide. Dailies are fetched **only** when
`is_project_today or include_overdue_carryover` (the client-today signal,
decision #2) — which is **exactly** the condition that makes
`filter_scope == "with_overdue"`, so a daily can only ever land in a
`"with_overdue"` entry and never in an `"exact"` one. Todos and dailies live
in the **same unified `(date, scope)` cache entry**; no daily ever leaks into
an `"exact"` entry.

### A.6 `views.py` + route registration (three paths / five handlers, mirror Todoist)

All non-2xx use the `{"errors": {"detail": ...}}` envelope. Map typed errors:
`HabiticaAuthError → 401`, `HabiticaTimeoutError → 504`,
`HabiticaProviderError → 502`; unknown → logged 500; `ImproperlyConfigured` →
config-shaped 500 (`"Habitica service is misconfigured. Contact the
administrator."`).

- `GET/POST/DELETE /api/habitica/account/` (`account` view):
  - GET → `_account_status_payload` `{connected, last_verified_at, api_user_id}`
    (**never** a token field). **Firm decision**: the payload diverges from the
    strict Todoist `{connected, last_verified_at}` shape by adding
    `api_user_id`. Rationale — unlike Todoist (one secret token, no non-secret
    id to echo), Habitica has a **non-secret** User ID that the connected
    Settings card should display so the user can confirm *which* account is
    connected (and pre-fill on reconnect/edit). `api_user_id` is plaintext at
    rest and safe to return; the token is never in the payload. When
    disconnected, `api_user_id` is `null`.
  - DELETE → idempotent `HabiticaAccount.objects.filter(user=...).delete()`.
  - POST → `reject_oversized_body` guard; `json.loads`; `validate_account_payload`;
    `service.verify_credentials(api_user_id, api_token)`; then in
    `transaction.atomic()` `select_for_update().get_or_create(user=...)`, set
    `api_user_id`, `set_token(api_token)`, `last_verified_at = now()`, plain
    `acc.save()` (fires `auto_now`, rotates cache key).
- `GET /api/habitica/tasks/<str:date>/` (`tasks` view): parse date (400 on
  bad format); resolve account (503 if `DoesNotExist`); read `carry_overdue=1`
  + `refresh=1` query flags; compute `filter_scope`; serve from cache unless
  `refresh=1`; else `service.fetch_tasks_for_date(...)` (catch
  `ImproperlyConfigured` → 500 config; typed errors → mapped status); serialise
  `normalized_task_to_dict`; `set_cached_tasks`; return `{"tasks": [...]}`.
- `POST /api/habitica/tasks/<str:task_id>/complete/` (`complete` view): resolve
  account (503); `service.complete_task`; catch `ImproperlyConfigured` → 500,
  typed → mapped; `cache.invalidate_tasks(account)`; return `{"ok": True}`. No
  request body parsed (id is in the path).

**Routes** (register **directly** in `backend/day_forge/urls.py`, mirroring the
Todoist block at `:117-133` — no `habitica_sync/urls.py`, no `include()`):
`import habitica_sync.views as habitica_views`, then three `path(...)` entries —
`api/habitica/account/` → `habitica_views.account`,
`api/habitica/tasks/<str:date>/` → `habitica_views.tasks`,
`api/habitica/tasks/<str:task_id>/complete/` → `habitica_views.complete`.

### A.7 `checks.py` — system checks

- **`habitica_sync.E001`** — blocks `DEBUG=False` boot when
  `HABITICA_ENCRYPTION_KEY` **OR** `HABITICA_CLIENT_ID` is unset/malformed
  (encryption key validated via `Fernet(key)`; client id validated as
  non-empty). **Settings-only, no DB query** — so the two Habitica secrets form
  **one boot gate** (decision #4): a `DEBUG=False` deploy must set both even if
  no user has connected Habitica yet, exactly as the encryption key already
  requires. Under `DEBUG=True`, neither is required (dev). The `x-client` header
  is required by Habitica, so a missing client id would make every request fail;
  gating it on `DEBUG=False` (not on "a `HabiticaAccount` exists") keeps the
  check DB-free and matches the encryption-key branch — same pattern as
  `todoist_sync.E001`.
- **`habitica_sync.W001`** — warn (not block) when `CACHES['default']` is an
  ineffective backend (LocMem/FileBased/Dummy) while ≥1 `HabiticaAccount`
  exists and `not DEBUG`; versioned keys keep correctness (perf-only concern).
  Copy the Todoist W001 body + the `try/except OperationalError/ProgrammingError`
  pre-migrate guard.

## Test coverage (mirror `backend/tests/` Todoist suite)

- `service`: header shape incl. `x-client`; `dailys` vs `dailies` spelling;
  todos filter (today+overdue+undated vs exact-date-only); dailies today-only +
  `isDue` gate; completed-task exclusion; date normalisation (datetime suffix,
  null); typed-error mapping; token never logged; **envelope unwrap** — a
  well-formed `{success:true,data:[...]}` yields tasks from `data` (not
  `results`), and a `200`-with-`{success:false}` (or non-list `data`) raises
  `HabiticaProviderError` rather than caching an empty list; `complete_task`
  for a daily asserts a **200-with-body** upstream response (not Todoist's
  `204`), sends an **empty `{}` JSON body**, and returns a bare ack.
- **client-today TZ-bridge (todos AND dailies)**: with `TIME_ZONE=UTC` and a
  `target_date` one day ahead of `localdate()` (`target = localdate()+1`,
  `include_overdue_carryover=True`): (a) a daily with `isDue` is still fetched +
  returned (non-UTC dailies-loss regression), AND (b) an unfinished todo due on
  the **server day** (`due_date == localdate()`) is returned as overdue-carry
  (the `< target_date` vs `< localdate()` boundary bug) — a `< localdate()`
  implementation would drop it. A past/future date with the flag off fetches
  **no** dailies and **no** undated/overdue todos.
- `checks`: E001 blocks `DEBUG=False` boot on missing/malformed
  `HABITICA_ENCRYPTION_KEY`; E001 blocks `DEBUG=False` boot on missing
  `HABITICA_CLIENT_ID` (both secrets = one boot gate); `DEBUG=True` → no error
  for either; W001 warns on ineffective cache backend with an account present;
  pre-migrate `try/except`.
- `views`: 503 on no account; 401/502/504 mapping; 500 on `ImproperlyConfigured`;
  cache hit/miss + `refresh=1` bypass; complete invalidates cache
  (`updated_at` bump); oversized-body guard on POST account.
- `cache`: versioned key rotation on `updated_at`; `with_overdue` vs `exact`
  scope separation; **invalidate safety** (mirror the Todoist suite) —
  `invalidate_tasks` uses `filter(pk=...).update(updated_at=now)` so a
  concurrent `set_token` is **not** clobbered (column-scoped update), and
  invalidating an **already-deleted** account is a silent no-op (the `.update()`
  matches zero rows — it must not resurrect the row).
- Schedule props: `external_tasks_poll_interval` reflects
  `EXTERNAL_TASKS_POLL_INTERVAL_SECONDS`.
- Frontend: `useHabitica` optimistic-complete rollback + silent refresh
  (mirror the Todoist Vitest specs); `useExternalTasksPoll`
  interval/visibility **and fan-out** — one tick invokes the caller's refresh
  fn which calls `refreshTasks` on **every** connected source (Todoist +
  Habitica), the core behavioral change from the single-callback
  `useTodoistPoll`; `active` stays false when `isWide`/`statusKnown` are false,
  **and when the LEFT task-rail is collapsed** (`externalTasksSidebarOpen ===
  false`) even if the chat rail is open — the guard that catches the chat-vs-task
  ref mix-up.
- **`useHabitica` no-status branch** (guards the feature-0020 regression in
  `tasks/lessons.md:27`): a **network/parse failure with no HTTP status**
  (`requestJson` returns `{ok:false, errors}`, `status === undefined`) for a
  **disconnected** user must **NOT** elevate `connected` / show the panel — the
  `connected`-elevation is gated on `result.status >= 401` (a definitive
  auth/server status that proves the account row exists). Also assert a **`400`
  bad-date** response (returned before account lookup) does **not** elevate.
  Mirror the Todoist divergence test exactly.
