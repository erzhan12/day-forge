# Feature 0026 — External calendar: add event to timeline + travel-time rules

GitHub issue #97. From the read-only **External Calendars** panel
(`ExternalEventsPanel.vue`, showing merged CalDAV + Google events), add a
per-event **"Add to schedule"** control that creates **one** `TimeBlock` on the
viewed date, stretched earlier/later by saved **travel-time rules** (e.g. 30 min
there + 30 min back), with a per-event override confirm dialog. Rules live in
**Settings → External Calendars** and are managed CRUD.

Block shape (from the issue, do not reopen):

```
title    = event.title
start    = event.start − travel_there   (clamped to 00:00)
end      = event.end   + travel_back    (clamped to 23:59)
category = rule.category ?? "other"
```

One block only — no separate commute blocks. On overlap with an existing block:
reject the create with the **same error UX as manual block create** (the
existing `{"errors": {"time": "This block overlaps with an existing block."}}`
400). Re-adding the same event is always allowed (no "already on timeline" lock).

## Locked decisions (do not reopen)

- **Rule match field:** title keyword only — case-insensitive substring of
  `rule.keyword` in `event.title`. No calendar-name/location/provider match.
- **Add flow:** always show a confirm dialog, prefilled from the matched rule's
  travel/category; user tweaks or confirms.
- **Multiple matching rules:** user-defined order; first match wins (orderable
  list in Settings, mirror the up/down reorder in `RulesList.vue`).
- **Day-bounds spill:** clamp to day (00:00 / 23:59), created with clamped times,
  no error.
- **All-day events:** hide the add button (no timed window to map).
- **Dialog fields:** travel-there minutes, travel-back minutes, category picker.
  Resulting start–end shown as a **read-only preview** (title/times derive from
  the event, not editable).
- **Backend placement:** `TravelRule` model + rule CRUD API in the
  **`calendar_sync`** app (provider-agnostic; `gcal_sync` untouched).
- **Snapping:** none — exact minutes from event ± travel.
- **Default category:** `other` unless a rule supplies one.
- **Works for** connected CalDAV and Google events already shown in the panel.

## Critical technical constraints (read before implementing)

1. **Off-grid times are required.** `TimeBlock.start_time`/`end_time` carry
   `validate_five_minute_granularity` (`backend/schedules/validators.py`:
   minute % 5 == 0, seconds == 0), and `schedules.api.create_block` enforces it
   via `full_clean()`. External events have arbitrary minutes (e.g. 14:07) and
   "no snapping" + "clamp to 23:59" both produce off-grid times. The new
   create path **must bypass 5-minute granularity** while keeping every other
   invariant (title, category, `start < end`, overlap). This is the reason for a
   **dedicated create endpoint** rather than reusing `create_block`. Off-grid
   blocks render fine (minute-precise `PX_PER_MINUTE` layout).

   **Off-grid blocks must survive later edits.** Two existing paths re-enforce
   granularity on *unchanged* times and would break from-event blocks:
   - `block_detail` (`backend/schedules/api.py`) ends with an unconditional
     `block.full_clean()`, which re-runs the `TimeField` five-minute validators
     even when `time_change` is false — so a **completion toggle or title edit
     on an off-grid block 400s**. Change it to
     `block.full_clean(exclude=["start_time", "end_time"])`: manual time edits
     already enforce granularity via `_validate_five_minute_or_error` and range
     via `_validate_time_range` *before* `full_clean`, and the model-level
     `clean()` (`start >= end` guard) still runs under `exclude`. Net loss of
     validation: none.
   - Drag-and-drop (`useDrag.ts` drop handler) snaps the new **start** to the
     5-minute grid but preserves the original duration (`newEnd = newStart +
     blockDuration`), so an off-grid *duration* (e.g. 26 min) yields an
     off-grid end that `reorder_blocks` rejects via
     `validate_five_minute_or_error` (`schedules/http.py`) — the drag 400s
     instead of snapping. Fix in `useDrag`: compute a `roundedDuration` (raw
     duration rounded **up** to the next `SNAP_MINUTES` multiple, min 5) and
     use it **both** for `newEnd = newStart + roundedDuration` **and** for the
     day-end clamp (`if (newStart + roundedDuration > DAY_END_MINUTES)
     newStart = DAY_END_MINUTES - roundedDuration`) — using the raw duration
     in the clamp would itself produce an off-grid `newStart` near the end of
     day. **Partially-visible clamped blocks:** the renderer clamps display
     to `[06:00, 23:00)` (`buildBaseDisplayItems` in `scheduleTime.ts` uses
     `clampedStart`/`clampedEnd`), but drag math and the ghost read the *raw*
     `block.start_time`/duration (`Schedule.vue:ghostHeight`), so dragging a
     day-bound-clamped from-event block (e.g. `00:00–06:30`, displayed as
     `06:00–06:30`) computes wrong grab-offset/ghost geometry. Fix: drag
     geometry and ghost sizing must use the same **display-clamped span** the
     renderer shows; the drop payload then derives from that clamped span
     (snapped start + rounded duration) — the hidden out-of-window remainder
     normalizing away on move is the same accepted normalize-on-move
     semantics as the round-up. **Apply the same round-up in
     `resolveConflicts`** (`useDrag.ts`):
     it shifts *neighbor* blocks preserving their raw `shiftDuration`, so an
     on-grid drag that displaces an off-grid neighbor still POSTs an off-grid
     `end_time` and `reorder_blocks` 400s — round every shifted block's
     duration up to the next `SNAP_MINUTES` multiple too. Frontend rounding
     alone is still not enough: `resolveConflicts` renumbers **every**
     block's `sort_order` and `endDrag` POSTs *sort-only* changes with their
     **original (off-grid) times**, so `reorder_blocks` must also stop
     re-validating granularity on **unchanged** times — it already loads the
     day's existing blocks, so for each update entry compare the submitted
     `start_time`/`end_time` to the stored values and enforce
     `validate_five_minute_or_error` **only when they differ** (format,
     `start < end`, and the batch overlap check always run) — **and** change
     the reorder pre-save `b.full_clean()` to
     `full_clean(exclude=["start_time", "end_time"])` (same footgun as
     `block_detail`: the field validators would re-fail unchanged off-grid
     times even after the HTTP-validator skip). Without this, nearly any
     drag on a day containing an off-grid from-event block 400s.
   - AI command/chat **move/resize** (`backend/ai/views.py:
     _compute_move_resize_times` / `_apply_move_or_resize`) fails on off-grid
     blocks two ways: (a) bare "move to HH:MM" preserves the off-grid
     duration exactly, so the recomputed `new_end` is off-grid; (b) a resize
     that supplies only `end_time` inherits the block's off-grid
     `start_time`, and `_check_granularity` runs on **both** values. Fix:
     enforce granularity only on times **supplied by the action** — a value
     inherited from the stored block skips the check (it was already
     persisted); for the bare-move branch, round the preserved duration
     **up** to the next 5-minute multiple before computing `new_end` (same
     normalize-on-move semantics as `useDrag`). The AI still can't introduce
     *new* off-grid times, since action-supplied values stay checked. Apply
     the **same action-supplied-only rule to `_check_day_window`** (it
     rejects `start < 06:00` / `end > 23:00`): a clamp-to-day from-event
     block (e.g. `00:00–00:30` or `23:10–23:59`) would otherwise fail every
     AI move/resize on its inherited time even after the granularity fix.
     Also change `_apply_move_or_resize`'s final `block.full_clean()` to
     `full_clean(exclude=["start_time", "end_time"])` (same rationale as
     `block_detail`): the field-level five-minute validators would otherwise
     re-fail the inherited off-grid time even after the selective
     `_check_granularity` fix — range and overlap stay explicitly checked in
     that path.
   - `restore_blocks` (`backend/schedules/api.py`) — the **undo** path —
     validates every snapshot entry with `_validate_block_times` (which
     includes `validate_five_minute_or_error`) and pre-passes unconditional
     `block.full_clean()` before `bulk_create`. Undo restores **full-day
     snapshots** (`useUndo.restoreBlocks`), so once one off-grid from-event
     block exists on a day, undoing *any* edit on that day (delete, rename,
     completion toggle, drag) 400s — day undo is broken. Fix: give
     `_validate_block_times` an `enforce_granularity=True` keyword and pass
     `False` from `restore_blocks` (restore re-persists previously-valid
     states; format, `start < end`, and the snapshot-overlap check all stay),
     and change the pre-pass to
     `block.full_clean(exclude=["start_time", "end_time"])`. `create_block`
     keeps granularity enforcement (default `True`).
   - **Accepted gap (ops-only):** the Django admin `ModelForm` still runs the
     field validators, so editing an off-grid from-event block via the admin
     UI fails validation. Not fixed in V1 — admin is not a product surface.

2. **Timezone / local-time mapping happens on the frontend.** Panel events carry
   `start`/`end` as ISO8601 **UTC** strings (`types/calendar.ts:NormalizedEvent`);
   `ExternalEventsPanel.formatTime` already renders them as local `HH:MM`.
   `TimeBlock` times are naive local `HH:MM`. The backend does not know the
   viewer's TZ, so the dialog computes final local `HH:MM` (event local time ±
   travel, then clamp) and POSTs those. The backend create endpoint stays
   TZ-agnostic — it only parses `HH:MM`, checks range/overlap, and inserts.

3. **Overlap + concurrency:** reuse the exact locked-insert pattern from
   `create_block` (`Schedule.objects.select_for_update().get(pk=...)` +
   overlapping-`TimeBlock` `select_for_update().exists()` inside
   `transaction.atomic()`), and call `schedule.mark_active_on_edit()` after a
   successful insert (forward-mutating edit).

### Create-block-from-event endpoint in `schedules` app

New handler `create_block_from_event` in `backend/schedules/api.py`, wired as:

```
path("api/schedules/<str:date>/blocks/from-event/", schedules_api.create_block_from_event, name="create_block_from_event")
```

Body: `{ "title", "start_time" (HH:MM), "end_time" (HH:MM), "category" }`.
Algorithm (a trimmed copy of `create_block` **minus 5-minute granularity**):

1. Parse `date` (400 on bad format).
2. `reject_oversized_body` first (same body-size cap as the templates_mgr
   rule endpoints; `create_block`/`block_detail` predate the helper and skip
   it — don't copy that), then
   `json.loads` body (400 on bad JSON). **Require the parsed body to be a
   `dict`, and `title` / `category` / `start_time` / `end_time` to be
   strings (`isinstance` checks) → 400 otherwise.** `create_block` skips
   these type checks and would 500 on e.g. a non-string `title`
   (`.strip()` → `AttributeError`) or an unhashable `category`
   (`in`-set → `TypeError`); do not inherit that hole into the new
   endpoint. (Fixing `create_block` itself is out of scope.)
3. `Schedule.objects.get_or_create(user, date)`.
4. Require `start_time`, `end_time`; parse each with
   `schedules.http.parse_time_or_error` (400 on non-`HH:MM`). **Do not call
   `validate_five_minute_or_error`.**
5. Validate `start < end` via `validate_time_range` (guards a degenerate clamp
   where a fully-out-of-day event collapses; the frontend clamps to
   `00:00`/`23:59`, so a same-value start/end 400s cleanly rather than saving a
   zero-length block).
6. Validate `title` (required, `≤ 255`) and `category` (in `VALID_CATEGORIES`,
   default `"other"`), same messages as `create_block`.
7. Inside `transaction.atomic()`: `Schedule…select_for_update().get(pk=…)` +
   overlapping-block `select_for_update().exists()`; on overlap return the
   **identical** `{"errors": {"time": "This block overlaps with an existing
   block."}}` 400.
8. Build the `TimeBlock` and save **bypassing the granularity validator** —
   call `block.full_clean(exclude=["start_time", "end_time"])` (keeps
   title/category field validation; range already checked in step 5), then
   `block.save()`. Wrap steps 7–8 in the same `try/except ValidationError`
   as `create_block` and return `{"errors": e.message_dict}` 400 — without
   it any residual `full_clean` failure surfaces as a 500.
9. `schedule.mark_active_on_edit()`; return `block_to_dict(block)` with 201.

### Matching util — `frontend/src/utils/travelRules.ts`

Pure function `matchTravelRule(rules: TravelRule[], title: string): TravelRule |
null`:
1. Assume `rules` already ordered ascending by `order` (server order); do not
   re-sort by anything else.
2. Return the **first** rule whose `keyword.trim().toLowerCase()` is a non-empty
   substring of `title.toLowerCase()`. Else `null`.

Second helper `computeEventBlockTimes(event, viewedDate, travelThere,
travelBack)` → `{ start_time, end_time }` local `HH:MM`. **Anchor everything
to the VIEWED date's local midnight, not the event's start day.** Both
providers return every event that *overlaps* the fetch window — a day
computed in `settings.TIME_ZONE` (**UTC**), not the viewer's local day
(window guards in `calendar_sync/service.py` / `gcal_sync/service.py` keep
any event with `end > day_start && start < day_end`) — so the panel can show
an event
that *started the previous day* (e.g. `23:00(prev)`→`00:30(viewed)`). Anchoring
to the event's start day would create `23:00–23:59` on the viewed schedule
instead of the correct `00:00–00:30`.

1. `startLocal = new Date(event.start)`, `endLocal = new Date(event.end)`;
   `viewedDay = new Date(y, m − 1, d)` parsed from the `viewedDate`
   (`YYYY-MM-DD`) string — local midnight of the viewed day.
2. **Read each value's local clock time, then fold in the whole-day delta to
   the viewed day, then apply travel in the minutes domain** — do NOT shift
   the `Date` object and re-read `getHours()`/`getMinutes()` off it (a large
   travel value can push the shifted instant onto another calendar day, whose
   clock reading defeats the clamp):
   - `startDay = new Date(startLocal.getFullYear(), startLocal.getMonth(), startLocal.getDate())`,
     same for `endDay`.
   - `startDelta = Math.round((startDay − viewedDay) / 86_400_000)`,
     `endDelta = Math.round((endDay − viewedDay) / 86_400_000)` (whole calendar
     days; `Math.round` absorbs the DST ±1h so the ratio still lands on an
     integer; deltas are `−1`/`0`/`+1` for day-overlapping events).
   - `startMin = startLocal.getHours()*60 + startLocal.getMinutes() + startDelta*1440 − travelThere`
   - `endMin   = endLocal.getHours()*60   + endLocal.getMinutes()   + endDelta*1440   + travelBack`
3. **Out-of-day guard, then clamp — in the minutes domain.** The provider day
   window is computed in `settings.TIME_ZONE` (**UTC** — `_day_window` in
   `calendar_sync/service.py`, same for gcal), while the viewed `date` string
   is **browser-local** (`utils/date.ts:toLocalDateString`). When the two
   timezones diverge, the panel can list an event that lies entirely
   *outside* the viewer's local day (e.g. local `03:00` next calendar day =
   `22:00` UTC of the viewed day). Blindly clamping such an event would
   produce an inverted or nonsense range, so first check intersection:
   - If `endMin <= 0` or `startMin >= 1440` (event ± travel does not touch
     the viewed local day at all): return a sentinel (e.g. `null`) — the
     dialog **disables Confirm** with an inline hint ("event lies outside the
     viewed day").
   - Else clamp: `startMin = max(0, startMin)` (→ `00:00`), `endMin =
     min(23*60 + 59, endMin)` (→ `23:59`). If `startMin >= endMin` after
     clamping (zero-length event, see the dialog's zero-length guard) —
     Confirm disabled as well.
   Worked examples: prev-day spillover `23:00(prev)`→`00:30(viewed)`, 0/0
   travel → `startMin = 1380 − 1440 = −60 → 0 = 00:00`, `endMin = 30` →
   `00:00–00:30` ✓; next-day spillover `23:00(viewed)`→`00:30(+1d)` →
   `endMin = 30 + 1440 = 1470 → 1439 = 23:59` ✓; morning event + `travelThere
   = 600` cap → clamps to `00:00`, never `22:00` of another day ✓; local
   `03:00(+1d)` event (UTC-window artifact), 0/0 travel → `startMin = 1620 ≥
   1440` → `null` → Confirm disabled ✓.
4. Format each clamped minute count back to zero-padded `HH:MM`
   (`floor(min/60)`:`min%60`). (Same local-TZ semantics as
   `ExternalEventsPanel.formatTime` — that one uses `toLocaleTimeString` for
   display; this util reads `getHours()`/`getMinutes()` like
   `utils/externalEventPast.ts`.)

Unit-test this util (matching + clamp + subtract/add) — it holds the core
algorithm. **Include a cross-midnight case**: an early-morning event (e.g.
`08:00` local) with `travelThere = 600` must yield `start_time = "00:00"` (not
`22:00` of the prior day), and a late event near `23:59` with a large
`travelBack` must yield `end_time = "23:59"` — i.e. exercise both the
below-`0` and above-`1439` clamp branches, not just in-day clamps.
**Include an event-span-crosses-midnight case**: `23:00`→`00:30(+1d)` local with
`0`/`0` travel must yield `start_time = "23:00", end_time = "23:59"` (the
`endDelta * 1440` fold makes `endMin = 30 + 1440 = 1470 → 23:59`), guarding
against the inverted-range 400 that the naive same-day `endMin` produced.
**Include a previous-day-spillover case**: event `23:00(prev)`→`00:30(viewed)`
with `0`/`0` travel and `viewedDate` = the viewed day must yield
`start_time = "00:00", end_time = "00:30"` (the `startDelta = −1` fold pushes
`startMin` to `−60 → clamp 0`), guarding against creating a `23:00–23:59`
block on the wrong day.
