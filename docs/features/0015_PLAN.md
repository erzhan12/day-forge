# 0015 — Migrate cache from LocMemCache to Redis

## Context

Day Forge's three AI rate-limit buckets (`ai_cmd_rl`, `ai_draft_rl`,
`ai_chat_rl`) store fixed-window counters in `CACHES['default']` via
`ai.views._consume_rate_limit` (uses `cache.aadd` / `cache.aincr` /
`cache.aset`). The current `default` backend is `FileBasedCache`
(`backend/day_forge/settings.py` lines 125-130) — **not** the
`LocMemCache` the issue assumes, but it is still effectively per-process
for counter correctness (FileBasedCache uses per-file locks but is not a
shared atomic-incr backend across workers and is not what the system
check or production target wants). The `ai.E001` system check
(`backend/ai/checks.py`) only blocks on a backend whose path
`.endswith(".LocMemCache")`, so it does **not** currently fire on
`FileBasedCache` — but the issue's production target is Redis regardless.

This feature switches `CACHES['default']` to Redis via Django's built-in
`django.core.cache.backends.redis.RedisCache` (Django 5.x ships native
Redis support — no third-party `django-redis` package required, which
keeps the `uv` dependency set minimal; the `redis` Python client is the
only new dep). It also wires a `REDIS_URL` env var, adds a `redis`
service to `docker-compose.yml`, documents the var, and updates
`ai.E001` so the rate-limit guard cannot regress to *any* ineffective
cache backend for rate limiting (LocMem **or** FileBased **or** Dummy).

The native `RedisCache` backend is the chosen path over the
third-party `django-redis` package: it is in Django core (no extra
maintenance surface), satisfies the `ai.E001` check, and provides
atomic `incr` across workers via Redis `INCR`. `django-redis` is only
needed for advanced features (custom serializers, sharding) we don't
require.

## Phase 1 — Config + dependency (data/infra layer)

This phase is the foundation; everything else depends on it.

1. **`pyproject.toml`**: add `"redis>=5.0"` to `[project].dependencies`.
   Run `uv lock` (regenerates `uv.lock`) so the Dockerfile's
   `uv sync --frozen` picks it up.

2. **`backend/day_forge/settings.py`** — replace the `CACHES` block
   (lines 125-130) with logic that:
   - Reads `REDIS_URL = os.environ.get("REDIS_URL", "")`.
   - When `REDIS_URL` is set: `CACHES['default']` uses
     `"BACKEND": "django.core.cache.backends.redis.RedisCache"` and
     `"LOCATION": REDIS_URL`. Add a `"KEY_PREFIX": "dayforge"` so the
     rate-limit keys (`ai_cmd_rl:<id>` etc.) and the `caldav_events:*`
     keys are namespaced if Redis is shared with another app.
   - When `REDIS_URL` is empty **and** `DEBUG=True`: fall back to
     `LocMemCache` (conventional dev backend; what tests/conftest
     already assume — see Phase 3). Do **not** keep FileBasedCache as the
     dev fallback.
   - **Dev policy (explicit):** `ai.E001` fires whenever `LLM_API_KEY` is
     set and `CACHES['default']` is an ineffective cache backend for
     rate limiting (LocMem / FileBased / Dummy), **independent of `DEBUG`**
     (`test_fires_even_when_debug_true`). After this feature broadens
     `ai.E001` to include FileBased, AI-enabled local dev **requires**
     `REDIS_URL` and a running Redis — the common case with
     `LLM_API_KEY` in `.env` and no `REDIS_URL` cannot pass
     `manage.py check` / `runserver` on LocMem alone (a regression vs
     today's FileBased default, which `ai.E001` does not block). That is
     intentional: once AI is on, rate-limit counters must be shared.
     Developers exercising AI locally must set `REDIS_URL` (e.g.
     `redis://localhost:6379/0`) and start Redis; developers without AI
     can leave `LLM_API_KEY` unset and run without Redis. Do **not**
     document "no Redis required" for AI-enabled dev.
   - When `REDIS_URL` is empty **and** `DEBUG=False`: same LocMem
     fallback; `ai.E001` blocks startup when `LLM_API_KEY` is set — keep
     LocMem so the check (not a raw connection `KeyError`) produces the
     actionable error.

3. Verify import-time: `REDIS_URL` is read with a default, so an unset
   var never raises at settings import (matches the existing
   `os.environ.get(..., default)` pattern used for every other var).

## Phase 2 — Harden the `ai.E001` system check

`backend/ai/checks.py::error_locmem_cache_with_ai_in_production`:

1. Replace the single-backend test
   `if backend.endswith(".LocMemCache")` with membership in an
   ineffective-backend tuple covering `LocMemCache`, `FileBasedCache`,
   and `DummyCache` (reuse the constant name pattern from
   `calendar_sync.checks._INEFFECTIVE_CACHE_BACKENDS`; define a local
   tuple in `ai/checks.py` — do not cross-import from `calendar_sync` to
   avoid an app-to-app dependency). **Wording discipline:** do not call
   every blocked backend "per-process." LocMem and Dummy are literally
   per-process; `FileBasedCache` is blocked because `incr` is not a
   shared atomic counter across workers (file locks ≠ Redis `INCR`), not
   because it shares LocMem's isolation model. The current FileBasedCache
   default would, post-change, correctly trip `ai.E001` whenever
   `LLM_API_KEY` is set (regardless of `DEBUG`) — the intended forcing
   function to make operators set `REDIS_URL`.

2. Update the `Error.msg` / `hint` wording so it no longer says
   "LocMemCache" specifically but names an **ineffective cache backend
   for rate limiting** (non-shared or non-atomic across workers), listing
   LocMemCache / FileBasedCache / DummyCache by name. Keep the
   `id="ai.E001"` and keep "Redis" in the hint (a `test_checks.py`
   assertion checks for the substring `"Redis"`).

3. The check must stay silent for `RedisCache` and `PyMemcacheCache`
   (shared, atomic backends) — the membership test gives this for free.

2. **`backend/tests/test_ai_views.py`** (`TestRateLimit`): add a
   **key-shape regression** test only — after one rate-limited request,
   assert `cache.get` sees the counter under `ai_cmd_rl:<user_id>` (the
   Redis `KEY_PREFIX` is transparent to `cache.get`, so the assertion
   holds on LocMem in the pinned suite and on Redis in integration).
   Do **not** label this a "worker-fanout" or cross-worker test: two
   `Client` instances in one process already share LocMem, FileBased, and
   Dummy, so a "two clients, one counter" assertion would pass on the
   exact broken configurations `ai.E001` guards against and proves
   nothing about multi-worker behavior. **Cross-worker verification:**
   rely on the hardened `test_checks.py` cases (FileBased/Dummy/LocMem +
   AI → `ai.E001`; Redis → silent) plus an **optional** Redis-backed
   integration or smoke test (e.g. two processes or a documented manual
   step: two `runserver` workers, same `REDIS_URL`, exhaust the limit on
   one, confirm 429 on the other) — out of scope for the default unit
   suite unless CI gains a Redis service.

3. **`backend/tests/conftest.py`**: tests must continue to run without a
   live Redis — and this must hold **regardless of the host shell / CI
   `.env`**. This is the trap: after Phase 1, `CACHES['default']` is
   built from `REDIS_URL` at settings-import time, and `settings.py`
   calls `load_dotenv(PROJECT_ROOT/.env)`. A developer or CI runner whose
   `.env` defines `REDIS_URL` (exactly the var this feature introduces
   and documents in `.env.example`) would make the entire unit suite try
   to connect to a live Redis at import — turning every cache-touching
   test (`TestRateLimit`, the `cache.get` assertions in
   `test_ai_views_chat.py` / `test_ai_views_draft.py`, and the autouse
   `_clear_cache` `cache.clear()`) into a connection-error failure with
   no Redis running. The existing `_force_test_safe_http_settings`
   session-autouse fixture overrides **only** the SSL/cookie/HSTS
   settings — it does **not** touch `CACHES` — so nothing currently pins
   the test backend.

   **Fix (deterministic pin, independent of host env):** add a
   session-scoped, autouse fixture in `conftest.py` that wraps the full
   suite in `django.test.override_settings(CACHES={"default": {"BACKEND":
   "django.core.cache.backends.locmem.LocMemCache"}})` (`with
   override_settings(...): yield`). Do **not** mutate `settings.CACHES`
   directly and do **not** rely on `caches.close_all()` alone to re-pin:
   `ConnectionHandler` caches the resolved config on first access;
   `close_all()` drops backend instances but does not invalidate the
   handler's cached settings, so `caches['default']` can still return the
   pre-pin backend (verified locally: after `settings.CACHES = LocMem`
   plus `close_all()`, `caches['default']` remained `FileBasedCache`).
   `override_settings` emits the `setting_changed` signal for `CACHES`,
   which is what makes the handler re-read the config. This guarantees
   LocMem under test no matter what `REDIS_URL` is set to in the ambient
   environment.
   (The `_clear_cache` autouse fixture's `cache.clear()` then operates on
   this pinned LocMem backend.) Note this is the same reason the existing
   `test_checks.py` / `test_calendar_sync_checks.py` tests pass an
   explicit `override_settings(CACHES=...)` per test — the suite already
   cannot rely on the ambient backend, and this fixture extends that
   discipline to the implicit-default callers. Do **not** rely on Phase
   1's dev fallback resolving to LocMem at the suite level: that fallback
   only fires when `REDIS_URL` is unset, which is precisely the
   precondition we cannot guarantee.

   Tests that genuinely need a different backend (e.g. the
   `test_checks.py` cases asserting on Redis/FileBased) keep using their
   own per-test `override_settings(CACHES=...)`, which transparently
   shadows this session pin for the duration of the test. Update the
   `_clear_cache` docstring if it hardcodes "LocMemCache" (it still names
   LocMem, which remains accurate under this pin).

## Algorithm notes

`_consume_rate_limit` uses the abstract cache API, which Django routes to
the configured backend. On `RedisCache`:

- `cache.aadd(key, 1, 3600)` → Redis `SET key 1 EX 3600 NX` (atomic
  set-if-absent, anchors the window TTL to the first call). `BaseCache.aadd`
  bridges to the *sync* `RedisCache.add`, so this path is genuinely atomic.
- The increment goes through `await sync_to_async(cache.incr)(key)` →
  Redis `INCR key` (atomic across all workers — the core fix; the
  per-process `incr` of LocMem/Dummy and the non-atomic FileBased `incr`
  do not share a counter across workers). **Implementation note (corrected
  during review — see 0015_REVIEW.md H1/M1):** the original draft called
  `cache.aincr`, but Django's `RedisCache` does **not** override the async
  `aincr` — it inherits `BaseCache.aincr`, a non-atomic `aget`→`aset`
  read-modify-write that *also* rewrites the TTL to `default_timeout`
  (300s), collapsing the 3600s window. Only the **sync** `incr` reaches
  atomic `INCR` and leaves the TTL untouched, so the implementation bridges
  to it via `sync_to_async` (the same pattern `BaseCache.aadd` uses for
  `add`).
- The `except ValueError → aset` reseed path (key evicted mid-window)
  still applies: `RedisCache.incr` raises `ValueError` for a missing key
  (matching the documented contract), so the existing
  `test_cache_incr_value_error_reseeds_counter` stays valid (it patches the
  sync `cache.incr`). `test_increment_preserves_window_ttl` guards M1.

No data migration. No model changes. No new Django app.
