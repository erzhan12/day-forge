# 0013 — Show time remaining for the current block

GitHub Issue #25. Surface how much time is left in the **currently active** schedule block (the block whose time range contains "now"), so users can see at a glance how long until the block ends without mental math.

**Out of scope:** notifications/alarms when a block ends; cross-day blocks.

This is a **frontend-only** change. No backend, API, or DB work. Block times are already `"HH:MM"` strings in local timezone; `Schedule.vue` already tracks `nowMinutes` on a 60-second interval for the today-only `NowLine` overlay.

## UI approach

Implement **inline remaining time on the active block row** (issue direction: badge/subtitle, e.g. `23m left`).

Do **not** add a separate header/summary strip in v1 — the active row is already visually marked by the existing `NowLine` overlay on `block-with-now` slots; adding the countdown beside the block’s time badge keeps compact blocks readable and avoids crowding `DateNavigator`.

**Compact blocks (≤ 30 min):** append a small `remaining-badge` after the time badge (same row, `flex-shrink: 0`, truncated title unchanged).

**Expanded blocks:** show `remaining-badge` in `.block-header` between the existing `.duration` (total block length) and `.delete-btn`. Wording distinguishes total vs remaining: duration stays `1h 30m`, remaining reads `23m left`.

Use theme tokens already in `TimeBlock.vue` (`--accent` or `--text-muted`) — no new CSS token freeze violations.

When no block contains “now” (gap, before first block, after last block, or viewing a non-today date): render nothing extra.

## Algorithms

### 1. Current-block selection

Inputs: `blocks: TimeBlock[]`, `nowMinutes: number`, `nowDate: string | null`.

`nowDate` is the date string the caller's `nowMinutes` was sampled against (typically `useNowMinutes`'s internal today reference, exposed alongside `nowMinutes`); pass `null` when the caller knows the view is not today. **Today-ness is an explicit input, not an ambient read inside the helper.** The helper does not call `todayString()` — this keeps it pure and avoids two independent today-reads disagreeing across a midnight tick (composable samples `nowMinutes` for May-22 at 23:59:59.9; helper otherwise runs at 00:00:00.05 May-23 and would mis-classify the same `viewedDate`).

Caller derives the boolean once: `const isToday = nowDate !== null && viewedDate === nowDate;` (or passes `nowDate = isToday ? viewedDate : null`).

1. If `nowDate === null` → no current block (`null`).
2. Filter blocks where `timeToMinutes(start_time) <= nowMinutes < timeToMinutes(end_time)`.
3. If zero matches → `null`.
4. If one match → that block.
5. If multiple matches (overlap race / bad data; backend normally rejects overlaps): return the **first** block in `(start_time, sort_order)` order whose half-open interval contains `nowMinutes`. This matches `TimeBlock.Meta.ordering = ["start_time", "sort_order"]`, the API payload order, and `displayList`'s NowLine splice (first matching item in iteration order). Document this in a one-line comment on the helper. Overlap is unlikely with normal data; no user-facing error.

### 2. Remaining minutes

For current block with `endMinutes = timeToMinutes(end_time)`:

```
remaining = endMinutes - nowMinutes
```

Clamp: if `remaining <= 0`, treat as no current block (edge: exactly at `end_time`, half-open interval excludes the block).

Granularity matches existing UI: minute-level only (no seconds), refreshed on the same 60s interval as `nowMinutes`. At 9:37 viewing a block ending 10:00, display `23m left` until the next tick.

### 3. Duration / remaining label formatters

Add shared `formatDurationMinutes(minutes: number): string` in `scheduleTime.ts`.

**Contract (locked — refactor must be semantics-preserving):**

- `minutes < 0` → caller error. Return `"0m"` (do not throw; the formatter is a leaf util reused by remaining/duration paths and a runtime throw would crash the row). Document as caller-error path.
- `minutes === 0` → `"0m"`.
- `0 < minutes < 60` → `"${minutes}m"` (e.g. `23` → `"23m"`).
- `minutes === 60` → `"1h"` (whole-hour case; matches existing `TimeBlock.vue` duration output for a 60-minute block).
- `minutes > 60 && minutes % 60 === 0` → `"${h}h"` (e.g. `120` → `"2h"`).
- `minutes > 60 && minutes % 60 !== 0` → `"${h}h ${m}m"` (e.g. `90` → `"1h 30m"`).
- `minutes >= 1440` (24h+): same formula keeps applying (`1440` → `"24h"`). Same-day blocks make this unreachable in practice, but the util is contract-stable for reuse.

Refactor `TimeBlock.vue` to import `formatDurationMinutes` for its `.duration` display instead of inline logic. Verify (and add if missing) `TimeBlock.test.ts` cases for 60-min (renders `"1h"`) and 120-min (renders `"2h"`) blocks to lock the refactor as semantics-preserving.

Add `formatRemainingMinutes(minutes: number): string` as a thin wrapper: `formatDurationMinutes(minutes) + " left"` (e.g. `23m left`, `1h left`, `1h 30m left`). Single source of truth prevents label drift between total duration and remaining countdown. `remainingMinutesForBlock` already clamps and returns `null` outside the window, so the wrapper never receives `<= 0` from the production path.

## Files to change

### Create / extend utilities

**Modify `frontend/src/utils/scheduleTime.ts`**

- `findCurrentBlock(blocks, nowMinutes, nowDate): TimeBlock | null` — implements algorithm §1. **Does not import `todayString`**; today-ness is an explicit input (`nowDate: string | null`) supplied by the caller, derived from `useNowMinutes`'s own today reference. This keeps the helper pure and deterministic in unit tests without `vi.setSystemTime` driving its branch logic.
- `remainingMinutesForBlock(block, nowMinutes): number | null` — returns clamped remaining or `null` when not current.
- `formatDurationMinutes(minutes: number): string` — algorithm §3; shared by `TimeBlock.vue` and remaining badge.
- `formatRemainingMinutes(minutes: number): string` — `formatDurationMinutes(minutes) + " left"`.

Alternative: `frontend/src/utils/scheduleNow.ts` if keeping `scheduleTime.ts` strictly layout-focused; either is fine — prefer colocation in `scheduleTime.ts` unless the file grows unwieldy.

**Create `frontend/src/composables/useNowMinutes.ts`** (required, not optional)

Extract the duplicated interval logic from `Schedule.vue` and `SkippedTasks.vue`. The current `Schedule.vue` interval starts only in `onMounted` when `isToday` is true; Inertia date navigation does **not** remount the page (`Schedule.test.ts`, `RULES.md`), so navigating from a non-today date to today leaves `nowMinutes` frozen until a full reload — breaking both NowLine and the new countdown.

Composable contract (pinned — do not leave round-trip behaviour to implementer choice):

- Accept `viewedDate: Ref<string>` (or equivalent getter).
- Internal state: `nowMinutes: Ref<number | null>` (null when not-today), `nowDate: Ref<string | null>` (the local date string `nowMinutes` was sampled against; null when not-today), `currentHHMM: Ref<string>` (`""` when not-today — see below).
- `watch(viewedDate, …, { immediate: true })` behaviour, **unconditional on every fire**:
  - Compute `today = todayString()`.
  - If `viewedDate === today`: synchronously call `getCurrentMinutes()` and assign to `nowMinutes`, set `nowDate = today`, set `currentHHMM` from the same sample. Then (re)start the 60s interval — clearing any prior interval handle first so re-entering today is idempotent and never leaks a second timer.
  - Else (leaving today, or non-today on first fire): clear the interval handle, set `nowMinutes = null`, `nowDate = null`, `currentHHMM = ""`.
  - This applies on **every** watcher fire, including same-instance round-trips (today → non-today → today): re-entering today always produces a fresh sample, never a stale one.
- **Interval-tick behaviour (CRITICAL — handles day rollover on a long-mounted session):** the 60s callback must NOT just refresh `nowMinutes` — it must re-sample today and re-evaluate the same branch as the watcher. Concretely, every tick:
  - Compute `today = todayString()` (the real wall-clock today, which may have advanced since the interval was last (re)started).
  - If `viewedDate.value === today`: re-sample `getCurrentMinutes()` / HH:MM and assign to `nowMinutes` / `nowDate = today` / `currentHHMM`. Carry on.
  - Else (day rolled over while the user stayed on a now-stale `viewedDate`, e.g. tab left open past midnight on yesterday's view): execute the leave-today path inline — clear the interval handle, set `nowMinutes = null`, `nowDate = null`, `currentHHMM = ""`. The watcher will not re-fire because `viewedDate` itself did not change; the interval callback is the only place this transition can be detected. **Implementing the tick as `nowMinutes = getCurrentMinutes()` and nothing else is the bug this clause exists to prevent.**
- Clear interval on unmount; reset refs is unnecessary on unmount but harmless.
- Returned shape: `{ nowMinutes, nowDate, currentHHMM }`. `nowDate` is what `Schedule.vue` passes to `findCurrentBlock` as the explicit today-input (§Algorithms 1).
- `currentHHMM` is exactly `""` (empty string) when not-today. `SkippedTasks.vue` consults `currentHHMM` **only** in its `isToday` branch (`SkippedTasks.vue:39`, inside an `isPastDay`/`isToday` switch at lines 29–41); the past-day branch returns `true` for every incomplete block without reading `currentHHMM`, and the future-day branch returns `[]` early without reading it either. Therefore `currentHHMM === ""` on non-today is **never consulted** by `SkippedTasks.vue` in practice — the refactor preserves past-day "all incomplete = skipped", today "incomplete && end_time < now", and future "[]" semantics exactly. No analytics behaviour change.

**`SkippedTasks.vue` mount lifecycle (corrected — was previously misstated):** the component is rendered by **`Analytics.vue`** at `Analytics.vue:124` (`<SkippedTasks :blocks="blocks" :date="date" />`), **unconditionally** at the parent — no `v-if="isToday"` gate. It is **not** rendered by `Schedule.vue` at all. The component itself has internal `isPastDay`/`isToday` branching (`SkippedTasks.vue:29-41`) plus a section-level `v-if="skipped.length"` (`SkippedTasks.vue:55`) that hides the panel when the computed `skipped` list is empty (e.g. on future dates). Implications for this refactor:
  - Do **not** add a `v-if="isToday"` gate anywhere — `SkippedTasks` already self-gates via its own date-aware computed.
  - The composable refactor for `SkippedTasks.vue` is **strictly behaviour-preserving** (pure de-duplication of the 60s interval): past-day analytics keeps showing all incomplete blocks (the only branch that fires on past days); today keeps showing incomplete-with-end-passed; future keeps showing nothing. Issue #25 is scoped to remaining-time on the active block — past-day analytics semantics are **out of scope** and must not change.
  - The composable's leave-today branch (`currentHHMM = ""`, `nowDate = null`) is reachable from `SkippedTasks.vue` (its parent does not gate by today), but as noted above, `SkippedTasks` does not consult those values outside its own `isToday` branch — so the leave-today values are inert with respect to the analytics output. They matter only for `Schedule.vue`, which uses `nowDate` to drive `findCurrentBlock` (and `Schedule.vue` is the only consumer where the leave-today semantics are user-visible).

Refactor `Schedule.vue` and `SkippedTasks.vue` to use this composable instead of local interval setup.

### `frontend/src/pages/Schedule.vue`

Use `useNowMinutes(toRef(props, 'date'))` and destructure `{ nowMinutes, nowDate }` (drop the local interval setup entirely).

Define exactly **two** parent computeds — one current-block lookup and one shared remaining-minutes value (at most one block is current per tick, so recomputing per-row would be wasted work and a hidden N-times-per-tick cost):

```ts
const currentBlock = computed(() =>
  findCurrentBlock(effectiveBlocks.value, nowMinutes.value ?? 0, nowDate.value)
)
const currentBlockRemaining = computed(() =>
  currentBlock.value && nowMinutes.value !== null
    ? remainingMinutesForBlock(currentBlock.value, nowMinutes.value)
    : null
)
```

(`nowMinutes.value ?? 0` is safe because `nowDate.value === null` when not-today causes `findCurrentBlock` to early-return `null` per algorithm §1 step 1, so the `nowMinutes` value is never consulted in that branch.)

Pass to **both** `TimeBlock` usages in the template (do not update only one branch):

- `block-with-now` slot (`item.type === 'block-with-now'`) — the row that carries the `NowLine` overlay.
- plain `block` slot (`item.type === 'block'`).
- Identical bindings on each:
  - `:is-current="item.block?.id === currentBlock?.id"`
  - `:remaining-minutes="item.block?.id === currentBlock?.id ? currentBlockRemaining : null"`
- **Only** block slots, not gaps. Parent computed keeps `TimeBlock` presentational.

Ensure drag preview path stays correct: use `effectiveBlocks` (same source as `displayList`) so the countdown follows the block being viewed, not a stale id during drag.

## Behaviour summary

| Condition | Display |
|-----------|---------|
| Viewing today, now inside block `[start, end)` | `Xm left` on that block only |
| Viewing today, now in a gap | Nothing |
| Viewing yesterday/tomorrow | Nothing |
| Block ends exactly at now | Nothing (half-open end) |
| Multiple overlapping blocks | Countdown on first block in `(start_time, sort_order)` order (same row as NowLine) |

Same timezone rules as the rest of the schedule UI: browser local time via `Date`, consistent with `todayString()` and `Schedule.vue`'s `getCurrentMinutes()`.
