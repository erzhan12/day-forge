# Phase 3: Drag-and-Drop & Undo

Time blocks become reorderable via drag-and-drop with 5-minute snap-to-grid, automatic conflict resolution (overlapping blocks shift forward), and an in-memory undo stack covering all mutations. A toast notification with clickable undo accompanies every change.

---

## Backend: Batch Reorder Endpoint

### New endpoint: `POST /api/blocks/reorder/`

**File:** `backend/schedules/api.py` — add `reorder_blocks()` view function

Accepts a JSON body with an array of block updates. Each entry contains `id`, `start_time`, `end_time`, and `sort_order`. All blocks must belong to the same schedule and the same authenticated user.

Logic:
1. Parse JSON body — expect `{ "updates": [{ "id": int, "start_time": "HH:MM", "end_time": "HH:MM", "sort_order": int }, ...] }`
2. Validate update IDs are unique, every block exists, every block belongs to the request user's schedule, and all share the same `schedule_id`
3. Validate times and ordering: 5-minute granularity, `start_time < end_time`, `sort_order` is an integer in the same range accepted by `PATCH /api/blocks/{id}/`
4. Build the full candidate schedule state by merging updated blocks with unchanged blocks from the same schedule
5. Validate the final candidate state has no overlaps after sorting by `start_time`, `sort_order` (do not trust the frontend conflict-resolution preview)
6. Inside `transaction.atomic()`: bulk-update each block's `start_time`, `end_time`, `sort_order`
7. Return `200` with the full updated block list (same format as `schedule_view` serializes)

**Validation guardrails:**
- Reject duplicate block IDs in one request.
- Reject cross-user and cross-schedule updates.
- Reject final states that overlap unchanged blocks, not only overlaps among the submitted updates.
- Run `full_clean()` (or the equivalent validators) on each candidate block before writing.
- Treat validation failures as all-or-nothing; no partial updates should be committed.
- Return blocks ordered by `start_time`, `sort_order`.

**File:** `backend/day_forge/urls.py` — add route `path("api/blocks/reorder/", api.reorder_blocks)`

**File:** `frontend/src/composables/useSchedule.ts` — add `reorderBlocks(updates)` method calling `apiFetch("/api/blocks/reorder/", "POST", { updates })`

## Frontend: Drag-and-Drop

### Approach: Custom pointer events (no library)

Use `pointerdown` / `pointermove` / `pointerup` on a drag handle element. This avoids adding a dependency and gives full control over the ghost preview, snap logic, and animations. Works on mouse and touch.

### New composable: `frontend/src/composables/useDrag.ts`

**Exported state:**
- `isDragging: Ref<boolean>`
- `dragBlockId: Ref<number | null>` — id of the block being dragged
- `ghostTop: Ref<number>` — current pixel offset of the ghost element
- `previewStartTime: Ref<string>` — real-time "HH:MM" preview during drag
- `previewEndTime: Ref<string>`
- `shiftedBlockIds: Ref<Set<number>>` — blocks that would be displaced at current position
- `previewBlocks: Ref<TimeBlock[]>` — tentative full block list after applying the dragged block + cascade shifts
- `shiftOffsets: Ref<Map<number, number>>` — optional pixel offsets for FLIP/transform-based visual previews

**Exported functions:**
- `startDrag(event: PointerEvent, block: TimeBlock, containerEl: HTMLElement)` — captures pointer, records initial offset, enters drag mode
- `endDrag()` — computes final times, calls conflict resolution, invokes `reorderBlocks()`, pushes to undo stack, releases pointer
- `cancelDrag()` — restores original state, releases pointer

**Internal logic — snap-to-grid:**
1. Convert pointer Y position relative to the schedule container to minutes: `minuteOffset = Math.round((pointerY - containerTop) / PX_PER_MINUTE)`
2. Snap to nearest 5 minutes: `snapped = Math.round(minuteOffset / 5) * 5`
3. Add `DAY_START` (360 min = 06:00) to get absolute start minute
4. `previewStartTime` = `minutesToTime(snapped + DAY_START)`
5. `previewEndTime` = add original block duration to preview start
6. Clamp the preview to the visible day window; if the resulting cascade would exceed `DAY_END`, mark the drop invalid and do not submit it

**Internal logic — conflict resolution (auto-shift):**
1. On every `pointermove` (debounced to ~16ms via rAF): identify blocks whose time range overlaps with the dragged block's preview range (excluding the dragged block itself)
2. For each overlapping block (sorted by `start_time`): compute the shift needed so it starts after the previous block ends. Cascade forward through all subsequent blocks.
3. Store the full tentative result in `previewBlocks`; use `shiftedBlockIds` only for styling and `shiftOffsets`/FLIP only if transform animation is needed
4. On `endDrag()`: collect the final times for the dragged block + all shifted blocks from `previewBlocks` → call `reorderBlocks()`
5. If the preview is invalid (for example it would push a block past `DAY_END`), cancel the drop and restore the original state

**Constants** (shared runtime config):
- `PX_PER_MINUTE = 2`
- `DAY_START = 360` (06:00)
- `DAY_END = 1380` (23:00)
- `SNAP_MINUTES = 5`

## Frontend: Undo System

### New composable: `frontend/src/composables/useUndo.ts`

**Types:**
```
UndoAction {
  description: string          // e.g. "Moved Standup to 10:00"
  type: "drag" | "edit" | "toggle" | "add" | "delete"
  previousBlocks: TimeBlock[]  // full snapshot of blocks before the action
  scheduleDate: string
}
```

**Exported state:**
- `undoStack: Ref<UndoAction[]>` — max 20 entries (shift oldest when full)
- `canUndo: ComputedRef<boolean>`
- `currentToast: Ref<{ description: string, timer: number } | null>`

**Exported functions:**
- `pushUndo(action: UndoAction)` — adds to stack, triggers toast
- `performUndo()` — peeks the last action, calls `restoreBlocks(scheduleDate, previousBlocks)` to restore state, pops the action only after success, triggers toast "Undone: {description}"
- `snapshotBlocks(): TimeBlock[]` — utility to deep-clone the current blocks array before a mutation (caller captures this before making changes)

**Keyboard shortcut:**
- `onMounted()`: register `keydown` listener for `Ctrl+Z` (or `Cmd+Z` on Mac) → calls `performUndo()`
- `onUnmounted()`: remove listener
- Ignore the shortcut while focus is inside `input`, `textarea`, `select`, or `[contenteditable]`

**Toast management:**
- On `pushUndo()`: set `currentToast` with description + start 8-second auto-dismiss timer
- On undo from toast click: clear current toast, show brief "Undone" confirmation toast
- Clear any active timer before replacing/dismissing a toast and on component unmount

### Integration points — all existing mutations must push to undo:

| Mutation | File | What to capture |
|----------|------|-----------------|
| Drag-and-drop | `useDrag.ts` → `endDrag()` | Snapshot before drag → push with type "drag" |
| Inline title edit | `TimeBlock.vue` → `saveTitle()` | Snapshot before PATCH → push with type "edit" |
| Checkbox toggle | `TimeBlock.vue` → `toggleCompleted()` | Snapshot before PATCH → push with type "toggle" |
| Add block | `AddBlockForm.vue` → `handleSubmit()` | Snapshot before POST → push with type "add" |
| Delete block | `TimeBlock.vue` → `handleDelete()` | Snapshot before DELETE → push with type "delete" |

Each mutation site follows the pattern:
1. `const snapshot = snapshotBlocks()`
2. Perform the API call
3. On success: `pushUndo({ description, type, previousBlocks: snapshot, scheduleDate })`

**Undo restore mechanism:**
- `performUndo()` sends the full `previousBlocks` snapshot to the new `POST /api/schedules/<str:date>/blocks/restore/` endpoint
- Since undo must handle add/delete (not just reorder), a dedicated **restore endpoint** is cleaner:
- Do not permanently pop the undo action until restore succeeds. If restore fails, keep the action on the stack and show an error toast/message.

### New endpoint: `POST /api/schedules/<str:date>/blocks/restore/`

**File:** `backend/schedules/api.py` — add `restore_blocks()` view function

Accepts `{ "blocks": TimeBlock[] }` as the full desired block state for a schedule date. Inside a transaction:
1. Validate each snapshot block (`title`, `start_time`, `end_time`, `category`, `is_completed`, `sort_order`) with the same rules as create/update
2. Validate the snapshot's final schedule has no overlaps
3. Delete all existing blocks for that schedule
4. Recreate blocks from the provided snapshot
5. Return the new block list

This makes undo atomic and handles all mutation types uniformly.

**ID behavior:** Do not rely on restored blocks preserving their previous database IDs. The restore endpoint may ignore incoming `id` values, recreate rows with new IDs, and return the new canonical block list. Frontend state should use the returned list / Inertia reload as source of truth after restore.

**File:** `backend/day_forge/urls.py` — add route `path("api/schedules/<str:date>/blocks/restore/", api.restore_blocks)`

**File:** `frontend/src/composables/useSchedule.ts` — add `restoreBlocks(date, blocks)` method

## Test Coverage To Add

### Backend tests
- `reorder_blocks()` success updates multiple blocks and returns the full ordered block list.
- Duplicate IDs in a reorder request return `400` and do not update anything.
- Cross-user and cross-schedule reorder attempts return `403`/`400` and do not update anything.
- Reorder rejects a final state that overlaps unchanged blocks.
- Reorder rejects malformed times, non-5-minute times, `start_time >= end_time`, and invalid `sort_order`.
- Reorder is atomic: if one update is invalid, no submitted block changes.
- `restore_blocks()` restores after add/delete snapshots and returns a canonical block list.
- `restore_blocks()` rejects invalid categories, empty titles, invalid times, overlaps, and cross-user schedule access.
- CSRF coverage mirrors the existing create/update/delete tests for both new endpoints.

### Frontend tests
- `useSchedule` sends `reorderBlocks()` and `restoreBlocks()` with CSRF headers and triggers the expected reload behavior.
- Conflict-resolution utility handles simple overlap, cascade through multiple blocks, no-op drop, and drop invalid because it exceeds `DAY_END`.
- `useUndo` enforces the max-20 stack, ignores shortcuts in editable fields, keeps the action if restore fails, and cleans up keyboard/timer listeners.
- `UndoToast` renders the message, calls undo on click, auto-dismisses after 8 seconds, and clears timers on unmount.
- Drag UI renders the ghost preview, updates the time badge in 5-minute increments, applies real shifted preview positions, and cancels invalid drops.
