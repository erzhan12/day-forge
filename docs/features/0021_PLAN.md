# Feature 0021 â€” Todoist two-way integration (complete-task + live sync)

## Context

Feature 0020 shipped a **read-only** Todoist sidebar on the Schedule page
(`todoist_sync` backend app + fixed left sidebar). This feature adds the
two follow-ups (GitHub issue 62) that make it a usable two-way workflow:

- **PART A â€” Complete a task (Day Forge â†’ Todoist):** a checkbox on each
  sidebar row that closes/completes the task in Todoist via a new write
  endpoint, with optimistic UI (row disappears immediately, rolls back on
  failure) and cache invalidation so the completed task does not reappear
  on the next fetch.
- **PART B â€” Live sync (Todoist â†’ Day Forge):** a manual **Refresh**
  button in the sidebar header (always visible) that force-refetches the
  current date bypassing the cache. Minimum viable (and the full V1 scope) =
  manual refresh + cache invalidation on complete. Short polling is
  **deferred to future work** â€” not built in this feature (see B3).

The Todoist v1 API host is `settings.TODOIST_BASE_URL`
(`https://api.todoist.com/api/v1`). All request URLs are built from that
base in `service.py` â€” the **only** module that decrypts the token
(`account.get_token()`). Views pass the `TodoistAccount` instance through
and never touch plaintext (service-boundary pattern).

### Out of scope (keep separate â€” do NOT build)

- Task creation / editing / due-date changes / re-open from Day Forge.
- OAuth (personal API token only).
- AI prompt integration with task text (no provider-egress of task text).
- Webhooks (documented as a future option only â€” see Trade-offs).
- Un-complete / undo-to-Todoist (the optimistic rollback is local UI only;
  it does NOT call a re-open endpoint).

### Acceptance criteria (verbatim from issue 62)

- User can mark a task done from the Day Forge Todoist sidebar; it
  disappears locally and is completed in Todoist.
- User completing a task in the Todoist app/web sees it removed in Day
  Forge within a reasonable time (manual refresh and/or polling â€” not
  "up to 5 minutes cache only").
- No regression to the read-only connect/disconnect flow or the overdue
  carryover filter (`carry_overdue=1` / `{date} | overdue`).

## PART B trade-offs (decision)

| Option | Latency | Ops cost | Decision |
|--------|---------|----------|----------|
| Manual refresh (force-refetch, cache-bust) | On click | None | **Ship (MVP)** |
| Short polling (every N s while mounted+open) | â‰¤ N s | Provider quota; battery | **Future work â€” NOT built in 0021** (see B3) |
| Cache-TTL reduction | â‰¤ TTL | None | Not sufficient alone; leave `TODOIST_CACHE_TTL_SECONDS` default 300 |
| Webhooks (Todoist push â†’ invalidate) | Near-real-time | Endpoint, signature verify, infra | Out of scope (documented future) |

**Chosen:** manual Refresh button (always visible) + cache invalidation on
complete = the **full V1 scope**. Short polling is **not built** in feature
0021 â€” no `TODOIST_POLL_INTERVAL_SECONDS` setting and no `setInterval` ship
(a backend-only flag with no frontend wiring would be a dead knob). The
deferred design is documented in B3 for a future feature.

### A2. Cache invalidation â€” `backend/todoist_sync/cache.py`

The cache key embeds `account.updated_at.isoformat()`. A successful
complete must not leave the just-completed task readable for up to the
TTL. There are two correctness-equivalent options; pick **bump the account
version** because it invalidates **all** date/scope entries at once with no
key-enumeration:

- Add **`invalidate_tasks(account) -> None`** to `cache.py`. The simplest
  correct implementation is to call `account.save()` (no `update_fields`)
  so `auto_now` bumps `updated_at` â€” every existing `todoist_tasks:*` key
  for that user becomes unreachable (versioned-key design; see the module
  docstring). To keep cache concerns inside `cache.py` and avoid a stray
  `save()` in the view, the view calls `cache.invalidate_tasks(account)`
  which performs `account.save()`.
  - **auto_now footgun (critical):** the `save()` MUST be a plain
    `account.save()` with **no** `update_fields`, or include
    `"updated_at"` â€” otherwise the key does not rotate and the completed
    task is still served. This is the exact footgun documented in
    `cache.py` and `models.py:set_token`.
- Alternative (documented, not chosen): patch the cached list in place by
  filtering out `task_id` for every `(date, scope)` key â€” rejected because
  it requires enumerating keys, which the versioned design deliberately
  avoids.

### A3. View + route â€” `backend/todoist_sync/views.py`, `backend/day_forge/urls.py`

Add a new view **`complete(request, task_id)`** in `views.py`:

- Decorators `@login_required`, `@require_http_methods(["POST"])`.
- **Explicit view ordering** (so the 503-vs-anything-else precedence is
  deterministic and testable):
  1. Resolve the account: `request.user.todoist_account`; on
     `TodoistAccount.DoesNotExist` return the `503` envelope
     `"No Todoist account configured"` (mirror the `tasks` view).
  2. `service.complete_task(account_row, task_id)` (the `try` ladder below).
- **No `reject_oversized_body` on this view** (decision â€” F-1-2): the id is
  in the URL path and the view parses **no** request body, so there is no
  oversized-body parse to guard. Django's global `DATA_UPLOAD_MAX_MEMORY_SIZE`
  already caps any stray body at the WSGI/ASGI boundary before the view
  runs. Unlike `POST /api/todoist/account/` (which `json.loads(request.body)`
  and therefore needs the explicit guard), the complete view never reads
  `request.body`. Omitting it keeps the precedence simple: there is no
  413 path, only `503` (no account) â†’ `2xx`/service-error.
- `try: service.complete_task(account_row, task_id)`:
  - `except ImproperlyConfigured:` â†’ log + `500` config-shaped envelope
    `"Todoist service is misconfigured. Contact the administrator."`
    (reuse the exact pattern/message from the `tasks` view).
  - `except service.TodoistError as e:` â†’ `_service_error_response(e)`
    (maps `TodoistAuthError â†’ 401`, `TodoistTimeoutError â†’ 504`,
    `TodoistProviderError â†’ 502` via the existing `_SERVICE_ERROR_STATUS`).
- On success: `cache.invalidate_tasks(account_row)`, then return
  `JsonResponse({"ok": True})` (200). (Returning the refreshed task list
  would require a second provider round-trip on the same request â€” the
  frontend already removed the row optimistically, so a bare ack is
  sufficient and cheaper.)

Add the route in `backend/day_forge/urls.py` next to the existing two
Todoist routes (import `from todoist_sync import views as todoist_views`
already present):

- `path("api/todoist/tasks/<str:task_id>/complete/", todoist_views.complete, name="todoist_complete")`.
- **Route-ordering note:** Django matches in order. `<str:task_id>` would
  also match a date string, but the new route's trailing `/complete/`
  literal disambiguates it from `api/todoist/tasks/<str:date>/` (which has
  no trailing segment). Register the `complete` route **adjacent** to the
  `tasks` route; verify no path collision (the existing `tasks` route ends
  at `<str:date>/`, the new one at `<str:task_id>/complete/`).

### A4. Frontend composable â€” `frontend/src/composables/useTodoist.ts`

Add **`completeTask(taskId: string): Promise<void>`** to the composable's
returned object. Optimistic update with rollback:

1. Capture **only the removed task + its index** for a surgical rollback â€”
   **not** a whole-list snapshot:
   `const idx = state.tasks.findIndex((t) => t.id === taskId);`
   `const removed = idx >= 0 ? state.tasks[idx] : null;`
   (Do **not** `const previous = state.tasks` â€” restoring a stale whole-list
   reference on failure would clobber a concurrent `refreshTasks` commit;
   see step 5 and the concurrency note.)
2. Optimistically remove the row:
   `state.tasks = state.tasks.filter((t) => t.id !== taskId)`.
3. `POST /api/todoist/tasks/${taskId}/complete/` via `requestJson`
   (**GET footgun does not apply** â€” this is a POST; pass `undefined` as
   the body third arg since there is no payload, options/signal optional).
4. On `result.ok` â†’ **idempotently re-filter** the optimistic state:
   `state.tasks = state.tasks.filter((t) => t.id !== taskId)`. This is not a
   no-op: if a concurrent `refreshTasks` commit landed between the optimistic
   remove and the success ack, the just-closed task may have been
   **re-inserted** by the refresh (the provider GET can race ahead of the
   `close` being processed). Re-filtering on success guarantees the closed
   row stays gone for the rest of the session regardless of refresh
   interleaving.
5. On any non-ok / thrown error â†’ **surgical roll back**: re-insert **only**
   the attempted task into the **current** list, and only if it is absent
   (a concurrent refresh may already have re-added it):
   `if (removed && !state.tasks.some((t) => t.id === taskId)) {`
   `  const next = state.tasks.slice();`
   `  next.splice(Math.min(idx, next.length), 0, removed); state.tasks = next; }`
   Then set `state.error` via the existing `statusToMessage(result.status)`
   fallback to `extractErrorMessage`. A `401` surfaces the reconnect
   message; `502/504` the unavailable message; a no-status network error
   falls through to `extractErrorMessage`.
   - **Why not `state.tasks = previous`:** a whole-list restore resurrects a
     stale snapshot. Scenario: complete A snapshots `[A,B]`, removes A â†’
     `[B]`; a `refreshTasks` commits `[B,C]`; then complete A fails â€” a
     wholesale restore to `[A,B]` **loses C and resurrects A**. The surgical
     re-insert above only puts A back (if still missing) and never drops C.
   - Do **not** touch `connected` / `statusKnown` here â€” completion failure
     does not change connection state (the sidebar is already mounted).
   - **Known limitation (inherited from 0020, accepted for V1):**
     `TodoistTasksPanel.vue` renders `v-if="error"` before the list
     `v-else-if`, so setting `state.error` on a failed complete **hides the
     whole task list** (rows restored by rollback are not shown) until the
     next Retry/Refresh clears `state.error`. This is the pre-existing
     fetch-error rendering behaviour, not new to this feature. Accepted as-is
     for V1; inline per-row complete errors (keep the list visible, mark only
     the failed row) are a **future follow-up**, out of scope here.
6. Concurrency: a complete-in-flight must not be clobbered by a simultaneous
   `fetchTasks`/`refreshTasks`, and vice-versa. `fetchTasks` reassigns
   `state.tasks` wholesale on commit; therefore `completeTask` must operate
   on whatever the **current** `state.tasks` is at success/failure time (the
   surgical re-filter in step 4 and re-insert in step 5), never on a snapshot
   captured at call time. No shared lock is needed â€” the optimistic remove +
   commit-guarded fetch + surgical rollback converge to the server-truth list
   on the next fetch.

## PART B â€” Live sync (Todoist â†’ Day Forge)

### B1. Force-refetch (cache bypass) â€” backend

The current `tasks` view always reads the cache first. Add a cache-bypass
path keyed on a query param so the manual Refresh (and any future polling)
gets fresh provider data:

- In `backend/todoist_sync/views.py` `tasks` view: read
  `force = request.GET.get("refresh") == "1"`. When `force` is true, skip
  the `get_cached_tasks` short-circuit (do **not** read the cache) and go
  straight to `service.fetch_tasks_for_date`; still **write** the result
  into the cache via `set_cached_tasks` so subsequent non-forced reads are
  warm. This is additive and does not change the default cached behavior
  (no regression to the read-only flow).
- The `carry_overdue=1` param continues to work alongside `refresh=1`
  (both are independent query flags; `filter_scope` is computed from
  `carry_overdue` exactly as today â€” no change to the overdue carryover
  filter, satisfying the no-regression acceptance criterion).

### B2. Manual refresh â€” frontend

`frontend/src/composables/useTodoist.ts`:

- Add **`refreshTasks(date: string): Promise<void>`** â€” same body as
  `fetchTasks` but appends `refresh=1` to the URL (alongside the existing
  conditional `carry_overdue=1` for browser-local today). Reuse the same
  dual commit-guard (`latestRequestedTaskDate` + `tasksRequestSeq`) and
  `AbortController` so a refresh that is superseded by a date change is
  dropped. The simplest DRY implementation: extract the existing
  `fetchTasks` body into an internal `_fetchTasks(date, { force, silent })`
  helper and have both `fetchTasks` and `refreshTasks` delegate to it
  (force flag appends `refresh=1`).
  - URL build: `/api/todoist/tasks/${date}/?` + params built from
    `[carry_overdue=1 if browser-today]` and `[refresh=1 if force]` joined
    with `&`. Preserve the existing `carry_overdue` logic exactly.
  - **No skeleton flash on refresh (critical UX â€” do not regress):**
    `TodoistTasksPanel.vue` gates the list on `v-else-if="!loading && ..."`
    and renders the skeleton on `loading`, so a refresh that sets
    `state.loading = true` would **hide every existing row** and show the
    skeleton until the fetch returns â€” worse than a silent background
    refresh. `refreshTasks` must call `_fetchTasks(date, { force: true,
    silent: true })`; when `silent` is set, `_fetchTasks` **skips**
    `state.loading = true` (it still clears `state.error` and runs the
    commit-guard / abort logic). The initial `fetchTasks` keeps
    `silent: false` (skeleton on first load is correct). The optimistic
    list stays visible during the refetch and is replaced atomically on
    commit. Add a test asserting `state.loading` stays `false` across a
    `refreshTasks` call while rows remain rendered.

`frontend/src/pages/Schedule.vue`:

- `function refreshTodoist() { todoist.refreshTasks(props.date) }` bound to
  the sidebar's `@refresh`.

### B3. Short polling â€” FUTURE WORK (not built in this feature)

**Decision:** short polling is **deferred entirely** â€” no setting, no
`setInterval`, no frontend constant ships in feature 0021. Manual Refresh
(B1 + B2) is the V1 live-sync mechanism and already satisfies the issue-62
acceptance criterion ("manual refresh **and/or** polling â€” not 5-minute
cache only"). A backend `TODOIST_POLL_INTERVAL_SECONDS` setting was
explicitly **rejected for V1**: it could not affect behaviour without also
wiring a real prop through the server prop builder + `defineProps`, so
shipping it alone would be a dead flag (validates at boot, documented, but
never polls). It is omitted to avoid that footgun.

**To add polling later (out of scope here):**
- Add `TODOIST_POLL_INTERVAL_SECONDS` to `settings.py` (default `0`, `< 0`
  â†’ `ValueError` mirroring the existing `TODOIST_REQUEST_TIMEOUT` /
  `TODOIST_CACHE_TTL_SECONDS` guards).
- Inject a `todoist_poll_interval` prop in `schedule_view`, add it to
  `Schedule.vue`'s `defineProps`, and read the setting there (today
  `defineProps` declares only `schedule, blocks, date, auto_draft_pending,
  has_template_for_type, slot_type` â€” no config/settings prop and no
  `usePage()` read, so the value has no path to the frontend without this).
- When the prop is `> 0` AND the sidebar is mounted (`isWide &&
  todoist.state.connected`) AND open (`todoistSidebarOpen`), run a
  `setInterval(() => todoist.refreshTasks(props.date), N*1000)`; clear it on
  `onBeforeUnmount`, on `watch(() => props.date)`, and when the sidebar
  closes. The existing `refreshTasks` `AbortController` already cancels an
  in-flight poll on date change. The `refresh=1` cache-bypass (B1) is the
  reusable building block; it is built in this feature and works for manual
  Refresh today.

## Algorithms (step-by-step)

### Optimistic complete with rollback (frontend)

1. User clicks the row checkbox â†’ `TodoistTasksPanel` emits
   `complete(task.id)` â†’ `TodoistSidebar` re-emits â†’ `Schedule.completeTask`
   â†’ `useTodoist.completeTask(taskId)`.
2. `completeTask` snapshots **only the removed task + its index** â€”
   `idx = state.tasks.findIndex(t => t.id === taskId)`,
   `removed = idx >= 0 ? state.tasks[idx] : null` â€” then sets
   `state.tasks = state.tasks.filter(t => t.id !== taskId)` (row vanishes).
   It does **not** snapshot the whole list (`previous = state.tasks`): a
   whole-list restore on failure would clobber a concurrent
   `refreshTasks` commit (see A4 and the Â§Failure scenario below).
3. `POST /api/todoist/tasks/${taskId}/complete/`.
4. **Success (`ok`):** idempotently re-filter the **current** list:
   `state.tasks = state.tasks.filter(t => t.id !== taskId)`. A concurrent
   `refreshTasks` commit may have re-inserted the row before the close ack
   landed; re-filtering guarantees it stays gone. Server has also
   invalidated the cache, so the next `fetchTasks`/`refreshTasks` returns
   server-truth without the completed task.
5. **Failure:** **surgical re-insert** into the **current** list â€” only if
   the task is absent (a refresh may have re-added it):
   `if (removed && !state.tasks.some(t => t.id === taskId)) { const next =
   state.tasks.slice(); next.splice(Math.min(idx, next.length), 0, removed);
   state.tasks = next }` â€” then `state.error` from
   `statusToMessage(status)` / `extractErrorMessage`. A whole-list
   `state.tasks = previous` here would resurrect a stale snapshot: complete
   A from `[A,B]` â†’ `[B]`, refresh commits `[B,C]`, A fails â†’ restore `[A,B]`
   **drops C and resurrects A**. The surgical re-insert puts only A back (if
   still missing) and never drops C.

### Force-refetch (backend `refresh=1`)

1. `GET /api/todoist/tasks/<date>/?refresh=1[&carry_overdue=1]`.
2. View parses date, resolves account (503 if absent), computes
   `filter_scope` from `carry_overdue` (unchanged).
3. `refresh=1` â†’ **skip** the `get_cached_tasks` read; call
   `service.fetch_tasks_for_date`.
4. Normalise â†’ `set_cached_tasks` (warm the cache) â†’ return `{tasks: [...]}`.
