# Feature 0028 — Desktop notifications at block start/end (issue #100)

## Context

Show a **browser desktop notification** (Chrome Notification API) when the wall
clock crosses a time block's **start** or **end** boundary, so the user is
alerted even without watching the schedule — alongside the existing sound
chimes from issue #56 / feature 0019. Frontend-only Vue 3 feature. Opt-in
(default **off**), persisted in localStorage on this device. Fires in both
foreground and background tabs, but **only while the Schedule page is mounted**
(reuses the same `useNowMinutes` 60s sampler and boundary-cross detector as
sound — no Service Worker, no closed-tab alerts). Sound and desktop
notifications are **independent toggles**; both may fire on the same boundary
when both are enabled.

### Locked decisions (verbatim from issue #100)

- **Settings:** separate opt-in toggle next to sound ("Desktop notifications"),
  independent of sound.
- **When to show:** always — foreground and background tab.
- **Lifecycle:** only while Schedule page is open; reuse the #56 boundary
  detector; no Service Worker / closed-tab alerts.
- **Copy:** start + end notifications; language follows UI locale (today:
  English strings; wire through i18n if/when it lands).
- **Click:** focus the existing Day Forge tab (`notification.onclick →
  window.focus()`).
- **Permission:** call `Notification.requestPermission()` when the toggle is
  turned **ON** (user gesture); persist enabled only when permission is
  `granted`; if denied/default, leave toggle off with an inline hint.
- **Scope:** MVP — Notification API + Settings toggle + reuse existing
  detector; tab must stay open.

### Suggested copy (English)

- **Start:** title `"Block started"`, body `"{block.title} · {start}–{end}"`
  (times are the block's `start_time` / `end_time` `"HH:MM"` strings).
- **End:** title `"Block ended"`, body `"{block.title} finished"`.

### Out of scope

Service Worker, push notifications, per-block mute/snooze, merging sound +
desktop into one master "Alerts" control, server-side preference persistence,
notifications when Settings (or any non-Schedule page) is open.

### Grounded facts the design depends on

- **Sound baseline (feature 0019):** `useSoundNotifications.ts` owns the
  boundary-cross detector (`lastSeenMinute`, `fired` Set, `inWindow` half-open
  `(prev, now]` logic), piggybacking `nowMinutes` / `nowDate` from
  `useNowMinutes`. `Schedule.vue` wires:
  `useSoundNotifications(nowMinutes, nowDate, getBlocks)` where
  `getBlocks = () => props.blocks`. Setting persists via
  `soundNotificationStorage.ts` (`day-forge:sound-notifications:enabled`,
  strict-only-on-`true`). Settings UI: `SoundNotificationToggle.vue` in
  `Settings.vue`, calling `useSoundNotificationSetting()` from the same
  composable module.
- **`useNowMinutes`** (`frontend/src/composables/useNowMinutes.ts`): 60s
  interval while `viewedDate === todayString()`; off-today nulls
  `nowMinutes`/`nowDate`. Inherited limitation: tab left open across midnight
  stops boundary detection until date navigation (documented in 0019).
- **Inertia page swap:** Settings ↔ Schedule remounts page components; each
  mount re-reads localStorage. Separate composable instances on the two pages
  are fine — persisted flags are the source of truth.
- **Time helpers:** `timeToMinutes` / `minutesToTime` in
  `frontend/src/utils/scheduleTime.ts`; block times are `"HH:MM"` strings
  (`TimeBlock` in `frontend/src/types/index.ts`). `SoundEventType =
  "start" | "end"` already exists.
- **Tests:** `frontend/tests/*.test.ts`, vitest + `@vue/test-utils`, global
  mocks via `vi.stubGlobal`. Run: `cd frontend && npm test`.
- **No i18n today:** grep shows no vue-i18n; keep English strings in a small
  constants module so a future i18n pass can swap the lookup.

## Phase 2 — Extract shared boundary detector

File: `frontend/src/composables/useBlockBoundaryDetector.ts`.

Move the detector state machine from `useSoundNotifications.ts` verbatim in
behaviour:

### Public signature

```
useBlockBoundaryDetector(
  nowMinutes: Ref<number | null>,
  nowDate: Ref<string | null>,
  getBlocks: () => TimeBlock[],
  options: {
    enabled: Ref<boolean>
    onBoundary: (event: {
      type: SoundEventType
      block: TimeBlock
      date: string
      boundaryMinutes: number
    }) => void
  },
): void
```

Side-effect only (no return). The callback receives enough context for both
consumers: sound needs `type`; desktop needs `type`, `block`, and `date` for
tag/copy.

### Internal algorithm (unchanged from 0019)

On each `nowMinutes` change:

1. **Gate:** if `!enabled.value`, return immediately **without** advancing
   `lastSeenMinute` (preserves re-enable no-backfill behaviour).
2. **Off-today:** if `nowMinutes` or `nowDate` is `null`, return.
3. Let `prev = lastSeenMinute`, `now = nowMinutes.value`, `date = nowDate.value`.
4. **Eligible window** `inWindow(boundary)`:
   - `prev === null` → `boundary === now` (first tick exact-only, no back-fill).
   - `now <= prev` → `false` (backward clock step).
   - else → `boundary > prev && boundary <= now` (half-open `(prev, now]`).
5. For each block from `getBlocks()`:
   - `s = timeToMinutes(block.start_time)`, `e = timeToMinutes(block.end_time)`.
   - Keys: `` `start:${block.id}:${date}:${s}` `` and
     `` `end:${block.id}:${date}:${e}` `` (time in key so re-timed blocks
     re-fire — same footgun documented in RULES.md for sound).
   - If `inWindow(s)` and key not in `fired`, call
     `onBoundary({ type: "start", block, date, boundaryMinutes: s })`, add key.
   - Same for end.
6. `lastSeenMinute = now`.

### Reset watches (unchanged)

- `watch(nowDate, …)` → clear `fired`, `lastSeenMinute = null`.
- `watch(enabled, on => { if (on) lastSeenMinute = null })` — no back-fire
  after a disabled gap.

### Refactor `useSoundNotifications.ts`

- `useSoundNotificationSetting()` — unchanged.
- `useSoundNotifications(...)` — call `useBlockBoundaryDetector` with
  `enabled` from the setting helper and `onBoundary: ({ type }) => playSound(type)`.
- Remove duplicated `lastSeenMinute` / `fired` / `watch(nowMinutes)` from this
  file.

Both `useSoundNotifications` and `useDesktopNotifications` will each invoke
`useBlockBoundaryDetector` with their own `enabled` ref and callback. Two
`watch(nowMinutes)` handlers on the same ref is intentional and cheap; each
detector maintains independent `lastSeenMinute`/`fired` so disabling one
channel does not advance the other's cursor (matching current sound-only
semantics per channel).

## Phase 3 — Desktop notifications composable

File: `frontend/src/composables/useDesktopNotifications.ts`.

### `useDesktopNotificationSetting()`

Returns `{ enabled, setEnabled, permissionDenied, notSupported }`, all
`Ref<boolean>` except `setEnabled`. `permissionDenied` drives the "browser
blocked it" hint; `notSupported` drives the **distinct** "this browser has no
Notification API" hint. Two separate flags because the copy differs and a bare
`permissionDenied` boolean cannot carry both messages — the component must not
show "allow in site settings" for a browser that has no notification support at
all. Both default `false`. **`notSupported` is initialised unconditionally at setup:
`notSupported.value = (typeof Notification === "undefined")`** — independent of
any stored flag, so an unsupported browser shows the disabled switch + "doesn't
support" hint on first render even with absent/`false` storage (do NOT gate it
on the storage-clear path). Support is immutable per page load, so it is never
recomputed or cleared afterward.

`setEnabled(v: boolean)`:

1. If `v === false`: **`++requestSeq`** (invalidates any in-flight
   `setEnabled(true)` continuation — this bump is a normative step, not
   optional), then write `false`, `enabled = false`, `permissionDenied = false`,
   return.
2. If `v === true`:
   - Bump the request token first: `const token = ++requestSeq`.
   - Guard: if `typeof Notification === "undefined"`, set `enabled = false`,
     `notSupported = true`, `permissionDenied = false`, and return (the switch is
     also disabled in Phase 4; the composable contract is: unsupported ⇒ off + a
     hint, never a silent no-op).
   - `const result = await Notification.requestPermission()` (async; called from
     the checkbox `@change` handler which is a user gesture). **Wrap in
     `try/catch`** — the legacy callback form (and some hardened browsers) can
     reject/throw; a throw MUST be treated exactly like `denied` below, never
     left as an unhandled rejection. The Phase 4 toggle `await`s `setEnabled`
     (to then resync the DOM checkbox — see Phase 4), so an escaping rejection
     would both surface as an unhandled promise AND skip the mandatory
     `el.checked = enabled.value` resync, leaving `enabled`/`permissionDenied` in
     an indeterminate state. Normalise a caught throw to `result = "denied"`.
   - **Stale-request guard (mandatory) — check FIRST, unconditionally:**
     immediately after the await (and after normalising a throw), if
     `token !== requestSeq` a newer `setEnabled` call has superseded this one —
     **`return` immediately, touching neither storage nor `enabled` nor
     `permissionDenied`.** This must be its own early return, NOT folded into
     the granted/denied branches below: a late `granted` (or `denied`) arriving
     after the user already toggled `false` must leave the final off-state
     **exactly** as `setEnabled(false)` left it (`permissionDenied === false`) —
     it must not resurrect `true` AND must not stamp a spurious "permission
     blocked" hint on a checkbox the user deliberately turned off. (Scenario: a
     click during the browser's non-modal permission prompt — `setEnabled(false)`
     runs synchronously to completion, then this suspended continuation resumes.)
   - Only past that early return: if `result === "granted"`,
     `writeDesktopNotificationsEnabled(true)`, `enabled = true`,
     `permissionDenied = false`. Else (`denied` / `default`): do **not** persist
     `true`; `enabled = false`; `permissionDenied = true` (show hint: permission
     blocked — user must change site settings in Chrome).

On mount, `enabled` initialises from `readDesktopNotificationsEnabled()` **and**
re-validates runtime permission: if storage says `true` but
`Notification.permission !== "granted"` (user revoked in browser settings),
treat as off **and clear the stale flag** (`writeDesktopNotificationsEnabled(false)`).
**Guard the read with `typeof Notification === "undefined"` FIRST** (same guard
`setEnabled` / `showDesktopNotification` use): on a non-supporting browser a
hand-edited storage `true` must resolve to off **without** dereferencing
`Notification.permission` (which would throw) — and **must clear the stale flag
too** (`writeDesktopNotificationsEnabled(false)`), consistent with the revoke
path so no stale `true` key survives either miss. (`notSupported` is already
`true` from the unconditional setup init above — the mount path does not need to
re-set it.)

### `showDesktopNotification(...)`

```
function showDesktopNotification(
  type: SoundEventType,
  block: TimeBlock,
  date: string,
  boundaryMinutes: number,
): void
```

Steps:

1. Guard: `typeof Notification === "undefined"` → return.
2. Guard: `Notification.permission !== "granted"` → return.
3. `try`:
   - `const n = new Notification(desktopNotificationTitle(type), {
       body: desktopNotificationBody(type, block),
       tag: `day-forge:${type}:${block.id}:${date}:${boundaryMinutes}`,
     })`
   - `n.onclick = () => { window.focus(); n.close() }`
4. `catch` — swallow (never throw into Schedule render path).

Tag dedupes OS-level stacking for the same boundary; independent of the
in-memory `fired` Set (belt-and-suspenders). **The tag embeds
`boundaryMinutes` to stay aligned with the detector's `fired` key
(`useSoundNotifications.ts:161` includes the minute):** a re-timed block
deliberately re-fires via a new `fired` key, so its OS tag must also differ or
the OS would replace/coalesce the new alert under the old minute's tag.

### `useDesktopNotifications(...)`

Same arity as sound:

```
useDesktopNotifications(
  nowMinutes: Ref<number | null>,
  nowDate: Ref<string | null>,
  getBlocks: () => TimeBlock[],
): { enabled; setEnabled; permissionDenied; notSupported }
```

Internally: `useDesktopNotificationSetting()` + `useBlockBoundaryDetector`
with `onBoundary: ({ type, block, date, boundaryMinutes }) =>
showDesktopNotification(type, block, date, boundaryMinutes)`.

Only show when `enabled.value` **and** permission granted (detector `enabled`
ref covers the setting; `showDesktopNotification` double-checks permission).

## Phase 4 — Settings UI

### `frontend/src/components/DesktopNotificationToggle.vue`

Mirror `SoundNotificationToggle.vue` structure (section heading, subtitle,
switch). Differences:

- Import `useDesktopNotificationSetting` from `useDesktopNotifications.ts`;
  destructure `{ enabled, setEnabled, permissionDenied, notSupported }`.
- `@change` handler (async): capture the element, `await setEnabled(el.checked)`,
  **then unconditionally resync the DOM to the model: `el.checked = enabled.value`.**
  This is **mandatory**, not cosmetic — `:checked="enabled"` is a one-way bind,
  so when a click resolves to `denied`/`default` and `enabled` was already
  `false` (a **repeated** denied attempt, where neither `enabled` nor
  `permissionDenied` changes), Vue skips the patch and the user-checked box
  stays visually ON while the setting is OFF. The explicit `el.checked =
  enabled.value` write after the await corrects the DOM even when no reactive
  ref changed. (A `key`-bump or `nextTick` reset works too; the explicit write
  is simplest.)
- Bind `:checked="enabled"` — when permission denied/unsupported, the resync
  above keeps the checkbox unchecked.
- Hints (mutually exclusive, `notSupported` takes precedence):
  - `notSupported` → disable the switch and render "This browser doesn't support
    desktop notifications." (the **distinct** unsupported copy — never the
    site-settings message).
  - else `permissionDenied` → muted inline hint "Browser blocked notifications.
    Allow them in site settings and try again."

Section title: **"Desktop notifications"**. Subtitle explains block start/end
alerts while the schedule is open, device-local, off by default.

  stays off.

## Edge cases

| Case | Behaviour |
|------|-----------|
| Toggle on, permission `default` | `requestPermission()` from click; granted → on; else off + hint |
| Toggle on, permission `denied` | Stay off + hint (browser may not re-prompt) |
| Permission revoked after enable | On mount, re-check → clear storage, show off |
| `Notification` undefined (non-supporting browser) | `notSupported` → switch disabled + "doesn't support" hint; clears any stale storage `true`; no throw |
| Repeated denied clicks | `@change` writes `el.checked = enabled.value` after await so the DOM stays unchecked even when no ref changes |
| Sound on, desktop off | Sound fires; desktop detector gated, no notification |
| Both on | Both fire same boundary (independent side effects) |
| Two blocks, same start minute | Both notifications (distinct tags per `block.id`) |
| Background-tab minute coalescing | Shared `(prev, now]` window fires all leapt boundaries |
| Past/future date navigation | `nowMinutes`/`nowDate` null → silent |
| DND / OS suppression | Best-effort; no app-level handling |
| Midnight rollover | Inherited 0019 limitation — no fire until navigation |
