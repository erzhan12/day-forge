# Phase 4: AI Command Bar

Bottom-docked command bar that accepts natural language in English or Russian (e.g. `"add standup at 10:00 for 15 min"`, `"move gym to 18:00"`, `"добавь звонок в 10 на 30 минут"`) and mutates the current day's schedule via OpenAI. The AI returns a structured JSON list of `add` / `move` / `remove` / `resize` actions; the backend validates and applies them atomically, logs the interaction, and returns the new block list. Each successful command pushes a snapshot to the existing undo stack. Failures show an inline error and degrade to manual editing.

Draft generation (`/api/ai/generate-draft/`) is explicitly **out of scope** — it belongs to Phase 5 along with templates and rules context.

## Phase 4B — Backend AI Service

### New file: `backend/ai/schemas.py`

Plain Python constants and validators. **No Pydantic** — the project has no Pydantic dependency today and the existing `schedules/api.py` does validation by hand; stay consistent.

Exports:

- `COMMAND_RESPONSE_JSON_SCHEMA` — the JSON Schema object used to validate the model's response. Passed to post-parse validators (`validate_response_envelope` / `validate_action_shape`) because we use `response_format={"type": "json_object"}` for provider compatibility rather than Structured Outputs. Matches PRD §6.3:
  - `actions: array` where each item is `{ type, task_id?, title?, start_time?, end_time?, category? }`.
  - `type` enum: `"add" | "move" | "remove" | "resize"`.
  - `category` enum matches `TimeBlock.Category.values`.
  - `start_time` / `end_time` string pattern `^([01]\d|2[0-3]):[0-5]\d$`.
  - `explanation: string` (short human-readable summary of what the AI did).
  - `additionalProperties: false` on every object, and `strict: true` enabled on the outer schema.
- `MAX_ACTIONS_PER_COMMAND = 20` — reject oversized plans before applying.
- `ALLOWED_ACTION_TYPES` — the set `{"add","move","remove","resize"}`.
- `validate_action_shape(action, allowed_categories) -> list[str]` — returns a list of per-action error strings. Used after parsing AI JSON; second line of defense because even with Structured Outputs we don't trust the provider.

### New file: `backend/ai/prompts.py`

Pure-function prompt builders, easy to unit-test without hitting the network:

- `SYSTEM_PROMPT: str` — static instructions to the model. Spells out: (a) valid action types and required fields per type, (b) 5-minute granularity, (c) working day window (06:00–23:00 from `scheduleTime` constants — mirror them as Python constants in this file to avoid coupling to the frontend), (d) category enum, (e) must return `task_id` (int block id) for `move` / `remove` / `resize`, (f) handle both English and Russian input, (g) never invent blocks that are not in the provided schedule when moving/removing/resizing, (h) if the command is ambiguous, return zero actions and use `explanation` to ask for clarification.
- `build_user_message(schedule, blocks, current_time_iso, user_command) -> str` — formats the dynamic context. Serializes each block as a compact line `id=<int> <HH:MM>-<HH:MM> <category> <is_completed> "<title>"` so the model can reference blocks by id. Includes the schedule date, today's weekday, and current clock time for time-aware commands.
- Template + rules + last-7-days context is deliberately **omitted** in Phase 4. Only Phase 5 needs it. Leave a one-line TODO comment so it's obvious where to splice that data in later.

### New file: `backend/ai/service.py`

Thin wrapper over the OpenAI SDK. One public function:

- `run_command(user_command: str, schedule, blocks, now) -> AICommandResult`

Algorithm:

1. Guard: if `settings.LLM_API_KEY` is empty, raise `AIUnavailableError`. The view translates this to a 503.
2. Trim and length-check `user_command` (`LLM_MAX_COMMAND_CHARS`). Empty → raise `AIInvalidInputError`.
3. Build messages via `prompts.build_user_message` + `SYSTEM_PROMPT`.
4. Call `client.chat.completions.create(model=settings.LLM_MODEL, messages=..., response_format={"type": "json_object"}, timeout=settings.LLM_REQUEST_TIMEOUT)`.
5. Parse the assistant message content as JSON. If parse fails or the schema check fails, raise `AIParseError` carrying the raw response text (so the view can still log it).
6. Return an `AICommandResult` dataclass: `{ raw_response_text, parsed_actions: list[dict], explanation: str }`.

Error classes (all subclass a module-local `AIError`):

- `AIUnavailableError` — no key, 503 at the view.
- `AIInvalidInputError` — empty / oversized user input, 400.
- `AITimeoutError` — SDK `APITimeoutError` / request timeout, 504.
- `AIProviderError` — any other SDK exception (auth, rate limit, network), 502.
- `AIParseError` — response wasn't valid JSON or didn't match the schema, 502.

**Do not** import this module at Django startup beyond the class/function defs — defer OpenAI client construction to inside `run_command` so tests can monkeypatch without needing the env var set.

### New file: `backend/ai/views.py` (replaces stub)

One view: `ai_command(request, date)`.

Routing: `POST /api/ai/schedules/<str:date>/command/`. Date-scoped because every command mutates a specific day; mirrors the shape of `restore_blocks`.

Decorators, same as existing endpoints: `@login_required`, `@require_http_methods(["POST"])`.

Algorithm:

1. `_reject_oversized_body(request)` (reuse helper from `schedules.api`). Move it into a new `backend/schedules/http.py` utility module if importing from `schedules.api` feels wrong; otherwise import directly.
2. Parse JSON body; require `{ "command": str }`. Validate length & type.
3. Parse `date` with the same `datetime.date.fromisoformat` logic used by `schedule_view` / `create_block`. Return 400 on malformed date.
4. Fetch-or-404 `Schedule` for this user + date. Lock it with `select_for_update()` inside the atomic block below.
5. Open `transaction.atomic()`:
   a. Re-fetch schedule + blocks under the lock.
   b. Call `service.run_command(command, schedule, blocks, now=timezone.localtime())`.
   c. Log an `AIInteraction` row **before** applying actions, with `actions_json=parsed_actions`. This guarantees we record intent even if application fails.
   d. Apply each parsed action (see §Action application below). On any validation failure: roll back the transaction and return 400 with a structured error.
   e. Re-serialize the full block list using the existing `_block_to_dict` helper from `schedules.api` (same format the other endpoints return).
6. Return `{ blocks: [...], explanation: str }` on success.
7. On `AIError`: log the interaction with `ai_response=<error text>` and `actions_json=[]`, return the mapped HTTP status. Failures are always logged — PRD §6.5.

### Action application — step-by-step

All four action types reduce to existing validated primitives already used by `create_block`, `block_detail` (PATCH / DELETE), and the domain invariants enforced inside `transaction.atomic()`. Reuse, do not reimplement:

- `add`: validate `title`, `start_time`, `end_time`, `category`; enforce 5-minute granularity and the no-overlap rule by computing `sort_order = max(existing) + 1` at the new start time, then `TimeBlock.objects.create(...)`. Run `full_clean()` same as `create_block`.
- `move`: resolve `task_id` → block (must belong to this schedule). Compute new `start_time` from the AI response; keep original duration if AI omitted `end_time` (common for "move to 18:00" commands). Update fields, `full_clean()`, save. Do not auto-shift overlapping blocks in Phase 4 — the drag feature already does cascade resolution; AI commands that overlap return a structured 400 so the user sees the conflict.
- `remove`: resolve `task_id` → block; `.delete()`. 404 if not in this schedule.
- `resize`: resolve `task_id` → block; update `end_time` (or `start_time` if AI returns one); validate as in `move`.

Overlap and `start_time < end_time` validation is centralized — factor a `_apply_ai_action(schedule, blocks_by_id, action)` helper inside `ai/views.py` that returns an updated blocks list (or raises). Actions are applied sequentially in the order the AI returns them; keep an in-memory `blocks_by_id` dict so a single batch like `{remove X, add Y at the same time}` works without spurious overlap errors.

**Guardrails:**

- Cross-user / cross-schedule `task_id` → 400, not 404, to avoid leaking which ids exist. Same principle as `block_detail`.
- Per-action validation errors return `{ errors: { action_index: "...", detail: "..." } }` and roll back everything. No partial application, same spirit as `reorder_blocks`.

## Phase 4C — Frontend Command Bar

### New composable: `frontend/src/composables/useAI.ts`

Mirrors `useSchedule.ts` style — wraps `apiFetch` and exposes reactive state plus one action.

**Exported state:**

- `isProcessing: Ref<boolean>` — true while a request is in flight, drives the spinner.
- `lastError: Ref<string | null>` — user-facing error message; cleared on next successful command or when the user starts typing.
- `lastExplanation: Ref<string | null>` — AI's `explanation` string to surface as a brief status line under the bar.
- `apiHealthy: Ref<boolean>` — flipped to `false` on a 503/502/504 response, back to `true` on any success. Drives the status indicator dot.

**Exported functions:**

- `submitCommand(date: string, command: string): Promise<ApiResult>` — POSTs to `/api/ai/schedules/<date>/command/`. Uses the existing `apiFetch` so CSRF + reload behaviour matches other mutations. Sets `isProcessing` around the call, updates `lastError` / `lastExplanation` / `apiHealthy` based on the response. Returns the `ApiResult` so callers can coordinate with the undo stack.

**Error mapping:** keep the Phase 3 convention — surface `errors.detail` if present, otherwise the first field error, otherwise a generic "AI command failed". The 503-with-`apiHealthy=false` case shows a dedicated "AI is unavailable — manual editing still works" message per PRD §6.5.

### New component: `frontend/src/components/CommandBar.vue`

**Props:** `date: string`, `blocks: TimeBlock[]`, `snapshotBlocks: () => TimeBlock[]`, `pushUndo: (action: UndoAction) => void`.

**Template:**

- Fixed position at the bottom of the schedule viewport (`position: fixed; left: 0; right: 0; bottom: 0`) with the same max-width / centering as the schedule body. Terminal-style appearance per PRD §5.
- Full-width `<input type="text">` with a prompt marker (e.g. `›`) and placeholder text that rotates between an English and a Russian example.
- Small status dot on the left showing `apiHealthy` state (green / red). Tooltip describes the state.
- Spinner on the right edge while `isProcessing`.
- Error text row below the input when `lastError` is non-null; clicking dismisses.
- Optional `explanation` row below the input when `lastExplanation` is non-null and `lastError` is null.

**Behaviour:**

- Submit on Enter. Block submission while `isProcessing`. Blur / Esc clears the input and `lastError`.
- Before submit: `const snapshot = snapshotBlocks()` (follow the exact pattern documented in `tasks/lessons.md` — call `snapshotBlocks()` through the prop so Vue's reactive proxy is unwrapped safely).
- On success: `pushUndo({ description: explanation || "AI command", type: "ai", previousBlocks: snapshot, scheduleDate: date })` after the Inertia reload resolves. Clear the input.
- On failure: do **not** push to undo (nothing changed server-side). Keep the input contents so the user can edit and retry.

**Keyboard focus:** global shortcut to focus the command bar (e.g. `/` when not inside an input). Document it in the placeholder text. Respect the same "ignore while typing in an editable field" logic already used by the Ctrl+Z listener in `useUndo.ts`.

### `frontend/src/types/index.ts`

- Extend the `UndoAction["type"]` union with `"ai"`. Everything else (payload shape) is reused.

### `frontend/src/pages/Schedule.vue`

- Import `useAI()` and render `<CommandBar>` once, after the schedule body.
- Pass `date`, `blocks`, and the existing `snapshotBlocks` / `pushUndo` from `useUndo()` — both are already destructured in this file.
- No layout change beyond adding bottom padding to the schedule scroll area so the fixed bar doesn't occlude the last block.

### `frontend/src/app.css`

- Minimal styles for the command bar: dark background, monospace-leaning font for the prompt marker, subtle top border, focus ring on the input. Reuse category-color tokens already defined.

## Test Coverage To Add

### Backend
- `service.run_command` — monkeypatch the OpenAI client; assert prompt assembly, schema passed to SDK, successful parse, and mapping of every exception class to the correct `AIError` subclass.
- `ai_command` view happy paths: `add`, `move`, `remove`, `resize` — each returns the mutated block list and writes an `AIInteraction` row.
- `ai_command` bilingual: Russian command returns the same shape (parsing is the model's job; mock a canned JSON response).
- `ai_command` failure paths: missing API key → 503 + logged; provider timeout → 504 + logged; invalid JSON from model → 502 + logged with raw text; oversized input → 400; oversized body → 413.
- Cross-user `task_id` attempts return 400 and do not log partial actions beyond the initial intent row.
- Atomicity: mid-batch validation failure rolls back all DB changes; the `AIInteraction` row for that request still exists (intent logging happens first) but `actions_json` reflects what the AI asked for, not what succeeded.
- CSRF coverage identical to the other POST endpoints.

### Frontend
- `useAI.submitCommand` sends the correct body and CSRF header; flips `isProcessing` around the call; updates `lastError` / `lastExplanation` / `apiHealthy` for each response class (200 / 400 / 503).
- `CommandBar` submits on Enter, ignores submit while `isProcessing`, focuses on `/`, does not intercept `/` while typing in another input.
- `CommandBar` integration: on success calls `pushUndo` with `type: "ai"`; on failure does not push; snapshot is taken via the prop (regression guard for the DataCloneError pattern in `tasks/lessons.md`).
- Visual/behavioural: status dot turns red on 503, green on next success.

## Implementation Notes

### Env-var naming: `LLM_*` instead of `OPENAI_*`

This plan named the config vars `OPENAI_API_KEY`, `OPENAI_COMMAND_MODEL`,
`OPENAI_REQUEST_TIMEOUT_SECONDS`, and `OPENAI_MAX_COMMAND_CHARS`. They shipped
as `LLM_API_KEY`, `LLM_MODEL`, `LLM_REQUEST_TIMEOUT`, `LLM_MAX_COMMAND_CHARS`,
plus a new `LLM_BASE_URL`. The rename keeps the code provider-agnostic — the
SDK is OpenAI's, but `LLM_BASE_URL` lets us swap to OpenRouter or a
self-hosted proxy without touching code. Authoritative list lives in
`.claude/rules/project.md` under **Environment Variables**.

The plan also anticipated a split into `LLM_COMMAND_MODEL` / `LLM_DRAFT_MODEL`
for the Phase 5 draft generator. Deferred: a single `LLM_MODEL` is enough
until draft generation actually exists.

### `response_format`: `json_object`, not `json_schema`

Strict `json_schema` Structured Outputs are OpenAI-only and not uniformly
supported across `LLM_BASE_URL` targets (OpenRouter routes, self-hosted
proxies). We use `{"type": "json_object"}` and rely on the post-parse
validators in `ai/schemas.py` (`validate_response_envelope` /
`validate_action_shape`) to enforce the contract. This is by design, not an
oversight — see comments in `ai/service.py` and `ai/schemas.py`.
