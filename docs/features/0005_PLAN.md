# Phase 5: Templates, Rules & Draft Generation

Auto-generate the morning schedule from a weekday/weekend `Template`, the last 7 days of the user's history, and active `Rule`s. New `POST /api/ai/schedules/<date>/generate-draft/` endpoint produces `add`-only actions; the frontend triggers it asynchronously after the page loads on a brand-new day, or on demand from a "Regenerate draft" button (only allowed on an empty schedule). A new `/settings/` Inertia page exposes CRUD for templates and rules. The existing `Schedule.status` field flips `draft → active` on the first user mutation; `reviewed` is reserved for Phase 6.

The AI service layer (Phase 4) — error taxonomy, rate limiter pattern, `AIInteraction` logging, post-parse validators — is reused. New work is largely additive.

**Decisions baked in (overridable):**

- Auto-draft is async: `schedule_view` returns immediately with `auto_draft_pending: true`; frontend fires `generate-draft` post-mount.
- Templates editor is form/list-based for this phase; drag-and-drop parity with the daily schedule is deferred.
- Drafts get a separate rate-limit counter (`LLM_DRAFT_RATE_LIMIT_PER_HOUR`, default 10) — lower than commands because they use a heavier model.

## Phase 5B — Backend

### `backend/ai/prompts.py` — additions and one refactor

- **Refactor**: split the existing `_format_block_line(block)` (which assumes a Django `TimeBlock` instance with `id`, `is_completed`, `start_time` as `time` objects) into a normalized format taking a dict, plus two adapters. The current helper can't accept template dicts (`{"title": ..., "start_time": "HH:MM", ...}`) as-is — they have no `id`, no `is_completed`, and times are strings. Concretely:
  - `_format_block_line(d: dict) -> str` — takes `{id, start_time: str, end_time: str, category, is_completed: bool, title}` and returns the existing line shape.
  - `_runtime_block_to_dict(block) -> dict` — adapter for `TimeBlock` model instances (used by `build_user_message` and history rendering).
  - `_template_entry_to_dict(entry, synthetic_id) -> dict` — adapter for template JSON entries; `is_completed=False`, `id` is a stable synthetic int for the model to reference if it wants.
- `SYSTEM_PROMPT_DRAFT: str` — separate system prompt. Spells out: (a) only `add` actions allowed (no `task_id` references possible — schedule is empty), (b) same 5-min granularity / day window / category enum as `SYSTEM_PROMPT`, (c) the draft must respect every active rule (rules text injected via `build_draft_user_message`), (d) must use the provided template as the baseline, (e) may diverge from the template based on patterns observed in the last-7-days history (e.g. if user consistently moved gym from 17:30 to 18:00, learn that), (f) bilingual is irrelevant for drafts (no user command), but rule text may be in either language and must still be respected, (g) JSON envelope identical to commands: `{actions: [...], explanation: "..."}`.
- `build_draft_user_message(schedule, template, history_schedules, rules, now) -> str`. Pure function. Sections:
  1. `Schedule date: YYYY-MM-DD (Weekday)`
  2. `Current local time: HH:MM`
  3. `Active template (<weekday|weekend>):` each template entry rendered via `_template_entry_to_dict` + `_format_block_line`.
  4. `Recent history (last N days):` each past schedule as a date header + its runtime blocks. Skip schedules with status `draft` to avoid the AI training itself on its own un-reviewed output.
  5. `Active rules (priority desc):` numbered list of rule texts.
  6. No `User command:` section.

### `backend/ai/schemas.py` — additions

- `validate_draft_response(parsed, allowed_categories) -> list[str]`. Wraps `validate_response_envelope` + per-action checks, but additionally rejects any action whose `type != "add"`. Drafts must not reference `task_id`s that don't exist.
- Cap drafts at the existing `MAX_ACTIONS_PER_COMMAND = 20` for now — a 20-block day is plausible (06:00–23:00 in 50-min slices is 20). Document the cap; bump if real drafts need more.

### `backend/ai/service.py` — additions

- New dataclass `AIDraftResult { raw_response_text, parsed_actions, explanation }` (same shape as `AICommandResult`; could share the type, but kept separate for clarity).
- New public function `run_draft(schedule, template, history_schedules, rules, now) -> AIDraftResult`. Algorithm:
  1. Guard: empty `LLM_API_KEY` → `AIUnavailableError`.
  2. Build messages via `prompts.SYSTEM_PROMPT_DRAFT` + `prompts.build_draft_user_message`.
  3. Call `client.chat.completions.create(model=settings.LLM_DRAFT_MODEL, ...)` with the same `response_format` / `temperature=0` / `timeout` as `run_command`.
  4. Parse, run `validate_draft_response`, raise `AIParseError` on failure with raw text preserved.
  5. Return `AIDraftResult`.
- Same exception taxonomy — view maps unchanged.

### New file: `backend/ai/views.py` — `ai_generate_draft` view

- Routing: `POST /api/ai/schedules/<str:date>/generate-draft/`.
- Decorators: `@login_required`, `@require_http_methods(["POST"])`, plus a new `@_rate_limit_drafts_per_user` decorator built from the same `cache.add` + `cache.incr` pattern as `_rate_limit_per_user` (separate cache key prefix `ai_draft_rl:`, separate budget `LLM_DRAFT_RATE_LIMIT_PER_HOUR`).
- Algorithm:
  1. `reject_oversized_body(request)` (no body expected, but keep the guard symmetric with the command endpoint).
  2. Parse `date` with `datetime.date.fromisoformat`.
  3. `Schedule.objects.get_or_create(user=..., date=...)` — same get-or-create pattern as `schedule_view` and `ai_command`.
  4. **Refuse if non-empty**: if `TimeBlock.objects.filter(schedule=schedule).exists()`, return 409 `{ "errors": { "detail": "Schedule already has blocks; delete them before regenerating." } }`. Prevents accidental destructive regenerate.
  5. Pick template: `Template.objects.filter(user=request.user, type=...).first()`. The unique `(user, type)` constraint guarantees zero-or-one row, so `.first()` is unambiguous. If none exists, return 422 `{"errors": {"detail": "No template configured for this day type. Open Settings."}}`.
  6. Pull history: `Schedule.objects.filter(user=request.user, date__lt=parsed_date, date__gte=parsed_date - timedelta(days=settings.LLM_HISTORY_DAYS), status__in=["active","reviewed"]).prefetch_related("time_blocks")`. Excluding `draft` status avoids the AI training on its own unreviewed output.
  7. Pull active rules: `Rule.objects.filter(user=request.user, is_active=True).order_by("-priority")`.
  8. Call `run_draft(schedule, template, history, rules, now=timezone.localtime())` outside any transaction (same rationale as `ai_command`: long network call, no DB locks held).
  9. `_log_interaction(schedule, command="[DRAFT]", response_text=result.raw_response_text, actions=result.parsed_actions, kind="draft")` — extend `_log_interaction` to accept `kind`. The synthetic `"[DRAFT]"` user_command is a placeholder; the real signal is the `kind` field.
  10. Open `transaction.atomic()`, re-fetch schedule with `select_for_update()`, **re-check emptiness** under the lock (concurrent `create_block` could have raced in), apply each `add` action via the existing `_apply_add` helper extracted from Phase 4.
  11. Mark interaction success, return `{ "blocks": [...], "explanation": "..." }`.
- Errors map identically to `ai_command` via `_AI_ERROR_STATUS`.
- Refactor: `_apply_add`, `_log_interaction`, `_mark_success`, `_check_day_window`, `_check_granularity`, `_check_no_overlap` from `views.py` are already factored — just need to make `_log_interaction` accept `kind`.

### `backend/schedules/views.py` — auto-draft signal in `schedule_view`

Modify `schedule_view`:

- Replace `Schedule.objects.get_or_create(...)` with the same call but capture the `created` flag.
- Compute `template_exists = Template.objects.filter(user=request.user, type=...).exists()` for the requested date's day type.
- Add two Inertia props:
  - `auto_draft_pending = bool(created and template_exists and settings.LLM_API_KEY and settings.LLM_API_KEY.strip())` — one-shot trigger for the post-mount auto-draft. Including the template check up front avoids burning the 10/hr draft budget on a guaranteed 422.
  - `has_template_for_type: bool` — the same `template_exists` value, exposed separately so `RegenerateDraftButton` can stay accurate beyond the first page load. (`auto_draft_pending` flips false after the first render and isn't usable as an ongoing capability signal.)
- Frontend uses `auto_draft_pending` to fire `generate-draft` post-mount; server does not block on the LLM call. `has_template_for_type` drives the regenerate button's enable/disable state and the inline "no template" message.

### `backend/schedules/api.py` + `backend/ai/views.py` — status flow

Add a method on `Schedule`:

- `Schedule.mark_active_if_draft()` — if `status == DRAFT`, set to `ACTIVE` and save with `update_fields=["status"]`. No-op otherwise.

Call this from every **forward-mutating** endpoint right before returning a 200:

- `create_block` (always — a successful POST adds a block).
- `block_detail` PATCH — any successful 200 (the existing handler doesn't dirty-check incoming fields, and adding one isn't worth it: an empty-body PATCH is degenerate, and every realistic payload — title edit, completion toggle, time change — is genuine user engagement that should flip the badge).
- `block_detail` DELETE (always).
- `reorder_blocks` (always).
- `ai_command` — **only** when `len(result.parsed_actions) > 0`. RULES.md already documents that a 200 with `actions: []` is a successful no-op (LLM ambiguity / out-of-window guard); flipping status on those would lie about user intent.

Two endpoints explicitly do **not** flip status:

- `restore_blocks` — pure state restoration (the undo target). Whatever status the schedule had stays put. Notably: undoing a freshly generated draft (`restore_blocks` with empty list) leaves `status="draft"` and zero blocks, which is exactly the state the auto-draft started from — the user can click "Regenerate draft" again.
- `ai_generate_draft` — drafts stay `draft` until the user actually edits.

### `backend/templates_mgr/api.py` — new file (Templates + Rules CRUD)

REST endpoints, `@login_required`, `@require_http_methods` per route. **Every query is filtered by `request.user`**; PKs from other users return 404 (not 403, to avoid id enumeration — same convention as `block_detail`).

- `GET /api/templates/` → `[{id, name, type, blocks}, ...]` for the current user only.
- `PUT /api/templates/<int:pk>/` → update `name`, `type`, `blocks`. Validate `blocks` shape: list of `{title, start_time, end_time, category}` with the same HH:MM regex, 5-min granularity, day-window guards used by `_apply_add`. Reject overlapping entries within one template. Changing `type` to a value the user already has another template for hits the unique constraint → 409.
- `POST /api/templates/` → create. Same validation. Unique `(user, type)` enforced at the DB layer.
- `DELETE /api/templates/<int:pk>/` → delete.

**Unique-constraint handling on POST/PUT.** Both endpoints wrap the `save()` in `transaction.atomic()` and explicitly catch `django.db.IntegrityError`, returning HTTP 409 with `{"errors": {"type": "Template for this <slotType> already exists."}}`. Without the catch, the integrity error becomes a 500 (and leaves the transaction in a broken state for any follow-up queries in the same view). Naming the field — `type` rather than a generic "constraint violation" — lets `TemplateEditor.vue` highlight the right input. This also handles the racy case where two requests for the same `(user, type)` hit the validation step simultaneously and only one survives the DB write.
- `GET /api/rules/` → `[{id, text, is_active, priority}, ...]` for the current user, ordered by `-priority`.
- `POST /api/rules/` → create. `text` required, length-capped (e.g. 500 chars).
- `PATCH /api/rules/<int:pk>/` → partial update of `text`, `is_active`, `priority`.
- `DELETE /api/rules/<int:pk>/` → delete.

Reuse `reject_oversized_body`, `parse_time_or_error`, `validate_five_minute_or_error`, `validate_time_range` from `schedules/http.py`. Centralize template-block validation as `validate_template_blocks(blocks: list) -> list[str]` in this new file.

### `backend/templates_mgr/views.py` — Inertia settings page

- `settings_view(request)` → `inertia_render(request, "Settings", {templates, rules})`. Both lists filtered by `request.user`. Server-side fetch so the page boots without a second round-trip.

## Phase 5C — Frontend Settings Page (parallel with 5D)

### New page: `frontend/src/pages/Settings.vue`

Two-section layout. Top bar back-link to `/schedule/<today>/`. Server-rendered initial data via Inertia props (`templates: Template[]`, `rules: Rule[]`).

The unique `(user, type)` constraint means each user has 0 or 1 weekday template and 0 or 1 weekend template. Rather than render a flat list, render two fixed slots — one for weekday, one for weekend — and pass the matching template (or `null` if missing) to a single `TemplateEditor` per slot. After the migration's orphan-row wipe, every existing user starts with both slots empty; this layout makes "create the first one" the obvious next action.

### New components:

- `frontend/src/components/TemplateEditor.vue` — props: `template: Template | null`, `slotType: "weekday" | "weekend"`.
  - **Create mode** (when `template === null`): renders a "No <slotType> template yet — Create" placeholder with a single "Create template" button. Click reveals the form pre-populated with the slot's `type` and an empty `blocks` array; "Save" calls `POST /api/templates/` and the parent re-fetches.
  - **Edit mode** (when `template` is set): same form, populated from the template; `type` field is read-only (changing it would collide with the unique constraint and isn't a meaningful operation here — delete + recreate instead).
  - Form fields: `name` text input; block list as a table of `title | start_time | end_time | category | [delete]`. Time inputs are `<input type="time" step="300">` (300s = 5-min step). Category is `<select>`.
  - "Add block" button appends an empty row.
  - "Save" calls `PUT /api/templates/<id>/` (edit) or `POST /api/templates/` (create) with the full `blocks` array. Validation errors surface inline; 409 from the server highlights the `type` select.
  - "Delete template" button (edit mode only) behind a confirm; on success the slot reverts to create mode.
  - **Drag-and-drop is deferred** — log as TODO comment in the file. Sort order in the rendered list is by `start_time` ascending; the array is sorted client-side before saving.
- `frontend/src/components/RulesList.vue` — props: `rules: Rule[]`.
  - Each rule row: text (inline-editable on click), `is_active` checkbox, priority arrows (▲/▼) that swap priorities with the neighbour and PATCH both.
  - "Add rule" inline form at the top — also acts as the empty state when `rules.length === 0`.
  - Delete button per row, behind a confirm.

## Phase 5D — Frontend Draft Trigger + Status (parallel with 5C)

### New composable: `frontend/src/composables/useDraft.ts`

Mirrors `useAI.ts`: module-level state (single consumer is `Schedule.vue`). Thin RPC wrapper — does **not** import `useUndo`. Undo registration happens in `Schedule.vue` after the promise resolves, mirroring the `useAI` / `CommandBar` split.

- State: `isGeneratingDraft: Ref<boolean>`, `lastDraftError: Ref<string | null>`.
- Return type: `interface DraftSubmitResult extends ApiResult { explanation?: string | null }`. Mirrors the `AISubmitResult` shape in `useAI.ts` so callers can read `result.explanation` directly without reaching into `result.data`. Implementation extracts `data.explanation` after a 200 and returns `{ ok, data, explanation }`.
- Action: `generateDraft(date: string): Promise<DraftSubmitResult>` — POSTs to `/api/ai/schedules/<date>/generate-draft/`. On 200: triggers `router.reload({ only: ["blocks", "schedule"] })` so the page picks up the new blocks; `schedule.status` remains `"draft"` per the status-flow spec — drafts only flip on the user's first edit. **Note:** `auto_draft_pending` is its own Inertia prop, not part of `schedule`, so a partial reload of `["blocks", "schedule"]` does not refresh it. We do not rely on the prop going stale; `Schedule.vue` instead keeps a per-instance `attemptedAutoDraftDates: Set<string>` keyed by date (see wire-up below) to prevent refire. On 409: silently no-op (race: the schedule already has blocks; the auto-trigger guard upstream should make this nearly impossible). On 422 / 429 / 503: surface the error via `lastDraftError` for the schedule page to render.

### New component: `frontend/src/components/DraftBadge.vue`

- Renders only when `schedule.status === "draft" && blocks.length > 0`.
- Small pill near the date navigator: text "Draft — edit to keep". Auto-disappears on first edit (server flips status; Inertia reload includes `schedule` and reflects it).

### New component: `frontend/src/components/RegenerateDraftButton.vue`

- **Props**: `hasTemplate: boolean`, `slotType: "weekday" | "weekend"`. Other state (`isGeneratingDraft`, `apiHealthy`) comes from the module-level composables.
- Renders only when `schedule.status === "draft" && blocks.length === 0` (i.e. an empty schedule that auto-creation produced or that the user emptied).
- Lives in `DateNavigator` next to the gear icon (passed in as a slot or rendered conditionally inside `Schedule.vue` adjacent to it).
- **Disabled and visibly muted** when `!hasTemplate`. In place of the click target, render an inline reason like "No <slotType> template configured." with the gear icon already visible in `DateNavigator` providing the one-click path to `/settings/`. This pre-empts the 422 round-trip — server-supplied `has_template_for_type` (Inertia prop) is the source of truth.
- Also disabled while `isGeneratingDraft`, while `apiHealthy === false`, or while the command bar's `isProcessing` is true (avoid concurrent AI calls).
- On click (when enabled): calls `generateDraft(date)`. On success, `Schedule.vue` pushes a `"draft"` undo entry (see below).
- **Runtime errors are rendered inline next to the button**, not via `UndoToast` (which only supports an Undo action). The schedule page's existing inline-error slot — already used for `useAI`'s `lastError` — shows `lastDraftError`, plain text. Messages:
  - 409 → "Schedule already has blocks. Clear them before regenerating." (defensive — visibility guard should prevent this.)
  - 422 → "No template configured. Open Settings to create one." (Should be unreachable in practice because `hasTemplate` gates the button, but kept as a defensive fallback for race conditions where a template is deleted between page load and click.)
  - 503 → "AI is unavailable. Manual editing still works."
  - 429 → "Draft rate limit reached. Try again later."

### `frontend/src/pages/Schedule.vue` — wire-up

- New props: `auto_draft_pending: boolean` (defaults to false), `has_template_for_type: boolean`, updated `schedule.status`. `has_template_for_type` is forwarded to `RegenerateDraftButton` as `hasTemplate`.
- **Auto-draft trigger lives in a watcher, not `onMounted`.** Inertia's same-component navigation behavior varies by version and config (full remount vs. preserved state vs. partial reload), and partial reloads — used by every mutation in this app — explicitly do not remount. A bare `onMounted` hook is therefore unreliable: it can miss new dates reached via partial reload and depends on framework internals we shouldn't be load-bearing on.
- Implementation: a module-local `attemptedAutoDraftDates = new Set<string>()` (keyed by date string) plus `watch(() => [props.date, props.auto_draft_pending, props.blocks.length], maybeGenerateDraft, { immediate: true })`. The handler fires `generateDraft(props.date)` only when `props.auto_draft_pending && props.blocks.length === 0 && !attemptedAutoDraftDates.has(props.date)`, and adds the date to the set immediately to prevent re-firing. `immediate: true` covers first-mount; the watcher covers date navigation. The set is per-component-instance, so a real remount (full Inertia visit) starts fresh — which is correct: the server's `auto_draft_pending` is the authoritative one-shot signal for "this is a brand-new schedule," and once we've attempted it for a given date we never re-attempt within the same component lifetime.
- After `generateDraft` resolves successfully (auto or manual), push undo: `pushUndo({ type: "draft", description: result.explanation || "Generated draft", previousBlocks: [], scheduleDate: props.date })`. Undo restores the empty state via the existing `restore_blocks` endpoint with an empty list — which (per the status-flow rules in 5B) does **not** flip status, leaving the schedule as `draft` so the regenerate button reappears.
- Render `DraftBadge` and `RegenerateDraftButton` near `DateNavigator`.
- While `isGeneratingDraft`, show a centered spinner overlay over the schedule body and forward a `disabled` flag to: `AddBlockForm` (suppress submit), `TimeBlock` (suppress inline edits, completion toggle, delete), `CommandBar` (already has `isProcessing`; OR them together), `GapSlot` (suppress click-to-add), and `useDrag.startDrag` (early-return when the disabled flag is true). Without the GapSlot + drag guards, the disable is decorative — those are the two ways a user can mutate a schedule that aren't already covered.
- On draft-generation failure (422 / 429 / 503): keep the schedule empty, surface `lastDraftError` as a one-line inline message above the schedule body. Manual editing stays enabled.

### `frontend/src/composables/useUndo.ts` — handle `"draft"` type

Extend the undo dispatcher: when the popped action's `type === "draft"`, the restore target is "no blocks", so the existing `restore_blocks` endpoint with an empty array is the correct primitive — no new endpoint needed. The toast description comes from the action's `description` field, populated from the AI's `explanation`.

## Implementation Notes

### Why exclude `draft`-status schedules from history context

If yesterday's schedule was auto-drafted but never edited (`status` still `draft`), it represents the AI's prior guess, not the user's actual day. Feeding it back as "history" would compound errors. Once the user edits anything, the status flips to `active` and that day rejoins the history corpus.

### Why a separate rate-limit counter for drafts

Drafts use `gpt-4o` (per PRD §15.3, ~5–10x cost of `gpt-4o-mini`). A shared counter would let a misbehaving auto-trigger loop drain the command budget too. Separate counters mean a draft outage doesn't kill the command bar.

### Why drafts are async-after-load instead of inline in `schedule_view`

Inline LLM calls in `schedule_view` would couple page TTFB to provider latency (~5–15s in the worst case). Cache hits on the inline path are rare (each day is unique). Async-after-load lets the page paint instantly and lets the user see "generating…" feedback while the LLM works.

### Why drag-and-drop is deferred for the templates editor

PRD §7.1 calls for parity with the daily schedule's drag UI. Honoring that in this phase requires synthesizing block ids for `useDrag` to track, mapping drag events back to JSON array reorders, and re-validating the result. Roughly doubles the frontend work for what is, in practice, a low-frequency UX (templates change ~weekly). Form/list editor unblocks the rest of Phase 5; DnD parity is a tracked follow-up.

### `auto_draft_pending` is a one-shot signal

The flag is `true` only on the request that *created* the `Schedule` row. On any subsequent visit to the same date, `get_or_create` returns `created=False` and the flag is `false` — so deleting all blocks manually does not retrigger the draft. Manual regeneration goes through the explicit "Regenerate draft" button, which calls `generateDraft` directly and is gated by the 409 check.

### `AIInteraction.kind` migration is additive only

Default `"command"` plus a Django migration with `default="command"` means the column is back-fill-safe on a hot DB. No data migration needed beyond the schema add.

### `Template` / `Rule` user-FK migration wipes orphan rows

The existing global rows have no owner, and synthesizing one (e.g. "first superuser") would silently bind another user's identity to data they didn't author. The migration's data step deletes them; users re-run `seed_templates --user <name>` to repopulate per user. Document this in the migration's docstring and in `tasks/todo.md` so the deploy step is obvious. Single-developer impact: regenerate seed locally; production has not been deployed yet.

### Status-flip gating: forward mutations only, and only when something actually changed

Two subtle rules from review feedback:

1. **`restore_blocks` is not a forward mutation.** It's the undo target. Calling it with the previous block list shouldn't pretend the user just made a fresh edit. Concretely: undoing a freshly auto-generated draft (`restore_blocks([])`) leaves `status="draft"`, so `RegenerateDraftButton` reappears as if no draft had ever fired.
2. **`ai_command` returning `actions: []` is a successful no-op.** RULES.md already treats this as the "AI didn't change anything" branch for undo registration; status flipping must follow the same gate. `mark_active_if_draft()` is called from `ai_command` only when `len(result.parsed_actions) > 0`. Otherwise an LLM responding with "I don't understand" silently promotes a draft to active.
