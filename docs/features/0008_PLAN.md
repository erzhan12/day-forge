# 0008 — AI chat as collapsible right sidebar

## Context

Feature 0007 shipped the multi-turn chat as a fixed bottom dock (`frontend/src/components/CommandBar.vue`, `position: fixed; bottom: 0`). The original 0007 plan referenced a future sidebar variant for wide screens (the "PR B" comment in `CommandBar.vue:7`); this feature delivers it.

User-requested changes (verbatim intent):

- **«AI чат был справа»** — chat moves to a right-hand side panel on wide viewports.
- **«больше высота поля, чтобы больше строк помещалось»** — input textarea grows to fit substantially more lines.
- **«сворачиваемая панель»** — sidebar is collapsible via a toggle button; state persists across reloads.

Decisions captured from clarification round:

- **Layout:** collapsible right sidebar on viewport ≥1024px; existing bottom dock on <1024px (unchanged for narrow screens). Sidebar width 380px.
- **Input size (sidebar variant):** textarea min 6 rows (~120px), max 20 rows (~400px). Current dock variant keeps min 1 / max 10.
- **Default state:** sidebar open on first visit. Toggle state persisted in `localStorage` under `day-forge:chat-sidebar:open` (boolean).
- **State ownership:** `Schedule.vue` owns the `sidebarOpen` ref (single source of truth). `ChatSidebar.vue` is a controlled component declaring `defineModel<boolean>("open")` and emits `update:open` on toggle. A tiny helper module `frontend/src/utils/chatSidebarStorage.ts` exposes `readChatSidebarOpen(): boolean` and `writeChatSidebarOpen(v: boolean): void` so localStorage I/O lives in exactly one place (and is trivially mockable in vitest).
- **No backend changes.** `useChat` remains the single source of truth for thread state — both variants subscribe to the same module-level state, so navigating between viewport widths or toggling the sidebar preserves the active thread.
- **Existing `data-testid` values are preserved** (`chat-input`, `chat-clear`, `chat-thread`, `chat-privacy-hint`, `command-bar`) so 0007's Playwright scripts keep passing without edits.

Non-goals: resizable/draggable sidebar; left-side or top placement option; per-user server-side preference; mobile slide-in drawer (we just fall back to the existing dock).

## Phase 1 — Shared viewport breakpoint

Files: `frontend/src/composables/useViewport.ts` (new).

- Export `useViewport()` returning `{ isWide: Ref<boolean> }` where `isWide.value === true` iff `window.matchMedia("(min-width: 1024px)").matches`.
- Internally subscribe to the `MediaQueryList`'s `change` event on mount; unsubscribe on unmount. Tracks one media query — no resize listener spam.
- The breakpoint constant `WIDE_VIEWPORT_QUERY = "(min-width: 1024px)"` is colocated in this file so a future change touches one place. Document that the 1024px value matches Tailwind's `lg` breakpoint convention but the project does not use Tailwind — the value is independent of any framework.

Edge: SSR / Inertia first paint. Inertia renders client-side in this project (`frontend/src/main.ts`), so `window` is available at composable invocation time. No SSR guard required, but the `matchMedia` call MUST happen inside the composable function body (not at module scope) to keep the file safe to import from any context.

## Phase 2 — Variant-aware `CommandBar.vue`

Files: `frontend/src/components/CommandBar.vue`.

### 2.1 New prop

- Add `variant: 'dock' | 'sidebar'` (required). Defaulting is intentionally omitted so a forgotten prop in `Schedule.vue` fails type-checking via `vue-tsc`.

### 2.2 Autosize bounds keyed to variant

The script currently exposes two constants used by `autosize()`:

- `LINE_HEIGHT_PX = 20`
- `MAX_TEXTAREA_LINES = 10`

Add a third — `MIN_TEXTAREA_LINES` — and make both `MIN_*` and `MAX_*` depend on `variant`:

| variant | min lines | max lines |
| --- | --- | --- |
| `dock` | 1 | 10 |
| `sidebar` | 6 | 20 |

`autosize()` keeps its current logic but clamps `el.scrollHeight` between `LINE_HEIGHT_PX * MIN_TEXTAREA_LINES` (lower bound) and `LINE_HEIGHT_PX * MAX_TEXTAREA_LINES` (upper bound). The matching CSS `min-height` / `max-height` rules on `.command-input` move into variant-scoped classes (`.command-input.variant-dock`, `.command-input.variant-sidebar`) so the JS clamp and CSS clamp stay in sync.

The template's current `rows="1"` attribute (hardcoded) becomes a dynamic bind: `:rows="textareaMinLines"` where `textareaMinLines = computed(() => props.variant === "sidebar" ? 6 : 1)`. Reasons: (a) the attribute is the textarea's semantic row count, so it stays consistent for assistive tech and any consumer that reads `el.rows`; (b) it is the no-CSS / unstyled-fallback height; (c) if `autosize()` is delayed or skipped (slow JS, error in the onMounted hook, future SSR), the textarea still defaults to a sensible size for the variant instead of one row. The CSS `min-height` rule remains the real visual floor under normal conditions.

> Preserve the existing "must equal `LINE_HEIGHT_PX`" comment on `.command-input`'s `line-height` rule — the contract still holds, just per-variant now.

### 2.3 Container styling per variant

Top-level `.command-bar` element gets the variant class (`.command-bar.variant-dock` / `.command-bar.variant-sidebar`). Style differences:

- `variant-dock` (unchanged from 0007):
  - `position: fixed; left: 0; right: 0; bottom: 0;`
  - Dark theme (`#111827` background, `#e5e7eb` text).
  - Inner rows capped at `max-width: 640px` and centered.
- `variant-sidebar`:
  - `position: static; height: 100%; display: flex; flex-direction: column;`
  - Light theme (`#ffffff` background, neutral border on left, body text colour `#1f2937`). Matches the Schedule page surface, not the dock.
  - Inner rows: full width (no 640px cap).
  - `.thread` becomes `flex: 1 1 auto; min-height: 0; overflow-y: auto;` so it absorbs the leftover vertical space between the privacy hint and the form. `max-height: 200px` is dropped for this variant.
  - `.command-row` keeps `display: flex; align-items: flex-start;` — same shape, just inside the column flex.

### 2.4 Variant-aware visible-message cap

The current dock caps the thread at `MAX_VISIBLE_MESSAGES = 4` (latest only). For the sidebar that's the whole point — the user wanted height to see more context. Make the cap variant-keyed:

| variant | visible messages |
| --- | --- |
| `dock` | 4 (unchanged — keeps the dock terse) |
| `sidebar` | unbounded (`messages` rendered in full) |

`.thread` in the sidebar variant already has `flex: 1; overflow-y: auto` (Phase 3) so long histories scroll inside the panel without pushing the textarea off-screen. The slice helper becomes `visibleMessages = computed(() => props.variant === "dock" ? messages.value.slice(-MAX_VISIBLE_MESSAGES) : messages.value)` — explicit `props.variant` access (defineProps result), no destructuring of `variant` at the top of `<script setup>`, since destructured props lose reactivity unless `defineProps` is wrapped in Vue 3.5's reactive-props transform and the rest of the file already reads `props.X`.

### 2.5 Variant class applied in template

The template adds `:class="\`variant-${variant}\`"` to both:

- `.command-bar` root (`<div class="command-bar" :class="\`variant-${variant}\`">`)
- `.command-input` textarea (`<textarea class="command-input" :class="\`variant-${variant}\`" ... />`)

> Template may reference `variant` directly; script code must use `props.variant`.

Variant-scoped CSS selectors (`.command-bar.variant-sidebar`, `.command-input.variant-dock`, etc.) drive Phase 2.2 and 2.3 styling. Adding the class on the textarea — separately from the root — keeps the `autosize()`-relevant CSS rules (`min-height`, `max-height`, `line-height`) on a single element where the JS clamp also lives.

### 2.6 Behaviour kept identical across variants

No changes to: `handleSubmit`, `handleKeydown`, `handleInput`, `handleGlobalKeydown` (`/`-to-focus still works in both variants — there's only ever one CommandBar mounted at a time, so no double-focus race), placeholder rotation, `pendingAsk` / `bubble-ask` styling, error row.

`data-testid` attributes stay exactly as in 0007.

## Phase 3 — `ChatSidebar.vue` wrapper

Files: `frontend/src/components/ChatSidebar.vue` (new).

A thin shell whose only responsibilities are positioning and the open/closed toggle UI. Persistence is **not** handled here — `Schedule.vue` owns the canonical `sidebarOpen` ref and writes localStorage via the helper from Phase 4.1. The chat itself is still rendered by `CommandBar` (passed `variant="sidebar"`).

### 3.1 Structure

```
.chat-sidebar
  .chat-sidebar-header                 ← title "AI Chat" + collapse button
  .chat-sidebar-body (slot or direct)  ← <CommandBar variant="sidebar" .../>
```

When collapsed, the body is unmounted and only a thin vertical rail (`.chat-sidebar.collapsed`, width ~32px) with an expand button remains visible. Unmounting (not just `display: none`) avoids keeping a focused-but-invisible textarea in the tab order; the chat state survives anyway because `useChat` is module-level.

### 3.2 Positioning

- `position: fixed; top: 0; right: 0; bottom: 0;`
- Width: 380px when open, 32px when collapsed. Transition with `transition: width 180ms ease`.
- `z-index: 30` — same layer as the dock used to occupy. The sidebar replaces the dock at this breakpoint, so they never coexist.
- `box-shadow: -2px 0 8px rgba(0,0,0,0.05)` on the left edge for separation from the schedule.

### 3.3 Controlled `open` state

- `ChatSidebar` declares `const open = defineModel<boolean>("open", { required: true })`. The parent (`Schedule.vue`) holds the canonical ref; toggling inside the sidebar fires `update:open`, and the parent's persistence wiring writes localStorage. The sidebar itself does NOT read or write localStorage — it has no knowledge of the persistence backend.
- The project runs Vue `^3.5` (see `frontend/package.json`), so the `defineModel` macro is supported natively. If a future Vue downgrade ever lands, the fallback is `defineProps<{ open: boolean }>()` + `defineEmits<{ "update:open": [boolean] }>()` with an internal `const open = computed<boolean>({ get: () => props.open, set: (v) => emit("update:open", v) })` — same external contract, more boilerplate. Mention this only if implementation discovers a `vue-tsc` regression around the macro.
- Toggle: one logical toggle control flips `open.value`. It is rendered in the header when `open === true` and in the collapsed rail when `open === false` — physically these are two `<button>` elements under a `v-if` / `v-else`, not literally the same instance, but they share the same `data-testid="chat-sidebar-toggle"` and emit the same event, so tests and external selectors only ever see one stable contract. When `open === false`, the inner body (header subtitle + `<CommandBar>`) is unmounted via `v-if`; only the toggle rail (32px wide column with the expand chevron) remains. Unmounting avoids keeping a focused-but-invisible textarea in the tab order.

### 3.4 Toggle button accessibility

The collapse/expand button MUST:

- Be a `<button type="button">` (default `type="submit"` would post the surrounding form on click — there is no form here today, but the rule prevents future regressions).
- Carry `data-testid="chat-sidebar-toggle"` (used by vitest + future Playwright).
- Set `aria-label` to a verb-phrase tied to its current effect: `"Collapse AI chat panel"` when open, `"Expand AI chat panel"` when collapsed.
- Set `aria-expanded="true"` / `"false"` mirroring `open.value`.
- Set `aria-controls="chat-sidebar-body"`; the inner body wrapper carries `id="chat-sidebar-body"`. When the body is unmounted (`v-if`), the `aria-controls` reference points to a non-existent element — that is legal per ARIA 1.2 and screen readers handle it as "controls something currently hidden", which matches the intent.
- Icon: chevron right (›) when open (collapses into the right edge), chevron left (‹) when collapsed (expands leftward).

### 3.5 Props

- `date`, `snapshotBlocks`, `pushUndo` — forwarded straight to the inner `CommandBar` (passed `variant="sidebar"`). Identical signature to how `Schedule.vue` calls `CommandBar` today.

### 3.6 Schedule body padding

When the sidebar is open on a wide viewport, the schedule body needs right-side breathing room so the rightmost edge of `TimeBlock` doesn't sit underneath the sidebar. Two options were considered; the chosen one is:

- **Chosen — page-level right padding via a CSS variable.** `Schedule.vue` exposes `--chat-sidebar-width` on `.schedule-page` (set to `380px` when open and wide, `32px` when collapsed and wide, `0` otherwise) and adds `padding-right: var(--chat-sidebar-width, 0)` to `.schedule-page`. The 640px-centered schedule column is wide enough that the right padding visually offsets it without distorting the existing layout.
- Rejected — shrinking `.schedule-page`'s `max-width`. Would change the look of every page even when the sidebar is closed.

## Phase 4 — `Schedule.vue` wiring

Files: `frontend/src/pages/Schedule.vue`, `frontend/src/utils/chatSidebarStorage.ts` (new).

### 4.1 Storage helper

`frontend/src/utils/chatSidebarStorage.ts` exports:

- `readChatSidebarOpen(): boolean` — reads `localStorage["day-forge:chat-sidebar:open"]`. Strict semantics: returns `false` **only** when `JSON.parse(raw) === false` (literal boolean false). Every other case — missing key, parse throws, parsed value is `null` / `0` / `"false"` / `[]` / `true` / any non-boolean — returns `true`. This collapses the only "hide the sidebar" path into a single exact match, so a corrupted or hand-edited payload can never accidentally hide the only AI entry point on wide screens.
- `writeChatSidebarOpen(v: boolean): void` — writes `JSON.stringify(v)` to the same key. Swallows `localStorage` access errors (private-mode browsers) silently — the in-memory ref still works for the session.
- The localStorage key string is the only exported constant (`CHAT_SIDEBAR_OPEN_KEY`) for tests that need to clear it.

### 4.2 Page wiring

- Import the new composable: `import { useViewport } from "../composables/useViewport"`.
- Import the new wrapper: `import ChatSidebar from "../components/ChatSidebar.vue"`.
- Import storage helper: `import { readChatSidebarOpen, writeChatSidebarOpen } from "../utils/chatSidebarStorage"`.
- `const { isWide } = useViewport()`.
- `const sidebarOpen = ref<boolean>(readChatSidebarOpen())` — single source of truth for sidebar visibility on this page.
- `watch(sidebarOpen, writeChatSidebarOpen)` — persists every change in exactly one place.
- Conditional render:

  ```
  <ChatSidebar v-if="isWide" v-model:open="sidebarOpen" :date :snapshot-blocks :push-undo />
  <CommandBar v-else :date :snapshot-blocks :push-undo variant="dock" />
  ```

  When `isWide` flips at runtime (window resize past the breakpoint), Vue unmounts the inactive variant and mounts the other. The active thread survives because `useChat` is module-level.

### 4.3 Schedule-body padding adjustments

- The existing `padding-bottom: 88px` on `.schedule-body` (to clear the fixed dock) becomes conditional — applied only when `!isWide` so wide-screen pages don't waste 88px of empty space at the bottom. Implemented via class binding `:class="{ 'has-dock': !isWide }"` and moving the padding rule onto `.schedule-body.has-dock`.
- Right-side padding via the `--chat-sidebar-width` CSS variable (Phase 3.6) is driven by both `isWide` and `sidebarOpen` through a `computed` (NOT an inline `:style` expression — the unwrap rules differ between template root access and nested object literals, and a computed sidesteps that ambiguity):

  ```ts
  const schedulePageStyle = computed(() => ({
    "--chat-sidebar-width": isWide.value
      ? (sidebarOpen.value ? "380px" : "32px")
      : "0px",
  }))
  ```

  Bound on the page root: `<div class="schedule-page" :style="schedulePageStyle">`. Because this is script context, both refs use explicit `.value`.

- **`box-sizing` requirement on `.schedule-page`.** The project's global reset in `frontend/src/app.css:4` sets `box-sizing: border-box` on `*`. With `border-box`, `max-width: 640px` includes padding, so a `padding-right: 380px` would shrink the content area from 640 to 260px when the sidebar is open — visually breaking the schedule. Explicitly override the box-sizing on `.schedule-page` only:

  ```css
  .schedule-page {
    box-sizing: content-box;
    /* existing rules: max-width: 640px; margin: 0 auto; min-height: 100vh; ... */
    padding-right: var(--chat-sidebar-width, 0);
  }
  ```

  This keeps the 640px content cap intact while the padding takes effect outside it. No other element on the page needs to change — the override is scoped to the single rule that uses the new padding.

## Phase 5 — Automated vitest tests

Files: extend `frontend/tests/CommandBar.test.ts`, extend `frontend/tests/Schedule.test.ts`, new `frontend/tests/ChatSidebar.test.ts`, new `frontend/tests/chatSidebarStorage.test.ts`, new `frontend/tests/useViewport.test.ts`.

The project already ships a vitest suite (see `frontend/package.json`'s `"test": "vitest run"`); each new component must have a matching test before this feature ships.

### 5.0 Shared cleanup contract

Every new **or modified** test file in this feature (including the existing `CommandBar.test.ts` and `Schedule.test.ts` once they grow viewport/localStorage cases) MUST **register a new `afterEach` block OR extend the existing one** so it covers:

1. Unmounts the wrapper if mounted (`wrapper?.unmount(); wrapper = null`) — prevents detached DOM nodes from bleeding focus/listeners into later cases.
2. Clears localStorage (`localStorage.clear()`) — `chatSidebarStorage.test.ts` and `Schedule.test.ts` both write to the same key, and vitest runs files in parallel only across workers; within a worker tests share the JSDOM instance.
3. Restores stubbed globals (`vi.unstubAllGlobals()`) — needed wherever `matchMedia` is stubbed (Schedule.test.ts, useViewport.test.ts).
4. Resets mocks (`vi.clearAllMocks()` and any module-level refs back to initial values, mirroring the existing pattern at `frontend/tests/CommandBar.test.ts:71`).

These tests touch global state (localStorage, matchMedia) — without disciplined cleanup, the order in which vitest happens to run cases starts to matter, and CI failures become non-reproducible locally. `CommandBar.test.ts` already has an `afterEach`; extend it (do NOT add a second one — vitest runs all `afterEach` hooks in registration order, and duplicate cleanup either no-ops or papers over a bug that should be caught).

### 5.4 `chatSidebarStorage.test.ts` (new)

### 5.5 `useViewport.test.ts` (new)

The composable is small but easy to break (subscription/unsubscription against `MediaQueryList`). Direct coverage:

- Initial `isWide.value` mirrors `window.matchMedia(WIDE_VIEWPORT_QUERY).matches` at mount time. Stub `matchMedia` to return `matches: true` → `isWide.value === true`; flip the stub to `matches: false` and re-mount → `isWide.value === false`.
- The composable subscribes to the `MediaQueryList.change` event via `addEventListener`. Firing a synthetic change event updates `isWide.value` reactively.
- On component unmount, `removeEventListener` is called with the same handler reference. Verified by asserting the spy's `mock.calls` count matches between `addEventListener` and `removeEventListener` — leaking listeners across page navigations would be a memory regression.
- **Breakpoint boundary — inclusive `min-width: 1024px`.** Stub `matchMedia` so that the query `(min-width: 1024px)` returns `matches: true` for exactly 1024px and `false` for 1023px. Mount the composable twice (once per stub) and assert `isWide.value === true` at 1024 and `false` at 1023. Off-by-one in the breakpoint string would slip past the high-level Schedule tests but explode this one — a fast loud failure.

## Phase 6 — Manual test plan

Files: `docs/features/0008_MANUAL_TEST.md` (new).

Add scenarios covering:

1. Wide viewport (≥1024px), first visit — sidebar open by default, textarea visibly larger (≥6 rows), thread renders inside sidebar, no bottom dock.
2. Toggle sidebar — collapses to a 32px rail; schedule content remains fully visible and does not slide under the sidebar (the page's right padding tracks `--chat-sidebar-width`). Expand restores the 380px panel.
3. Persistence — reload after collapsing on wide; sidebar stays collapsed.
3a. Cross-viewport persistence — collapse on wide, then resize the window below 1024px (or open the same browser tab on a narrow screen / reload narrow). The bottom dock MUST be visible and functional — the persisted `false` from the wide session does NOT hide the only AI entry point on narrow viewports. This is enforced by Phase 4.3's `isWide`-takes-precedence rule but easy to regress, so verify visually.
3b. Breakpoint boundary — open Chrome/Firefox DevTools, set the viewport to **exactly 1024×768**. The sidebar variant MUST render (right-side panel, no bottom dock). Resize to **1023×768**. The dock variant MUST render (no sidebar). This catches off-by-one mistakes in the `matchMedia` query that would otherwise slip past automated tests if those tests also got the boundary wrong. Companion to the vitest `useViewport.test.ts` boundary case in Phase 5.5.
3c. Horizontal-overflow check at the boundary — at exactly 1024px viewport (sidebar open), the geometry is:

- `.schedule-page`: 640px content + 380px right padding = **1020px total layout width** (content-box), centered with `margin: 0 auto` → 2px margin on each side of viewport.
