# Feature 0027 ‚Äî Habitica sidebar: preserve Habitica app order

GitHub issue #106. Habitica tasks in the Day Forge left sidebar should appear in
the **same order as in the Habitica app**, within each type: **dailies first**
(in Habitica daily-list order), then **todos** (in Habitica todo-list order).

Habitica v3 does **not** store drag-order on task documents. Order lives in
`user.tasksOrder` (per-type ID arrays), and `GET /api/v3/tasks/user?type=todos|dailys`
returns tasks **already sorted server-side** to match the app. Day Forge must
capture that array order at fetch time and preserve it through filtering ‚Äî not
re-sort by due date/title/id and not read a nonexistent `position` key from the
payload.

Today `fetch_tasks_for_date` re-sorts after fetch by
`(-daily, due_date, title, id)`, which discards the API's pre-sorted order.
Filtering (due / overdue-carry / undated / `isDue` / exclude completed) stays
unchanged ‚Äî only **relative order among the filtered subset** should match
Habitica.

Related: #73 / PR #101 (`feature/0024-habitica`). No frontend changes required:
`HabiticaTasksSection.vue` renders `tasks` in API order; the fix is server-side
sort only.

### Out of scope

- Separate UI sub-headers ("Dailies" / "To Dos").
- Interleaving types into one mixed column.
- Client-side re-sort.
- Changing complete / poll / cache semantics beyond order of the cached payload.
- Exposing `position` in the JSON API (optional; not needed if sort is server-side).
- Fetching or parsing `user.tasksOrder` directly (the per-type GET responses
  already reflect it).

## Current state

**`backend/habitica_sync/schemas.py`** ‚Äî `NormalizedHabiticaTask` has
`id`, `title`, `type`, `due_date`, `completed`; no `position`.
`normalized_task_to_dict` serialises those five fields.

**`backend/habitica_sync/service.py`** ‚Äî `_normalize_todo` / `_normalize_daily`
read `id`, `text`, `date`/`completed`/`isDue` but do not record fetch-order.
`_fetch_tasks` returns the API array as-is (Habitica pre-sorts via
`tasksOrder`); the service then re-sorts after filtering:

```python
normalized.sort(
    key=lambda t: (
        -int(t.type == "daily"),
        t.due_date or "9999-12-31",
        t.title.casefold(),
        t.id,
    )
)
```

**`backend/habitica_sync/views.py`** ‚Äî the `tasks` view (function
`tasks`, `views.py:119`) calls
`fetch_tasks_for_date`, maps via `normalized_task_to_dict`, caches and returns
`{"tasks": [...]}`. Order in the response is whatever the service returns.

**`backend/tests/test_habitica_sync_service.py`** ‚Äî filter/carryover/error
tests. Some existing order assertions match the **old due-date sort only**,
not the fixture array order, and **will break** under the new sort:
`test_fetch_filters_todos_and_due_dailies_for_client_today` asserts
`[daily-due, overdue, today, undated]`, but its todos fixture is ordered
`[today (07-22), overdue (07-21), undated]` (indices 0/1/2 after filtering
`done`/`future`). The new fetch-order sort therefore yields
`[daily-due, today, overdue, undated]` ‚Äî the `overdue`/`today` pair flips.
This test's expected output **must be updated** (see ¬ß4). The
carryover/future tests (`test_client_today_carryover_‚Ä¶` with 1 daily + 1 todo;
`test_future_exact_date_skips‚Ä¶` with 1 todo) each return a single-item
per-type subset, so their assertions are order-invariant and stay green.

**Docs** ‚Äî `docs/features/0024_PLAN.md` ¬ß A.3 step 6 documents the
`due_date ‚Üí title ‚Üí id` sort. `RULES.md` Habitica gotchas mention filtering
but not sort order.

## Changes

### 1. Schema ‚Äî `backend/habitica_sync/schemas.py`

Add `position: int` to `NormalizedHabiticaTask` (internal sort key ‚Äî **not**
sourced from the Habitica payload; see ¬ß2).

Do **not** add `position` to `normalized_task_to_dict` unless a future consumer
needs it ‚Äî the sidebar only needs stable server-side order. Keeps
`HabiticaTask` / `docs/api.md` response shape unchanged.

### 2. Normalisation ‚Äî `backend/habitica_sync/service.py`

Habitica task JSON has **no** `position` field. Derive order from the **array
index** of each task in the `_fetch_tasks` response (Habitica has already
applied `tasksOrder` before returning the list).

Add an optional `position: int = 0` parameter to `_normalize_todo` and
`_normalize_daily`; pass it from the fetch loops:

```python
for idx, raw in enumerate(_fetch_tasks(account.api_user_id, token, "todos")):
    try:
        task = _normalize_todo(raw, position=idx)
        ...
```

Same pattern for the dailies loop (`enumerate` over the `dailys` response).

**Important:** assign `position=idx` **before** filtering so filtered subsets
keep relative order among included tasks. Do **not** read `raw.get("position")`
or add a `_read_position` helper ‚Äî that key is absent in live API responses.

### 3. Sort ‚Äî `fetch_tasks_for_date` in `service.py`

Replace the sort key with Habitica app order:

```python
normalized.sort(
    key=lambda t: (
        -int(t.type == "daily"),  # dailies block first (unchanged)
        t.position,               # ascending fetch-order within type (from enumerate)
        t.id,                     # stable tie-break when positions equal
    )
)
```

Remove `due_date` and `title` from the sort key entirely.

**Algorithm (unchanged fetch, new sort):**

1. Fetch todos (`GET ...?type=todos`); normalise + filter ‚Üí append to list.
2. If client-today or `include_overdue_carryover`: fetch dailies
   (`type=dailys`); keep `isDue` and not completed ‚Üí append.
3. Sort the combined filtered list with the key above.
4. Return.

Relative order among included tasks of the same type matches Habitica because
each task carries the index it had in the pre-sorted API response; ties break
on `id`.

### 4. Tests ‚Äî `backend/tests/test_habitica_sync_service.py`

**New test** `test_fetch_preserves_api_array_order_within_type` (name flexible):

- Mock `_fetch_tasks` to return todo and daily lists where **array order**
  contradicts due-date/title/id order (e.g. a later-due todo appears before an
  earlier-due todo in the mocked list; dailies in non-alphabetic title order).
- Mock payloads must **omit** any `position` key ‚Äî order is defined solely by
  list order, matching live Habitica responses.
- Assert returned `[(id, type), ...]` matches:
  - all dailies first, in the order they appeared in the dailies mock list;
  - then all todos, in the order they appeared in the todos mock list.

**New test** `test_fetch_filtered_subset_preserves_relative_api_order`:

- Mock todos list with four tasks in a deliberate array order; only a subset
  passes the day filter (e.g. indices 1 and 3 included, 0 and 2 filtered out).
- Assert included tasks appear in the same relative order as in the full mock
  list (`[task-at-index-1, task-at-index-3]`).

**New test** `test_fetch_position_tiebreaks_on_id`:

- Two tasks same type, same enumerate index is impossible in practice, but if
  positions were ever equal ‚Üí order by `id` ascending. Construct two
  `NormalizedHabiticaTask` instances with `position=0` directly (unit-level
  sort-key assertion) or document that enumerate makes ties impossible and
  skip if redundant.

**Update existing fixtures** ‚Äî do **not** add `"position"` to mocked Habitica
payloads. Instead, order mock lists to reflect the intended Habitica app order
and update assertions to match array order (the behaviour under test):

- `test_fetch_filters_todos_and_due_dailies_for_client_today` **must change**:
  its expected output flips from `[daily-due, overdue, today, undated]` to
  `[daily-due, today, overdue, undated]` (todos fixture is ordered
  `today, overdue, undated`, so fetch-order preserves that within the todo
  block). This is a required edit, not conditional.
- `test_client_today_carryover_uses_target_date_boundary_for_todos_and_dailies`
  returns a single daily + single todo, so its `[daily-due, server-day-overdue]`
  assertion is unaffected (dailies-first key still puts the daily first). No
  change needed, but re-confirm after the sort edit.

**View tests** (`backend/tests/test_habitica_sync_views.py`) ‚Äî no change expected;
they mock `fetch_tasks_for_date` / construct `NormalizedHabiticaTask` directly.
Any direct `NormalizedHabiticaTask(...)` constructions need the new
`position` argument (default `0` if the dataclass does not default it ‚Äî prefer
an explicit field with default `0` on the dataclass to minimise call-site churn).
