# 0019 — Sound notifications at block start/end (issue #56)

## Context

Play a distinct synthesized audio cue when the wall clock crosses a time
block's **start** or **end** boundary, so the user is alerted without
watching the schedule. Frontend-only Vue 3 feature. Opt-in (default
**off**), persisted across reloads. Two tones via the **Web Audio API**
(no external assets, no CDN): a rising chime for start, a falling chime
for end. Fires even when the tab is backgrounded (via
crossed-since-last-sample detection that recovers boundaries skipped by a
throttled/coalesced background tick — see Phase 2), with **two documented
limitations**: (1) the first boundary's tone may be dropped only in a
brand-new browser session where the persisted flag is `true` but no
gesture has occurred yet this page-load; (2) a tab left open **across
midnight** stops firing until the user navigates the date, because the
shared `useNowMinutes` sampler leaves-today and does not re-arm its timer
on rollover (see Phase 2 "midnight behavior"). No browser
`Notification` permission required.

This rides on the existing 60-second clock sampler `useNowMinutes`
(`frontend/src/composables/useNowMinutes.ts`), which already exposes a
reactive `nowMinutes` (integer minutes-since-midnight) and `currentHHMM`
(`"HH:MM"` string) ref, the same granularity
the block boundaries are stored at (`TimeBlock.start_time` /
`TimeBlock.end_time` are `"HH:MM"` strings — see
`frontend/src/types/index.ts`).

### Grounded facts the design depends on

- `useNowMinutes(viewedDate: Ref<string>)` ticks every `60_000`ms while
  `viewedDate === todayString()`. Off-today it sets `nowMinutes`/`nowDate`
  to `null`, `currentHHMM` to `""`, and runs **no** interval. It is a
  per-component-instance composable (one instance per `Schedule.vue`
  mount). Schedule.vue already calls it:
  `const { nowMinutes, nowDate } = useNowMinutes(toRef(props, "date"))`
  (Schedule.vue:203). `currentHHMM` is currently consumed by
  `SkippedTasks.vue` only.
- Blocks reach `Schedule.vue` as the Inertia `blocks` prop
  (`props.blocks: TimeBlock[]`). Add/remove/edit go through `useSchedule`
  mutations which call `router.reload({ only: ["blocks", "schedule"] })`,
  replacing `props.blocks` — so any `watch`/`computed` over `props.blocks`
  updates reactively without a remount. Date navigation also re-flows
  `props.blocks` and re-anchors `useNowMinutes` via the `viewedDate` ref.
- `TimeBlock` shape (`frontend/src/types/index.ts`):
  `{ id: number; title: string; start_time: string /* "HH:MM" */;
  end_time: string /* "HH:MM" */; category; is_completed: boolean;
  sort_order: number }`.
- `todayString()` and `toLocalDateString(date)` live in
  `frontend/src/utils/date.ts` (local-tz `"YYYY-MM-DD"`).
- localStorage persistence convention: `frontend/src/utils/chatSidebarStorage.ts`
  — namespaced key (`"day-forge:..."`), `try/catch` around both read and
  write, defensive default on missing/malformed value, swallow write
  errors for private-mode browsers.
- Settings UI is composed of self-contained sectioned components imported
  into `frontend/src/pages/Settings.vue` (e.g. `DesignSelector.vue`).
- Tests: `frontend/tests/*.test.ts` (NOT co-located). Stack: `vitest` +
  `@vue/test-utils` (`mount` + a `defineComponent` harness for
  composables, see `tests/useNowMinutes.test.ts`), fake timers via
  `vi.useFakeTimers()` / `vi.advanceTimersByTime`, global mocking via
  `vi.stubGlobal(...)` (see `tests/chatSidebarStorage.test.ts`). Run with
  `cd frontend && npm test`.

### Scope decision (verbatim from issue)

The issue's "Proposed approach" step 1 says the setting lives in
**"frontend state / localStorage"** with a "UI toggle in settings panel,
default false" — explicitly a client-only preference, distinct from the
server-persisted `theme` (`/api/user/preferences/`). This plan therefore
persists `soundNotifications` in **localStorage** (matching
`chatSidebarStorage`), NOT via the preferences API. No backend changes.

## Phase 2 — `useSoundNotifications` composable

File: `frontend/src/composables/useSoundNotifications.ts`.

### Public signature

```
useSoundNotifications(
  nowMinutes: Ref<number | null>, // from useNowMinutes (mins-since-midnight or null)
  nowDate: Ref<string | null>,    // from useNowMinutes ("YYYY-MM-DD" or null)
  getBlocks: () => TimeBlock[],   // getter over props.blocks (avoids stale capture)
)
```

The detector watches `nowMinutes` (the integer minute), **not**
`currentHHMM`, because the crossed-since-last-sample logic does interval
arithmetic on minutes-since-midnight (see "Boundary-cross detection").
Block `start_time`/`end_time` strings are parsed to minutes per tick via
`hhmmToMinutes`.

Returns at minimum `{ enabled, setEnabled }` where `setEnabled(v)` writes
through to storage and flips the ref AND attempts to resume/unlock the
`AudioContext` (see autoplay-policy handling). The composable owns the
storage write so the toggle component stays presentational. (`enabled`
may instead be created internally from `readSoundNotificationsEnabled()`
and returned — see "Ownership" below.)

### Ownership of the `enabled` ref

Create the `enabled` ref **inside** the composable, initialised from
`readSoundNotificationsEnabled()`, and return it together with
`setEnabled`. Both `Schedule.vue` (detector) and
`SoundNotificationToggle.vue` (UI) need the same value; since the
composable is **per-instance**, each surface that calls
`useSoundNotifications()` gets an independent ref. To keep a single
source of truth across the two surfaces without a store, the toggle is
rendered on the **Settings page** and the detector runs on the
**Schedule page** — they are never mounted simultaneously, and each reads
the persisted localStorage value on its own mount. This avoids
introducing a Pinia/global-state dependency the codebase does not use.

**Verified — Inertia remounts the page on navigation.** `app.ts` uses a
plain `createInertiaApp` with no persistent/shared `layout` (pages are
bare Inertia page components; `Schedule.vue` and `Settings.vue` declare
no `layout`). Inertia's default without a persistent layout is to swap
(unmount old, mount new) the page component on every navigation. So
back-navigation from Settings → Schedule **remounts** `Schedule.vue`,
running a fresh `setup()` that re-reads `readSoundNotificationsEnabled()`
— the detector cannot be a stale cached instance with `enabled=false`.
The persisted flag is the single source of truth and is re-read on each
mount. (The `enabled` *ref* being per-instance is therefore fine; it is
the **AudioContext** that must be shared across instances — handled by the
module-level singleton in "Web Audio tone synthesis" above, which is what
actually makes the Settings-toggle gesture unlock the Schedule detector's
audio.)

`setEnabled(v)`:
1. `writeSoundNotificationsEnabled(v)`.
2. `enabled.value = v`.
3. If `v === true`, call `unlockAudioContext()` from the shared
   `audioContext.ts` module. This runs inside the user's click handler,
   satisfying the autoplay gesture requirement, and resumes the
   **shared singleton** context that the Schedule detector will reuse.

### Web Audio tone synthesis — shared module-level singleton

**Context identity matters.** The toggle lives on the **Settings** page
and the detector lives on the **Schedule** page; with a per-instance
context, the gesture that unlocks the Settings page's context would
unlock a *different* context than the one the Schedule detector later
constructs — so the detector's context would be `suspended` again and the
"first boundary dropped" caveat would apply to **every** session where
the user enabled via Settings (the common path), not a rare edge case.

To avoid that, the `AudioContext` is a **module-level singleton** owned by
a small shared module `frontend/src/utils/audioContext.ts`, imported by
both the setting helper and the detector. One unlocked context is reused
across page navigations, so a gesture on the Settings toggle unlocks the
*same* context the Schedule detector plays through.

`frontend/src/utils/audioContext.ts` exports:
- `getAudioContext(): AudioContext | null` — lazily constructs
  `new (window.AudioContext || (window as any).webkitAudioContext)()` on
  first call (do **not** construct at module load — that can throw or
  auto-suspend before any gesture), caches it in a module-local variable,
  and returns it. Returns `null` if Web Audio is unavailable
  (constructor missing or throws), wrapped in `try/catch`.
- `unlockAudioContext(): void` — `getAudioContext()` then
  `void ctx?.resume().catch(() => {})`. Called from inside a user-gesture
  handler (the toggle click). Idempotent.
- `closeAudioContext(): void` — closes and nulls the cached context
  (`void ctx?.close().catch(() => {})`). **Refcounted** — see Cleanup.

Because the context is a singleton, **no composable instance constructs
or closes it directly.** This is consistent with the rest of the plan:
the composable is still per-instance, but the *audio context* is shared
process-wide via this module. The earlier "per composable instance"
language is replaced by this singleton ownership.

**Autoplay-policy gotcha (must handle):** a freshly constructed
`AudioContext` starts in `state === "suspended"` until a user gesture
resumes it. The opt-in toggle click **is** that gesture, so
`setEnabled(true)` calls `unlockAudioContext()`. Because that resumes the
**shared** singleton, the Schedule detector that runs after
back-navigation plays through an already-running context — no dropped
first boundary in the enabled-via-Settings path. The detector's
`playSound` still attempts `ctx.resume()` defensively (cheap no-op if
already running) and catches rejection silently, which covers the only
remaining suspended-on-arrival case: a brand-new browser session where
the persisted flag is `true` but **no** gesture has occurred yet in this
page load (the singleton is freshly reconstructed and suspended). In that
narrow case the first boundary's tone may be dropped by the browser;
subsequent ticks succeed once any gesture occurs (e.g. a block toggle).
This is the genuine, unavoidable autoplay-policy floor — not the common
path.

`playSound(type: SoundEventType)` algorithm:
1. `ctx = getAudioContext()` (the shared singleton). If `null` (no Web
   Audio), return.
2. `void ctx.resume().catch(() => {})` (defensive; no-op if already running).
3. Create an `OscillatorNode` and a `GainNode`; connect
   `osc → gain → ctx.destination`.
4. Apply a short two-note envelope (~400ms total):
   - **start** (rising chime): first note lower, second note higher
     (e.g. ~660Hz then ~880Hz), each ~150–200ms.
   - **end** (falling chime): first note higher, second note lower
     (e.g. ~880Hz then ~660Hz).
   - Schedule the two frequencies on the oscillator via
     `osc.frequency.setValueAtTime(...)` at `ctx.currentTime` and
     `ctx.currentTime + noteDuration`.
   - Shape the `gain` to avoid clicks: ramp up from ~0 to a low peak
     (~0.15) at the start, ramp/exponential-decay to near-zero by the
     end (`gain.gain.setValueAtTime` / `linearRampToValueAtTime` /
     `exponentialRampToValueAtTime`).
5. `osc.start(ctx.currentTime)`; `osc.stop(ctx.currentTime + totalDuration)`.
   Let GC reclaim the nodes after `stop` (or attach an `onended` cleanup;
   not strictly required).
6. Wrap the whole body in `try/catch` and swallow errors so a flaky audio
   stack never breaks the schedule UI.

Exact frequency/timing values are implementation detail and left to the
implementer; the only contract the tests assert is: start ≠ end tone, and
both produce an oscillator + gain wired to `destination`.

### Boundary-cross detection (the "clock tick hook")

The detector is a `watch` on `nowMinutes` (the integer minute
`useNowMinutes` mutates every 60s). It does NOT create its own interval —
it piggybacks on the single existing sampler, which already self-suspends
off-today.

**Crossed-since-last-sample, NOT exact equality.** Background-tab
`setInterval` timers are throttled/coalesced by the browser (Chromium's
intensive throttling after ~5 min hidden aligns wake-ups to ~1-min
buckets and can drift), and a foreground tick can be delayed by
main-thread jank. Either way a 60s sampler can skip the exact wall-clock
minute a boundary lands on — it may sample `13:01` and never observe
`13:00`. An exact `start_time === currentHHMM` test would silently miss
that boundary with no recovery. The detector therefore fires every
boundary in the **half-open minute interval `(lastSeen, now]`**, so a
single coalesced tick that jumps several minutes still fires each
boundary it leapt over. This is what makes the "fires even when the tab
is backgrounded" criterion actually achievable; exact-equality cannot
meet it.

Work in **minutes-since-midnight integers** (0–1439), not `"HH:MM"`
strings, so interval arithmetic is trivial. `useNowMinutes` already
exposes `nowMinutes` (the integer form of `currentHHMM`); feed that to
the detector instead of `currentHHMM`. Parse each block's
`start_time`/`end_time` once per tick into minutes via a small
`hhmmToMinutes("HH:MM") → number` helper.

State per detector instance:
- `lastSeenMinute: number | null` — the `nowMinutes` value observed on
  the previous tick for the current date; `null` before the first tick
  of a date (and reset to `null` on date change, see reset semantics).
- `fired: Set<string>` keyed by `` `${type}:${block.id}:${date}` ``
  (event type + block id + date). The fired-Set is a **second, idempotent
  guard** layered on the interval logic: it prevents a re-fire if the
  same date is re-entered mid-day, or if `props.blocks` re-flows
  mid-minute from an unrelated `router.reload` and the watch re-runs at
  the same `nowMinutes`. Independent keys mean two blocks sharing the same
  boundary minute each fire, and start vs end at the same minute each
  fire.

On each `nowMinutes` change (run handler steps in this order):
1. If `!enabled.value` → return (no sound when setting off). This is the
   primary suppression gate; keep it first so a disabled setting short-
   circuits before any block iteration.
2. If `nowDate.value === null` or `nowMinutes.value === null` → return
   (off-today; `useNowMinutes` already guarantees these are null
   off-today, so no future-date block ever fires — satisfies "no sounds
   for past/future date navigation"). Note `lastSeenMinute` is **not**
   updated here; the date-change watch resets it (see below).
3. Let `now = nowMinutes.value`, `date = nowDate.value`,
   `prev = lastSeenMinute`.
4. **First tick of a date** (`prev === null`): fire only boundaries that
   land **exactly on `now`** — do NOT back-fill the whole day from
   midnight. (A page opened at 14:00 must not blast every morning
   boundary at once.) Concretely, treat the eligible window as the single
   point `now`, then proceed to step 6. (`lastSeenMinute` is set in step
   7 as usual.)
5. **Subsequent ticks** (`prev !== null`): the eligible window is the
   half-open interval `(prev, now]`. Normal forward progress makes this
   `(prev, prev+1, …, now]`; a coalesced/throttled tick that jumped makes
   it span every minute leapt over. (If `now <= prev` — clock moved
   backwards, e.g. DST fall-back or a manual clock change — fire nothing
   and just set `lastSeenMinute = now`; the date-change reset handles the
   genuine new-day case, and refusing to fire on a backward step avoids a
   burst of stale boundaries.)
6. For each `block` in `getBlocks()`, compute
   `s = hhmmToMinutes(block.start_time)` and
   `e = hhmmToMinutes(block.end_time)`:
   - If `s` is in the eligible window (step 4: `s === now`; step 5:
     `prev < s <= now`) and `!fired.has(` `start:${block.id}:${date}` `)`:
     `playSound("start")`; `fired.add(...)`.
   - If `e` is in the eligible window and
     `!fired.has(` `end:${block.id}:${date}` `)`:
     `playSound("end")`; `fired.add(...)`.
7. Set `lastSeenMinute = now`.

Because each boundary is gated by both the half-open window AND the
fired-Set, there is no double-fire even if `props.blocks` re-flows
mid-minute from an unrelated `router.reload` (the watch re-runs with
`now === prev`, so the window `(prev, now]` is empty and nothing fires).
New blocks added while the page is open are picked up automatically
because `getBlocks()` reads the live `props.blocks` on every tick — no
separate "watch list" to maintain. A block whose boundary minute already
passed earlier in the current session does NOT retro-fire when added,
because the window only covers `(prev, now]`.

### fired-Set + lastSeenMinute reset semantics (precise)

Reset both `fired` and `lastSeenMinute` on **date change** so navigating
away and back, then forward into the new today, does not permanently
suppress a boundary. Watch `nowDate` (the date `useNowMinutes` owns):

- When `nowDate` transitions to a **new non-null date** (i.e. explicit
  date navigation into a *different* today via the `viewedDate` ref):
  `fired.clear()` and `lastSeenMinute = null`. The `null` forces the next
  `nowMinutes` tick onto the "first tick of a date" branch (exact-`now`
  firing only, no back-fill).
- When `nowDate` transitions to `null` (left today / off-today): no
  firing happens anyway (gate step 2). Set `lastSeenMinute = null` here
  too (cheap, keeps the next entry clean); the `fired.clear()` on the
  next non-null entry is sufficient for the Set.

**Important — `useNowMinutes` midnight behavior (verified against source
+ `tests/useNowMinutes.test.ts`):** when the wall clock crosses midnight
on an open tab **without navigation**, `useNowMinutes.tick()` calls
`leaveToday()`, which **clears the interval and sets `nowDate = null`** —
and does **NOT** arm a fresh interval for the new calendar day
(`tests/useNowMinutes.test.ts:138-165` asserts `getTimerCount() === 0`
and `setInterval` call count stays at 1 after rollover). So a tab left
open across midnight **silently stops ticking**: `nowDate` freezes at
`null`, `currentHHMM`/`nowMinutes` freeze at `""`/`null`, and the
detector fires **no boundaries on the new day** until the user performs
an explicit date navigation that re-anchors `viewedDate` to the new
today (which re-arms the interval and triggers the `nowDate`-transition
reset above). This is an **inherited limitation of the shared sampler**,
not something this feature can fix without adding its own timer (which
the plan explicitly forbids — see Notes). It is reflected in the
acceptance criteria and the reset/navigation test (Phase 5 Test #9).

The only real reset trigger is therefore **explicit date navigation into
a new today** (the `null → non-null` or `dateA → dateB` `nowDate`
transition). There is no "midnight rollover re-enters today" transition —
the composable does not produce one.

Keying the fired entries by `date` is belt-and-suspenders on top of the
clear: even without the clear, a key like `start:42:2026-06-15` cannot
collide with `start:42:2026-06-16`. The explicit `nowDate`-watch clear
keeps the Set from growing unbounded across the days a long-lived tab
actually re-enters via navigation.

**Edge case — enabling mid-minute:** while `enabled === false` the watch
returns at gate step 1 **without** advancing `lastSeenMinute` (the
disabled gate precedes step 7). So when the user toggles **on** at, say,
09:30:40 with a block starting at 09:30, `lastSeenMinute` is still `null`;
the next tick (09:31) runs the **first-tick-of-a-date branch**, which
fires only boundaries exactly at 09:31 — the already-passed 09:30 boundary
does NOT retroactively fire. This matches the acceptance criterion "plays
at each block's start minute" for sessions where the setting was already
on — a mid-minute enable takes effect from the next boundary. (Treating
the enable as a fresh first-tick is intentional: it also prevents a window
`(prev, now]` from back-firing every boundary that passed while the
setting was off, which would be jarring.) Document this in a code comment;
do not add a catch-up pass.

### Cleanup

The `watch`/`watchEffect` stop handles are auto-disposed by Vue on
unmount (composable called in `setup()`) — nothing to do for those.

The shared singleton `AudioContext` is **NOT** closed on a single
composable unmount, because the same context must survive the
Settings → Schedule navigation (closing it on the Settings unmount would
re-suspend/destroy exactly the context the Schedule detector needs).
Manage it with a tiny refcount in `audioContext.ts`:

- Each composable that wants audio calls `acquireAudioContext()` in
  `setup()` (increments the refcount; lazily constructs nothing on its
  own — construction is still deferred to `getAudioContext()`/
  `unlockAudioContext()`).
- `onUnmounted`, each calls `releaseAudioContext()` (decrements). Only
  when the refcount hits **0** does the module actually
  `closeAudioContext()` and null the cached handle, so a later remount
  rebuilds it.

This keeps the context alive across the brief Settings↔Schedule unmount
gap (the decrement-to-0 path only runs when *no* surface holds it, e.g.
full app teardown) while still avoiding a leaked, never-closed context.
If the refcount bookkeeping is judged not worth it for a two-surface app,
the acceptable fallback is to **never** close the singleton (a single
suspended/running `AudioContext` for the app lifetime is cheap) and drop
`acquire`/`release` entirely — but do **not** close it on per-instance
unmount.

  `"{}"`) → `false`
- write `true`/`false` round-trips; `setItem` throwing does not throw out
  of `writeSoundNotificationsEnabled`.

## Notes / pitfalls

- **Do not create a second interval.** The whole feature hangs off the
  existing 60s `useNowMinutes` sampler. A separate timer would double the
  wake-ups and could drift out of phase with the now-line UI.
- **`webkitAudioContext` fallback** is needed for older Safari; guard the
  constructor access with `(window.AudioContext || (window as any).webkitAudioContext)`.
- **Backgrounded tab:** `setInterval` at 60s is throttled/coalesced but
  not killed when the tab is hidden, and a *running* (already-resumed)
  `AudioContext` emits audio without focus. The "fires in background"
  criterion is met by the **crossed-since-last-sample** logic (window
  `(prev, now]`): even when a hidden-tab tick is coalesced and jumps
  several minutes, every boundary it leapt over still fires on that one
  tick — exact-equality could not do this. The remaining preconditions are
  (a) the shared singleton context was unlocked by the toggle gesture
  (handled by the module-level singleton, so a Settings-page enable
  carries over to the Schedule detector), and (b) the tab has not crossed
  midnight without navigation (see next bullet). There is no way around
  the browser autoplay policy for a truly never-interacted page; the
  opt-in toggle is the unlock point and the issue accepts "catch
  NotAllowedError silently."
- **Midnight on an open tab stops notifications (inherited limitation).**
  `useNowMinutes.tick()` calls `leaveToday()` on rollover — it clears the
  interval and sets `nowDate=null` and does **not** re-arm a timer for the
  new day (verified: `tests/useNowMinutes.test.ts:138-165`). So a tab left
  open across midnight silently stops ticking and the detector fires
  nothing on the new day until the user navigates the date (which re-arms
  the sampler and triggers the detector's reset). This feature deliberately
  does **not** add its own timer to work around it (see "Do not create a
  second interval"); the limitation is documented in the acceptance
  criteria.
- **No `Notification` API** anywhere — Web Audio only, per acceptance
  criteria.
- Keep `playSound` fully `try/catch`-guarded; audio must never throw into
  the schedule render path.
