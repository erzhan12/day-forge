# Feature 0020 — Todoist Integration (read-only task panel)

## Context

Add a new `todoist_sync` Django app that mirrors the existing
`calendar_sync` (CalDAV) app one-for-one. A user connects their Todoist
account by pasting a **personal API token** into Settings; the token is
Fernet-encrypted at rest exactly like `CalDAVAccount.password_encrypted`.
A **read-only** panel on the Schedule page lists their Todoist tasks for
the currently-selected schedule date. No writes to Todoist, no OAuth, no
AI coupling in V1.

The Todoist API v1 host is **`https://api.todoist.com/api/v1`** (NOT the
documentation host `developer.todoist.com` — requests there 404/return
HTML and would surface as a confusing `TodoistProviderError` 502). The
host is parameterised as a `TODOIST_BASE_URL` setting (default
`https://api.todoist.com/api/v1`), mirroring `CALDAV_DEFAULT_BASE_URL`, so
it is not hardcoded.

The dedicated **filter** endpoint is `GET /api/v1/tasks/filter` (see the
Date → filter algorithm); it returns active (non-completed) tasks with
fields including `content` (title), `priority` (1–4 where 4 = P1/highest),
`due` (object — see below), `id`. The active/filter endpoints never return
completed tasks, so "not done" is implicit.

The task `due` object shape is `{date, timezone, is_recurring, string,
lang}` — there is **no** `datetime` field. The `date` field is
**polymorphic**: full-day tasks return `"2025-02-12"` (`YYYY-MM-DD`);
**timed** tasks return a full datetime in the *same* `date` key, e.g.
`"2018-11-15T12:00:00.000000"`. A task with no due date has `due == null`.

`calendar_sync` is the exact analog: reuse its structure, error
hierarchy, versioned cache, system checks, admin, frontend composables,
and panel component. Differences from CalDAV: single secret field (token,
not apple_id+password+base_url), an HTTP REST client (not a DAV library),
and the date→filter algorithm below.

---

## Out of scope (V1)

State explicitly — do **not** build any of these:

- **OAuth** — personal API token only.
- **Task creation / completion / editing** — read-only display.
- **Link-out** — no per-task "open in Todoist" link. (The task object has
  no `url` field; a link would require synthesising
  `https://app.todoist.com/app/task/<id>` — not built in V1.)
- **Bidirectional sync** — no writes to Todoist of any kind.
- **AI prompt integration** — task text is NOT fed into draft/command/chat
  prompts; no provider-egress of task content via the LLM service.
- **Project / section / label display** — no project-name chip.
- **Due-time display** — for timed tasks the time component of
  `due.date` is dropped; the panel shows date-only. (There is no
  `due.datetime` field — see the polymorphic `due.date` note in Context.)

## Date → filter algorithm

The panel follows the selected schedule date (`GET /api/todoist/tasks/<date>/`).
The service maps the selected date to a Todoist filter query and fetches
via the **dedicated filter endpoint `GET /api/v1/tasks/filter`** whose
**required** param is `query` (NOT `filter`; plain `GET /api/v1/tasks`
does *not* accept a free-text filter — only `project_id`/`section_id`/
`label`/`ids`). The endpoint is **cursor-paginated**: params `cursor` and
`limit` (default 50, **max 200**); the response envelope is
`{results: [...], next_cursor}` (NOT a bare array).

1. Resolve "today" in the project timezone (`django.utils.timezone.localdate()`,
   consistent with `settings.TIME_ZONE` used by `calendar_sync.service._day_window`).
2. **Pagination** — fetch *all* matching tasks, never a truncated page.
   Set `limit=200` and loop: issue `GET /api/v1/tasks/filter?query=<q>&limit=200`,
   then while the response `next_cursor` is non-null re-issue with
   `&cursor=<next_cursor>`, concatenating `results` until `next_cursor` is
   `null`/absent. (A user with `today | overdue` sweeping a large overdue
   backlog can exceed one page; a single un-paginated call would silently
   drop tasks.)
3. **If `selected_date == today`** → `query = "today | overdue"`. This
   returns tasks due today **plus** all not-done tasks whose due date is in
   the past (overdue carryover). Carryover is only meaningful when the
   selected date is today.
4. **Else (past or future date)** → query for that exact date. Use the
   **single literal-date token** `query = "<YYYY-MM-DD>"`, which Todoist
   interprets as "due on that date" (this is `due:` semantics; do **not**
   use the `date:` keyword, which scopes by absolute date and ignores
   recurrence). Pin this exact token form in `service.py` and lock it with
   a test asserting the generated `query` string for a representative past
   and future date (see Testing).
5. The filter endpoint never returns completed tasks, so the "not done"
   half of the overdue rule needs no extra client-side filter.
6. Sort the resulting list deterministically before returning with a
   **null-safe key** — `due_date` is nullable (`due == null` →
   `due_date = None`, see schemas) and Python 3 raises `TypeError` the
   moment a `None` and a `str` date are compared in the same sort. Use a
   key that coerces `None`, e.g.
   `key=lambda t: (-t.priority, t.due_date or "", t.title.casefold(), t.id)`
   (highest priority first; `None` due date sorts as `""`). Do **not** use
   a bare `(priority desc, due_date, title, id)` tuple — it raises as soon
   as one no-due task is present, and the normaliser + its tests explicitly
   support `due_date = None`. This is an analogous final deterministic sort
   to `calendar_sync.service.fetch_events_for_date`'s `normalized.sort(...)`
   (the sort *keys* differ — priority/title here vs. start-time there).

> **Note — tasks with no due date never appear.** The date filter only
> matches tasks that have a `due`; a task with `due == null` matches
> neither `today | overdue` nor an exact-date query, so it is absent from
> every panel view. This is by design (the panel is date-scoped). The
> panel empty-state copy must therefore **not** imply the user has zero
> Todoist tasks — phrase it "No tasks scheduled for this day."

## Priority mapping

Todoist `priority` is an int `4..1` where **4 is the highest (P1)**. The UI
label is inverted:

| Todoist `priority` | UI flag | colored dot |
|--------------------|---------|-------------|
| 4 | P1 | red (highest) |
| 3 | P2 | orange |
| 2 | P3 | blue |
| 1 | P4 | grey (default/none) |

Mapping rule: `ui_p = 5 - todoist_priority`. The normaliser emits **both**
the raw `priority` int (for the deterministic sort) **and** a precomputed
`ui_priority` string (`P1..P4`) — the derivation lives in **one** place
(the normaliser) so the component stays purely presentational and the flag
is locked by a service test.

## Phase 2 — Backend service + views

- **`backend/todoist_sync/schemas.py`** — mirror `calendar_sync/schemas.py`:
  - `NormalizedTask` frozen dataclass. **The wire field for the task name
    is `title`** — single field, end-to-end (mirrors `calendar_sync`
    normalising VEVENT `SUMMARY` → `title` at `schemas.py:26` /
    `normalized_event_to_dict`). The normaliser maps Todoist's raw
    `content` → `title`; the name `content` never reaches the wire payload,
    `types/todoist.ts`, or `docs/api.md`. Fields: `id` (str), `title`
    (str), `priority` (int 1–4), `ui_priority` (str `P1..P4`, precomputed),
    and the normalised `due_date` (ISO date str `YYYY-MM-DD`, **nullable**
    — `None` when the source `due` is `null`; the time component of timed
    tasks is dropped, see § service.py `due` normalisation). Add
    `normalized_task_to_dict(task)` for the wire payload
    (`{id, title, priority, ui_priority, due_date}`).
  - `validate_account_payload(data)` → `(cleaned, errors)`: validate a
    single `token` string field (required, non-empty, length cap — Todoist
    tokens are ~40 hex chars; cap generously e.g. 128 like the CalDAV
    password cap). No email/URL validators needed. Returns the standard
    `errors` dict shape.

- **`backend/todoist_sync/service.py`** — mirror the service-boundary
  pattern (the secret is decrypted **only** here, never in views):
  - Typed error hierarchy: `TodoistError` (base), `TodoistAuthError`
    (401/403 from Todoist), `TodoistTimeoutError` (network timeout),
    `TodoistProviderError` (anything else). Mirrors
    `CalDAVError`/`CalDAVAuthError`/`CalDAVTimeoutError`/`CalDAVProviderError`.
  - HTTP client: use `requests` (already a transitive dep — see
    `calendar_sync/service.py` import of `requests.exceptions.Timeout`).
    Build all URLs from `settings.TODOIST_BASE_URL` (default
    `https://api.todoist.com/api/v1` — the real API host, never the
    `developer.todoist.com` docs host) with
    `timeout=settings.TODOIST_REQUEST_TIMEOUT` and an
    `Authorization: Bearer <token>` header. Map `requests.Timeout` →
    `TodoistTimeoutError`; `401/403` status → `TodoistAuthError`; other
    non-2xx / connection errors → `TodoistProviderError`. Never log the
    token.
  - `verify_credentials(token)` — prove the Bearer token works with a
    **single** cheap authenticated call: `GET /api/v1/projects?limit=1`
    (or `GET /api/v1/user`). Do **not** use `GET /api/v1/tasks/filter` as
    the probe — its `query` param is **required** (1..1024 chars), so there
    is no "trivial filter"; a missing/empty `query` is a 400, not an auth
    signal. Map `401/403` → `TodoistAuthError`, `requests.Timeout` →
    `TodoistTimeoutError`, anything else → `TodoistProviderError`. Returns
    `None` on success. Unlike CalDAV's two-endpoint probe (CalDAV can
    partially succeed on incomplete creds), a single Bearer-token call
    suffices here — the token is all-or-nothing, so no second probe is
    needed.
  - `fetch_tasks_for_date(account, target_date)` — the **only** call site
    of `account.get_token()`. Compute the filter per the **Date → filter
    algorithm** above, call `GET /api/v1/tasks/filter` (paginating —
    see step 2 of the algorithm), normalise each task to `NormalizedTask`
    (apply the priority mapping; normalise `due` — see below), sort
    deterministically, return `list[NormalizedTask]`. **Per-task
    resilience**: wrap each task's normalisation in `try/except` and on
    failure `logger.warning("Failed to normalize task (...); skipping")`
    + skip that task only — a single malformed task must not fail the
    whole fetch (mirror `calendar_sync.service`'s VEVENT skip at
    `service.py:163-168` + the `if ev is None: continue` at `service.py:259`).
    Wrap provider failures in the typed hierarchy; let `ImproperlyConfigured`
    (key-rotation) propagate so the view maps it to a config-shaped 500
    (mirror the `except (CalDAVError, ImproperlyConfigured): raise` arm).
    `del token` in a `finally` to drop the plaintext binding.
  - **`due` normalisation** — Todoist's `due` is either `null` or
    `{date, timezone, is_recurring, string, lang}` (no `datetime` field).
    Map to `NormalizedTask.due_date`:
    - `due is None` → `due_date = None`.
    - Else take `raw = due["date"]`. The field is **polymorphic**: if it
      contains a `"T"` (timed task, e.g. `"2018-11-15T12:00:00.000000"`)
      take the date component — `datetime.date.fromisoformat(raw[:10])`
      (equivalently `datetime.datetime.fromisoformat(raw).date()`); else
      it is a plain `YYYY-MM-DD` full-day date — `date.fromisoformat(raw)`.
    - A timed `due.date` may carry a fixed-offset/UTC suffix
      (e.g. `"2018-11-15T12:00:00Z"` or `"…+00:00"`). The `raw[:10]` date
      slice is suffix-agnostic and still yields the correct `YYYY-MM-DD`;
      if parsing the full value instead, use
      `datetime.fromisoformat(raw.replace("Z", "+00:00")).date()`
      (`fromisoformat` rejects a bare `Z` on older Python versions).
    - The time component is intentionally dropped (date-only display — see
      Out of scope). Do **not** call `date.fromisoformat(raw)` directly on
      the raw value: it raises on every timed task.

- **`frontend/src/composables/useTodoist.ts`** (new) — copy
  `useCalendar.ts`: `fetchEvents(date)` → `fetchTasks(date)` hitting
  `GET /api/todoist/tasks/<date>/`, with the dual commit guard
  (`latestRequestedTaskDate` + `tasksRequestSeq`). **Connected-state
  transitions intentionally diverge from `useCalendar.ts:99-113`** in one
  principled way that the Todoist panel's `!connected` gate requires:
  - on `ok` → `connected = true`, `statusKnown = true`.
  - on `503` (not-connected) → `connected = false`, `statusKnown = true`.
  - on **any non-503 task-fetch error** (`401`/`500`/`502`/`504`) →
    `connected = **true**`, `statusKnown = true`, `state.error` (via
    `statusToMessage`). Setting `connected = true` here is **sound, not a
    guess**: `GET /api/todoist/tasks/<date>/` returns `503` **only** when
    `TodoistAccount.DoesNotExist`, so reaching a `401`/`500`/`502`/`504`
    *proves* the account row exists — the account is connected; the
    provider/config call merely failed (revoked token, key-rotation 500,
    timeout, upstream 5xx). This is the fix for the **first-load race**: a
    fresh `useTodoist()` starts `connected = false`, so if a non-503 error
    left `connected` untouched (as CalDAV's `useCalendar.ts:112-113` does)
    the error would stay hidden behind the `!connected` panel gate until
    `fetchAccountStatus()` independently won the race and seeded
    `connected = true`. CalDAV does not hit this because its panel surfaces
    errors without depending on `connected` the same way; the Todoist panel
    gates purely on `connected`, so the composable must set it.
  `fetchTasks` therefore sets `statusKnown = true` on **every** non-abort,
  non-stale terminal path (`useCalendar.ts:112-113` omits this on its error
  branch — we do not replicate that). The abort early-return and the
  stale-commit-guard early-return intentionally touch **neither**
  `connected` nor `statusKnown` (they own no state). Cover both divergences
  (the `connected = true`-on-non-503-error and the
  `statusKnown`-on-every-path) with `useTodoist.test.ts` cases (see
  Testing). Also a `fetchAccountStatus()` reading
  `GET /api/todoist/account/` writing a disjoint state slice. State:
  `{ tasks, loading, error, connected, statusKnown }`. **`requestJson` GET
  footgun** (RULES.md:255-257): pass `undefined` as the **third** positional
  arg (the JSON body) and put `{ signal }` in the **fourth** —
  `requestJson(url, "GET", { signal })` would otherwise serialise the
  options object as the request body.
