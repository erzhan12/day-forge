# Phase 6: Analytics & End-of-Day Review

Daily review panel with completion stats, category breakdown, streak tracking, and analytics-as-context for the AI draft prompt. New `/analytics/<date>/` Inertia page exposes the panel; an explicit **"Mark reviewed"** button flips `Schedule.status` from `active` to `reviewed`. Editing any block on a `reviewed` schedule flips it back to `active` (review unfreezes — analytics recomputes on the next visit). Per-day completion ratios are appended to history lines in `build_draft_user_message` so future drafts learn from past follow-through. The existing `DailyReview` model (Phase 1) gains category/duration aggregates and an `updated_at` field; the row is recomputed every time the page is visited while the schedule is `active`, and frozen once `reviewed`.

The service layer reuses the architecture from Phase 4/5 (pure-function aggregators, view-as-thin-shell, atomic recompute). New work is largely additive.

**Decisions baked in (overridable):**

- `active → reviewed` flips on a deliberate **Mark reviewed** click, mirroring how Phase 5's `draft → active` waits for a real edit.
- `reviewed → active` flips automatically on any forward-mutating edit. The `mark_active_if_draft()` helper from Phase 5 is generalised to **`mark_active_on_edit()`** covering both `draft` and `reviewed` source states. Single method, same call sites — keeps the transition matrix in one place.
- Streak walks **calendar dates** backward from yesterday: a missing `Schedule` row = **break** (the user didn't plan that day); a zero-block schedule = **skip** (nothing to grade, doesn't break or count).
- AI prompt injection is per-day only: each history line in `build_draft_user_message` gets a `(completed: X/Y)` suffix when a `DailyReview` row exists.
- The review row is **recomputed on every GET** of `/analytics/<date>/` while `status != reviewed`. After review it is frozen, and `mark_reviewed` is **idempotent** — a second call on an already-reviewed schedule returns the persisted snapshot without recomputing.

## Phase 6A — Data Layer & Config

### `backend/analytics/models.py` + new migration

Extend `DailyReview` with category aggregates and a freshness timestamp:

- `planned_minutes_by_category: JSONField(default=dict)` — `{"work": 240, "personal": 60, ...}`. Sum of `(end - start)` minutes per category across all blocks.
- `completed_minutes_by_category: JSONField(default=dict)` — same shape, restricted to `is_completed=True`.
- `updated_at = DateTimeField(auto_now=True)` — distinguishes "fresh recompute" rows from frozen review snapshots.
- Add a `@property completion_rate` returning `completed_count / planned_count` as a float, or `None` when `planned_count == 0`. Pure-Python; no DB column. Pure derived data so it stays on the model — no service deps, no circular-import risk.

**No upsert classmethod on the model.** Persisting a recomputed review is a service-layer concern (writes + transaction boundaries), so it lives in `analytics/services.py` (see 6B). Putting it on the model would force `models.py → services.py → models.py` import cycles.

Migration `analytics/0002_review_aggregates.py`: adds the three new fields with defaults; back-fill is unnecessary (existing rows get `{}` and `auto_now`).

### `backend/schedules/models.py`

Generalise the Phase-5 helper, **but switch the implementation to a DB-conditional UPDATE** rather than the Python-side `self.status` check.

- **Replace** `Schedule.mark_active_if_draft() -> bool` with **`Schedule.mark_active_on_edit() -> bool`**. Flips `status="draft" → "active"` OR `status="reviewed" → "active"`. No-op when already `active`. Returns `True` when a flip occurred.

  Implementation must be a **conditional `Queryset.update`**, NOT a check on `self.status`:

  ```
  updated = Schedule.objects.filter(
      pk=self.pk,
      status__in=[self.Status.DRAFT, self.Status.REVIEWED],
  ).update(status=self.Status.ACTIVE)
  ```

  The Python-side check would consult `self.status` from a Schedule object loaded **before** a concurrent `mark_reviewed` ran, so a stale `ACTIVE` instance would short-circuit the flip even when the DB row is now `REVIEWED`. The conditional UPDATE acquires its row lock at the SQL level: under contention with `mark_reviewed`'s `select_for_update`, the UPDATE waits until the lock releases, then re-evaluates the WHERE clause against the current DB state. This is the **load-bearing piece** of the recovery property — without it, an edit racing a review would silently leave the schedule frozen-reviewed with stale snapshot data.

  Sync `self.status = ACTIVE` after a successful UPDATE so the in-memory object reflects the change for any caller that re-reads it without a refetch.

- Update **all** existing call sites (Phase 5 wired `mark_active_if_draft` into `create_block`, `block_detail` PATCH/DELETE, `reorder_blocks`, and `ai_command` gated on `len(actions) > 0`). Behaviour for the `draft` path is unchanged; the `reviewed` path is new.

- Add `Schedule.mark_reviewed_if_active() -> bool` for the forward direction. Symmetric implementation — also use a DB-conditional UPDATE for consistency, even though `mark_reviewed`'s caller holds `select_for_update` so staleness isn't a risk in practice. Single transition pattern across both helpers reduces cognitive load and prevents future call sites from accidentally bypassing the lock.

  Sync `self.status = self.Status.REVIEWED` after a successful UPDATE so the in-memory object reflects the change, mirroring `mark_active_on_edit` exactly. Without this the caller would hold a stale instance that says `ACTIVE`, which sooner or later traps a future feature.

  Refuses (returns `False`) on `draft` — a never-edited day cannot be reviewed (analytics would be meaningless on auto-draft data the user never touched). The WHERE clause `status="active"` makes this enforcement automatic: an UPDATE on a draft row matches zero rows.

The `Status.REVIEWED = "reviewed"` choice (declared in Phase 1) is finally used.

### `backend/day_forge/settings.py`

Two new env-driven knobs, both **validated at startup**:

- `ANALYTICS_STREAK_THRESHOLD` — default `0.8`. Must be a float in `[0.0, 1.0]` inclusive; out-of-range raises `ValueError` at import time. A misconfigured `1.5` would silently produce `streak=0` forever — fail loud instead.
- `ANALYTICS_STREAK_WINDOW_DAYS` — default `30`. Must be a positive int. Hard cap on how far back the streak query walks; prevents `O(forever)` scans on long-lived accounts.

Document both in `.claude/rules/project.md` and `.env.example`.

### `frontend/src/types/index.ts`

New interfaces, keyed off the existing union to prevent drift:

- `CategoryMinutes = Record<TimeBlock["category"], number>` — auto-stays in sync if a category is ever added or renamed.
- `DailyReview { id, schedule_id, date, status, planned_count, completed_count, skipped_count, completion_rate: number | null, planned_minutes_by_category: CategoryMinutes, completed_minutes_by_category: CategoryMinutes, notes: string, created_at: string, updated_at: string }` — `created_at` exposed for admin/debug parity even though the UI primarily reads `updated_at`.
- `StreakInfo { current: number; threshold: number; window_days: number }`

`Schedule.status` already accepts `"reviewed"` from Phase 5.

## Phase 6B — Backend service + view (parallel with 6C)

### `backend/analytics/services.py` — new file

Pure-ish functions, no Django HTTP layer.

#### `compute_review_stats(schedule, *, now=None) -> dict`

Pure aggregator. `now` is an optional `datetime`; defaults to `timezone.localtime()` only inside the call so tests can pin behaviour deterministically.

Returns a dict matching the `DailyReview` writable fields: `{planned_count, completed_count, skipped_count, planned_minutes_by_category, completed_minutes_by_category}`.

Algorithm:
1. `today = (now or timezone.localtime()).date()`.
2. Initialise both `*_minutes_by_category` maps to `{c.value: 0 for c in TimeBlock.Category}`.
3. For each block on the schedule (uses prefetched `time_blocks`):
   - `duration = (end_time - start_time)` in minutes.
   - `planned_count += 1`. `planned_minutes_by_category[block.category] += duration`.
   - If `block.is_completed`: `completed_count += 1`. `completed_minutes_by_category[block.category] += duration`.
   - Else compute `is_skipped`:
     - `schedule.date < today` → all uncompleted blocks are skipped (the day is over).
     - `schedule.date == today` → only blocks whose `end_time < now.time()` are skipped.
     - `schedule.date > today` → never skipped (analytics_view rejects future dates anyway).
   - If `is_skipped`: `skipped_count += 1`.
4. Return the dict.

Pure (given `now`), unit-testable without the DB. Tests live in `test_analytics_services.py`.

#### `recompute_review_from_schedule(schedule, *, now=None) -> DailyReview`

Lives in `services.py` (NOT on the model) to avoid circular imports.

1. `stats = compute_review_stats(schedule, now=now)`.
2. `DailyReview.objects.update_or_create(schedule=schedule, defaults=stats)` — `notes` is preserved because it's not in `defaults`.
3. Return the row.

Idempotent. `updated_at` advances on every call (auto_now).

#### `compute_streak(user, *, today=None) -> int`

Walks **calendar dates** backward from yesterday — a missing schedule on day `D` is a hard break, not a skip.

Algorithm:
1. `today = today or timezone.localdate()`.
2. `streak = 0`.
3. For `offset in range(1, ANALYTICS_STREAK_WINDOW_DAYS + 1)`:
   - `target_date = today - timedelta(days=offset)`.
   - `schedule = Schedule.objects.filter(user=user, date=target_date).select_related("daily_review").prefetch_related("time_blocks").first()`.
   - **If `schedule is None`** → `break` (gap day; user didn't plan that day).
   - **If `schedule` has no `time_blocks`** → `continue` (zero-block "rest day"; doesn't count, doesn't break).
   - Compute the day's `completion_rate`:
     - Prefer `schedule.daily_review.completion_rate` if a row exists (cheap read).
     - Otherwise derive on the fly via `compute_review_stats(schedule)` so the streak reflects actual behaviour, not page-visit habits.
   - If `rate is None` (still zero planned somehow) → `continue` (defensive).
   - If `rate >= settings.ANALYTICS_STREAK_THRESHOLD` → `streak += 1` and `continue`.
   - Else → `break`.
4. Return `streak`.

The on-the-fly recompute path is read-only and scoped to the streak window; no DB writes happen inside `compute_streak`.

### `backend/analytics/views.py`

All views `@login_required`. Date parsing uses `datetime.date.fromisoformat` with the same `{"errors": {"date": "..."}}` envelope as the rest of the API. Body parsing reuses `reject_oversized_body` (100 KB cap).

**Empty-body convention**: an empty request body (`request.body == b""`) parses as `{}`, NOT a JSON parse error. Endpoints with required fields may still reject it during validation (e.g. `update_review_notes` requires `notes` and so 400s on `{}`). The convention exists so a bare `fetch(url, { method: "POST" })` for an idempotent action like "Mark reviewed" doesn't 400 on the parser before the validator can decide whether the body is needed at all.

#### `analytics_view(request, date)` — Inertia render

Path: `GET /analytics/<str:date>/`.

1. Parse `date`. Reject future dates with **400** (analytics is past-/today-only).
2. `schedule = Schedule.objects.filter(user=request.user, date=parsed_date).first()`. Return **404** if missing — analytics is read-only on past data; don't auto-create.
3. **If `schedule.status == REVIEWED`**: load the persisted `DailyReview` directly (frozen snapshot). If a `reviewed` row pre-dates Phase 6 and has no review row, recompute once and persist (back-compat).
4. **Else**: `recompute_review_from_schedule(schedule)` to refresh aggregates. The row's `updated_at` advances on every visit until Mark reviewed.
5. `streak = compute_streak(request.user)`.
6. Inertia render with `{ review: <DailyReview>, streak: <StreakInfo>, schedule: { id, date, status }, blocks: [...] }`. `blocks` is forwarded so the Skipped Tasks list can show titles + times without a second round-trip.

#### `mark_reviewed(request, date)` — API endpoint

Path: `POST /api/analytics/schedules/<str:date>/mark-reviewed/`.

The endpoint must be a **hard idempotent** for `status == REVIEWED` — a retry with stale or malformed body still returns the frozen snapshot, never mutates it. That requires two careful choices below: (a) the body is **not parsed** until after the under-lock status check confirms `ACTIVE`, so a retry with invalid JSON to a reviewed schedule still gets the snapshot, not a 400; (b) the status is **re-checked under the row lock** to close a TOCTOU race where two concurrent calls both see `ACTIVE` pre-transaction.

Algorithm:

1. `reject_oversized_body(request)`. **Do NOT parse the body yet.**
2. Date parse → `{"errors": {"date": "..."}}` 400 on invalid format.
3. `schedule = Schedule.objects.filter(user=request.user, date=parsed_date).first()` — **404** when missing.
4. **Pre-lock status check** (cheap early-out, NOT load-bearing for correctness):
   - `status == DRAFT` → 400. Clients shouldn't surface the button on draft schedules; the check is defence-in-depth.
   - `status == REVIEWED` → return the existing persisted `DailyReview` JSON directly, **without parsing the body, without acquiring a lock, without recompute**. This is the fast idempotent path.
5. `with transaction.atomic():`
   - **Acquire the parent-row lock**: `schedule = Schedule.objects.select_for_update().prefetch_related("time_blocks").get(pk=schedule.pk)`. Without this, `transaction.atomic()` only provides isolation, not blocking — under READ COMMITTED a concurrent `block_detail` PATCH can commit a block change between our recompute-read and our snapshot-write, leaving a stale snapshot frozen as `reviewed`. Locking the parent serialises all writers for this schedule (mirrors the pattern from Phase 5's `ai_generate_draft`); see RULES.md "Locking an empty child queryset locks nothing".
   - **Re-check status under the lock** — closes the TOCTOU race where two concurrent `mark_reviewed` calls both observe `ACTIVE` in step 4 and both enter the transaction. The first commits → REVIEWED; the second wakes from the lock and would otherwise proceed to recompute and overwrite the freshly-frozen snapshot.
     - `status == REVIEWED` → return the persisted `DailyReview` row (re-fetched under the lock to capture the just-committed snapshot). No recompute, body still untouched.
     - `status == DRAFT` → 400. Defensive — would mean a concurrent edit cycled the schedule back to draft, which shouldn't be reachable, but bail out cleanly rather than recomputing on un-validated data.
     - `status == ACTIVE` → continue.
   - **Now parse the body** — only reachable when `status == ACTIVE` under the lock. Empty body → `data = {}` (convention). Otherwise JSON-parse → 400 on invalid. Validate only `notes` (string, ≤ 2000 chars); **silently ignore any other fields**, matching the existing `_parse_rule_create_payload` / `_parse_template_payload` / `block_detail` PATCH style. No "strict mode" exists anywhere in the project; introducing one here would be inconsistent.
   - `recompute_review_from_schedule(schedule)` — capture the final snapshot under the lock.
   - If `notes` was supplied, write it to the row.
   - `schedule.mark_reviewed_if_active()` — flips status under the same transaction.

   Recovery property: if a forward-mutating endpoint (e.g. `block_detail` PATCH) is racing this and gets blocked on the schedule lock, it eventually proceeds after the commit — its `mark_active_on_edit()` then flips `reviewed → active`, and the next analytics visit recomputes. End state is always either "fresh frozen snapshot" or "active + recomputed-on-next-visit", never "frozen + stale".
6. Return updated review JSON.

#### `update_review_notes(request, pk)` — API endpoint

Path: `PATCH /api/analytics/reviews/<int:pk>/notes/`.

1. `reject_oversized_body(request)` (100 KB cap) + JSON parse → 400 on invalid.
2. Per-user PK guard: `DailyReview.objects.filter(pk=pk, schedule__user=request.user).first()` — 404 on cross-user.
3. Accept `{"notes": "<= 2000 chars>"}`. Reject missing/non-string → 400. Notes are the only field editable post-review; everything else is frozen.
4. `update_fields=["notes"]` save. Return the updated row.

#### `frontend/src/components/SkippedTasks.vue`
Props: `blocks: TimeBlock[]`, `date: string`. Filters blocks **mirroring the backend semantics**:

- Compute `today = todayString()` once.
- A block is shown iff `!is_completed && (date < today || end_time < currentHHMM)`.
- For **today**: future-time uncompleted blocks (`end_time >= currentHHMM`) are **not** shown — they're still active, not skipped.
- For **past dates**: all uncompleted blocks are shown.

Without this guard the today-view would show every future block as skipped. Refresh the `currentHHMM` reading on a 1-min interval (matches `Schedule.vue`'s `nowMinutes` cadence) so blocks transition into the list as their windows close.

Renders a flat list with title, time range, and category swatch. Hidden via `v-if` in the parent when the filtered list is empty.

## Test Coverage To Add

### Backend
- `services.compute_review_stats` (parametrised, `now` injected explicitly):
  - All-completed past day → skipped=0.
  - All-uncompleted past day → skipped equals planned.
  - Mixed completion past day.
  - Today, all blocks ended before `now` → all uncompleted are skipped.
  - Today, some blocks still in the future → only ended blocks count toward skipped.
  - Empty schedule (rest day) → all zeros.
  - Category aggregates sum correctly across multiple blocks of the same category.
- `services.compute_streak` (parametrised):
  - Cold start (no schedules) → 0.
  - 5 consecutive ≥ threshold days → 5.
  - Gap day in the middle (no Schedule row) → **breaks** (matches the algorithm).
  - Zero-block schedule on a day → **skip**, doesn't break a 5-day streak.
  - Below-threshold day → break.
  - Window-edge truncation: 60 consecutive ≥ threshold but `WINDOW_DAYS=30` → returns 30.
  - Day with no `daily_review` row but blocks present → counts via on-the-fly recompute (no DB write).
- `services.recompute_review_from_schedule`: idempotent (run twice, same row, `updated_at` advances); preserves `notes` across recomputes.
- `Schedule.mark_active_on_edit` (parametrised over each forward-mutating endpoint):
  - `draft` → flips to `active` (Phase-5 behaviour preserved).
  - `reviewed` → flips to `active` (NEW).
  - `active` → no-op.
- `Schedule.mark_active_on_edit` **stale-instance recovery** (regression test for the conditional-UPDATE requirement):
  - Load a `Schedule` row with `status=ACTIVE` into a Python instance `s`.
  - Bypass the in-memory copy: `Schedule.objects.filter(pk=s.pk).update(status="reviewed")` — simulates a concurrent `mark_reviewed` committing in another transaction.
  - Call `s.mark_active_on_edit()` on the **stale** instance (still has `s.status == "active"` in memory).
  - Assert: returns `True`, `Schedule.objects.get(pk=s.pk).status == "active"`, AND `s.status == "active"` (in-memory copy synced from the UPDATE).
  - Without the conditional-UPDATE pattern, this test fails: the Python-side check sees `self.status == ACTIVE` and short-circuits the flip, leaving the DB row frozen as `reviewed`.
- `Schedule.mark_reviewed_if_active`:
  - `active` → flips to `reviewed`. **In-memory `self.status` is also `reviewed` after the call** (symmetric with `mark_active_on_edit`).
  - `reviewed` → no-op.
  - `draft` → returns `False`, status unchanged.
- `analytics_view`:
  - 404 on unknown date.
  - 400 on future date.
  - Recomputes for `active` schedules (verify `updated_at` advances on second visit).
  - Frozen for `reviewed` schedules (verify `updated_at` does NOT advance on second visit).
  - Back-compat: `reviewed` schedule with no `DailyReview` row → recomputes once, persists, returns.
- `mark_reviewed`:
  - 400 on draft schedules; no row written.
  - Active → reviewed: row persisted, status flipped, both under one transaction.
  - Idempotent on already-reviewed: returns persisted snapshot, **`updated_at` is unchanged** between two calls (the regression test for the "don't recompute frozen" rule).
  - **Idempotency ignores body on reviewed (different notes)**: second call with `{"notes": "different"}` returns the snapshot and **`notes` in DB is unchanged**. Pins the "POST body ignored on reviewed" rule.
  - **Idempotency tolerates malformed body on reviewed**: second call with `request.body = b"{not json"` returns 200 with the persisted snapshot, NOT 400. ## Implementation Notes

### Why `mark_active_on_edit` instead of two separate methods

The transition matrix is small but cohesive — `(draft|reviewed) → active` on edit is a single rule expressed two ways. Splitting into `mark_active_if_draft` + `mark_active_if_reviewed` would mean every forward-mutating endpoint calls both, in order, with no shared logic. One method, one call site, one rule.

### Why `mark_active_on_edit` uses a DB-conditional UPDATE (not a Python check)

The Phase-5 implementation of `mark_active_if_draft` reads `self.status` from the in-memory Python instance:

```python
if self.status == self.Status.DRAFT:
    self.status = self.Status.ACTIVE
    self.save(update_fields=["status"])
```

That worked because nothing else in Phase 5 wrote to `Schedule.status` mid-transaction. Phase 6 changes that: `mark_reviewed` actively writes `status=REVIEWED` while a concurrent edit may be in flight on a Schedule object loaded before the lock contention started. Walk through:

1. T1 (`block_detail` PATCH): loads `block.schedule` into memory — `s.status == ACTIVE`.
2. T2 (`mark_reviewed`): acquires the row lock, recomputes, writes `status=REVIEWED`, commits.
3. T1 calls `block.schedule.mark_active_on_edit()`. The in-memory `s.status` is still `ACTIVE` (loaded in step 1). A Python-side check returns `False`, no flip happens.
4. T1 commits with the block change applied — but `Schedule.status` is still `REVIEWED` in the DB. Snapshot is now stale and frozen.

The recovery property the plan claims requires T1's `mark_active_on_edit` to actually flip the row back. The fix is to issue a single `UPDATE ... WHERE status IN ('draft', 'reviewed')` SQL statement: the WHERE clause is evaluated against the current DB state at lock-acquisition time, not against the stale Python copy. Postgres's row lock for the UPDATE serialises naturally with T2's `select_for_update` — T1 waits until T2 commits, then the UPDATE sees `REVIEWED` and flips it.

This is why the plan specifies `Queryset.update(...)` rather than `self.save(update_fields=...)`. The regression test "stale-instance recovery" pins the contract.

### Why recompute on every visit while `active`

Real-time freshness without a polling loop or WebSocket. Toggle a checkbox on `/schedule/today/`, navigate to `/analytics/today/`, the panel reflects the toggle immediately. Polling/streaming would be over-engineering for what is fundamentally a read-on-page-load surface.

### Why `mark_reviewed` is idempotent (no recompute on already-reviewed)

The whole point of `reviewed` is that the snapshot is **frozen** — it captures the state at the moment of review. Recomputing on a second `mark_reviewed` call (e.g. a double-click, a retry from a flaky network) would silently overwrite the frozen data with the current state of the schedule, defeating the freeze. The idempotent path returns the persisted snapshot verbatim; the regression test asserts `updated_at` is identical between two calls.

The body is ignored entirely on the second call — even a different `notes` value. If the rule were "merge notes from the second body", a retry with a stale notes draft would silently overwrite a more recent edit. Notes after review go through `PATCH /api/analytics/reviews/<pk>/notes/`, which has its own optimistic-overwrite semantics (last-write-wins is fine for plain text).

### Why `mark_reviewed` needs `select_for_update` on the parent + re-check under the lock

Two distinct races, each requiring its own piece of the fix.

**Race A — concurrent edit during review** (`mark_reviewed` vs. `block_detail` PATCH):

`transaction.atomic()` provides isolation under READ COMMITTED (Postgres default), not blocking. Without an explicit row lock:

1. T1 (`mark_reviewed`) opens atomic, reads blocks, builds snapshot in memory.
2. T2 (`block_detail` PATCH) commits a block toggle.
3. T1 writes its snapshot (based on the pre-T2 read), flips status to REVIEWED, commits.
4. End state: snapshot is stale (missing T2's change), status is REVIEWED → analytics_view never recomputes.

Locking the schedule row at the top of T1's atomic block forces T2's `mark_active_on_edit` UPDATE to wait. The end state becomes either "fresh frozen snapshot" or "T2 commits after T1, flips back to active, next visit recomputes" — both correct.

**Race B — two concurrent reviews of the same schedule** (`mark_reviewed` vs. `mark_reviewed`):

The lock alone isn't enough. Both calls observe `status=ACTIVE` in the pre-transaction check, both enter the atomic block, both queue on the row lock:

1. T1 acquires the lock first, recomputes, persists snapshot, flips REVIEWED, commits, releases.
2. T2 acquires the lock — but T2's pre-transaction status read (now stale) said ACTIVE.
3. T2 proceeds to recompute, overwrites the freshly-frozen snapshot. `mark_reviewed_if_active()` is a no-op (already REVIEWED), so T2 returns 200 silently.

The fix is **double-checked locking**: re-read `status` *after* the lock is held, and bail out to the idempotent path when REVIEWED. The pre-transaction check stays as a cheap fast-path; the under-lock check is the load-bearing one for correctness.

Both fixes follow the same pattern as Phase 5's `ai_generate_draft`: lock the parent because locking the (potentially empty) child queryset locks nothing, then re-validate the precondition under the lock. RULES.md captures the pattern.

### Why parse the body after the under-lock status check

The endpoint is hard-idempotent on `REVIEWED`: a retry should always succeed and return the persisted snapshot, regardless of payload. If the body were parsed before the status check, a network retry with a corrupted/half-truncated body would 400 on `JSONDecodeError` even though the previous attempt already succeeded.

Order matters: the only path that reads the body is the `ACTIVE` branch under the lock, where parsing failures are user errors (the request can still mutate). On REVIEWED the body is dead weight — discard it without parsing.

Trade-off: the active path pays a slightly heavier cost (lock acquired before validation could 400), but the active path is also the one that would recompute anyway. The reviewed path — by far the more common retry case — is a fast read with no JSON cost.

### Why streak treats missing schedules as a break

If gaps were "skip", a user could plan one day, take a 30-day vacation, plan another day at >80%, and see a 2-day streak. That's misleading. Calendar-walk semantics align "streak" with what users intuitively mean — consecutive days of disciplined planning AND execution. Rest days (zero-block schedules) are the documented exception.

### Why `compute_streak` reads `daily_review` lazily and recomputes on the fly

If the streak required an explicit "open analytics" visit per day to count, a user who plans well but rarely opens the page would see streak=0 forever. Reading `daily_review` first (cheap), falling back to a transient `compute_review_stats` (no DB write), means the streak reflects actual behaviour, not page-visit habits. Persistence happens elsewhere.

### Why `mark_reviewed_if_active` refuses on draft

A `draft` schedule has never been edited. Its blocks are AI-generated guesses the user didn't validate. Letting the user "review" that data would seed analytics with content that doesn't reflect their day. The `False` return + the view's 400 response is conservative: any client surfacing the button on a draft schedule is buggy.

### Status flow recap (post-Phase 6)

```
draft  ──user edits any block──▶  active  ──Mark reviewed click──▶  reviewed
   ▲                              ▲ │                                    │
   │                              │ └────────────────────────────────────┘
   └─ (no path back from active)  └── any subsequent edit on reviewed ────┘
                                           (mark_active_on_edit)
```

The full matrix is documented in RULES.md.

### Why `(completed: X/Y)` instead of structured stats in the draft prompt

The LLM reads natural-language context well. `May 4 (Mon, completed: 5/7)` is one extra token per history day and gives the model enough signal to learn "user finishes ~70% of weekday plans". A structured "completion stats by category" block adds tokens and complexity for marginal gain — defer until evidence shows drafts are still over-planning.

### `CategoryMinutes = Record<TimeBlock["category"], number>`

Hardcoding `{ work: number; personal: number; health: number; other: number }` would silently drift if a category is ever added or renamed. Keying off the existing union forces a compile error at the right place.
