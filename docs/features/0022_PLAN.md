# Feature 0022 — Google Calendar events via OAuth 2.0 (issue #31)

## Context

Read-only, **multi-account, multi-calendar** Google Calendar event retrieval, the
companion to feature 0011 (Apple Calendar / CalDAV). It feeds the **same**
provider-agnostic frontend `ExternalEventsPanel` so Google + Apple events render in
one merged, time-sorted list. Backend lives in a new `backend/gcal_sync/` Django
app that mirrors the `calendar_sync` / `todoist_sync` skeleton. Auth is OAuth 2.0
(authorization-code flow with offline access + refresh tokens) using
`google-auth(-oauthlib)` for the security-critical token bits and `httpx.AsyncClient`
for the Calendar REST calls. The events view is `async def` and fetches across every
connected account × every selected calendar concurrently via `asyncio.gather`. Deploy
is ASGI/uvicorn (feature 0016), so the async view serves concurrently off one event
loop.

Key divergences from 0011 baked in by the clarification interview:
- `GoogleCalendarAccount` is **multi-row per user** (`ForeignKey`, not `OneToOne`), with a uniqueness constraint so reconnect updates the existing row.
- Per-account partial success: one account's revoked grant must not blank the whole panel — the endpoint returns the events that loaded **plus** an `account_errors[]` array.
- `NormalizedEvent` gains an account-identifying field, shared wire-compatibly with the Apple path (Apple fills a constant sentinel).
- Dedicated env vars, dedicated Fernet key, dedicated system checks (`gcal_sync.E001` / `gcal_sync.W001`).

### Out of scope (explicitly NOT planned)
Two-way sync / event creation; Google Tasks / Contacts; Notion / Microsoft 365 /
Exchange; local on-device calendars; per-calendar selection / toggle UI in V1; popup
OAuth (full-page redirect only); per-calendar grouping/sections in the panel.

- Token accessors (caller must `save()`):
  - `get_refresh_token() -> str` / `set_refresh_token(plaintext)`.
  - `get_access_token() -> str | None` / `set_access_token(plaintext)`.

## Phase 2A — Backend service layer (`backend/gcal_sync/`) [parallelizable after Phase 1]

### 2A.1 `service.py` — typed error hierarchy (mirror `calendar_sync/service.py:36`)
- `GoogleCalError(Exception)` (base).
- `GoogleCalAuthError(GoogleCalError)` — token refresh failed / revoked grant (maps to the per-account `reconnect_required`).
- `GoogleCalTimeoutError(GoogleCalError)`.
- `GoogleCalProviderError(GoogleCalError)` — any other REST/transport failure.
`ImproperlyConfigured` (key rotation) propagates unwrapped so views map it to a config-shaped 500 — same pattern as CalDAV's outer `except (CalDAVError, ImproperlyConfigured): raise`.

### 2A.2 OAuth flow helpers (`service.py`, using `google-auth-oauthlib`)
**Server-side config dict** (built once per call from settings, never client-provided):
`{"web": {"client_id": GOOGLE_OAUTH_CLIENT_ID, "client_secret": GOOGLE_OAUTH_CLIENT_SECRET, "auth_uri": GOOGLE_OAUTH_AUTH_URI, "token_uri": GOOGLE_OAUTH_TOKEN_URI, "redirect_uris": [GOOGLE_OAUTH_REDIRECT_URI]}}`.

- `build_authorization_url(state: str) -> str`:
  1. Construct `google_auth_oauthlib.flow.Flow.from_client_config(config, scopes=settings.GOOGLE_OAUTH_SCOPE.split(), redirect_uri=GOOGLE_OAUTH_REDIRECT_URI)` — **`.split()` the space-separated scope string into a list; never pass `[GOOGLE_OAUTH_SCOPE]` (a single opaque multi-scope string Google rejects).**
  2. `flow.authorization_url(access_type="offline", prompt="consent", state=state)` — `access_type=offline` + `prompt=consent` guarantees a **refresh token is returned even on re-consent**. **Shipped deviation:** `include_granted_scopes` is **omitted** (the plan originally specified `"true"`). Incremental auth would let Google merge a user's *other* grants on this client into the returned token (a superset scope), tripping oauthlib's scope-equality check in `fetch_token`. The feature requests a fixed read-only scope set and doesn't use incremental auth, so it's dropped; `service.py` also sets `OAUTHLIB_RELAX_TOKEN_SCOPE=1` as belt-and-suspenders. See `RULES.md` § "Google Calendar / OAuth".
  3. Return the URL only; the `state` is generated and stored by the view (see 2B).

- `exchange_code(code: str, state: str) -> dict`:
  1. Rebuild the `Flow` (same config, same `scopes=settings.GOOGLE_OAUTH_SCOPE.split()`, same `redirect_uri`, `state=state`).
  2. `flow.fetch_token(code=code)` — performs the code→token POST to `GOOGLE_OAUTH_TOKEN_URI`. **The `code` is never logged.**
  3. From `flow.credentials`: extract `refresh_token`, `token` (access token), `expiry` (UTC datetime).
  4. Identify the account. With `openid` + `userinfo.email` in the scope (see env block), `flow.credentials.id_token` is a signed JWT carrying `sub` + `email`. **Verify it** with
     `google.oauth2.id_token.verify_oauth2_token(flow.credentials.id_token, google.auth.transport.requests.Request(), audience=GOOGLE_OAUTH_CLIENT_ID)` — this performs a network fetch of Google's signing certs and checks the signature, `aud == client_id`, issuer, and expiry; it returns the decoded claims dict. Take `google_account_id = claims["sub"]`, `email = claims["email"]`. This call is **synchronous + network-bound**, so when invoked from the async events path it runs via `asyncio.to_thread`; in the **callback view it is fine to call directly** (the callback is a sync `def` view — see 2B). **Fallback** (only if `id_token` is somehow absent — should not happen with `openid` + `userinfo.email` requested): one `GET {GOOGLE_CALENDAR_API_BASE}/users/me/calendarList` and read the `primary` calendar entry (its `id` equals the account email, used for **both** `email` and `google_account_id`). Return `{"google_account_id", "email", "refresh_token", "access_token", "expiry"}`.
  5. **Grant probe (connect-time verify):** after extracting tokens, issue one `GET {GOOGLE_CALENDAR_API_BASE}/users/me/calendarList` to confirm the grant is live before persisting (mirrors CalDAV's second-endpoint `verify_credentials` probe); failure → `GoogleCalAuthError` / `GoogleCalProviderError`. **Use a synchronous `httpx.Client(timeout=settings.GOOGLE_REQUEST_TIMEOUT)` here** — `exchange_code` is a sync function (called from the sync callback view); do **not** reach for `httpx.AsyncClient` (that's only the async events path). Same for the step-4 `id_token`-absent fallback `calendarList` call. This is the **only** connect-time identity/verify path — there is no separate `verify_and_fetch_account_identity` helper (it would be orphaned; the callback calls `exchange_code` directly, see 2B step 3). The same `calendarList` call doubles as the `id_token`-absent fallback in step 4.
  6. Wrap transport/refused errors as `GoogleCalAuthError` / `GoogleCalProviderError`. **A missing `refresh_token` is a hard error** (`GoogleCalAuthError`) — without it we cannot fetch later; the `prompt=consent` request makes this unreachable in practice but we fail loud, never silently store a half-account.

### 2A.3 Access-token lifecycle (`service.py`) — cache-then-refresh
`async def _ensure_access_token(account) -> str` (the only refresh-token decryption call site):
1. If `account.access_token_expiry` is set and `now < expiry - SKEW` (small skew, e.g. 60s) and a cached access token exists → decrypt and return it (no network call).
2. Else refresh: build a `google.oauth2.credentials.Credentials` from the decrypted `refresh_token` + client id/secret + `GOOGLE_OAUTH_TOKEN_URI`, call `credentials.refresh(google.auth.transport.requests.Request())`.
   - **Refresh failure (revoked / expired / deleted account)** → raise `GoogleCalAuthError`. **Never a silent fallback** (acceptance criterion + security req).
3. On success, persist via a **locked read-modify-write** to avoid a concurrent-rotation lost update (two requests refreshing the same expired account could otherwise have the second full-save its **stale** `refresh_token_encrypted`, clobbering the rotated token the first stored). Do **not** `asave()` the stale in-memory `account` instance. Instead call `await sync_to_async(_persist_refreshed_tokens, thread_sensitive=True)(account.pk, new_access_token, new_expiry, rotated_refresh_or_None)`, where `_persist_refreshed_tokens` is a **sync** function that:
   - opens `with transaction.atomic(): row = GoogleCalendarAccount.objects.select_for_update().get(pk=pk)` — reloads + row-locks the freshest copy;
   - **double-checked refresh:** if `row.access_token_expiry` is already fresh (`now < row.access_token_expiry - SKEW`) **and this caller received no rotated refresh token** (`rotated_refresh_or_None is None`), another concurrent request already refreshed → return that row's decrypted access token and skip the write. **But if this caller DID receive a rotated refresh token, never skip** — persist the rotated `refresh_token` under the lock even when the access token is already fresh: Google may have invalidated the old refresh token when it issued the rotated one, so discarding it would break future refreshes (the exact bug P1 flagged). When persisting a rotated token this way, also write this caller's `access_token` + `expiry` (both are valid);
   - else `row.set_access_token(new_access_token)`, `row.access_token_expiry = new_expiry`, and set the refresh token **only when Google returned a rotated one** (`rotated_refresh_or_None is not None` → `row.set_refresh_token(rotated)`; never overwrite the stored refresh token with a stale value when no rotation was returned);
   - `row.save()` with **no `update_fields`** (full save) so `auto_now` advances `updated_at` and the events cache version rotates (respect the footgun).
   The `select_for_update` row lock serialises the **persist** for concurrent refreshers of the same account; the stale outer `account` instance is never the save source. (`_ensure_access_token` is `async def`, so a plain sync `.save()` called directly in the coroutine would also raise `SynchronousOnlyOperation` — the `sync_to_async` wrapper resolves both the threading and the lost-update concerns at once.)
   **Single-flight is NOT guaranteed (deliberate, per P2):** the lock is acquired *after* the provider refresh (step 2), **not** held across it — pinning a `select_for_update` lock across an external Google HTTP refresh (network latency) is an anti-pattern. So two callers that both find the token expired can both refresh against Google before either locks; the lock only serialises the persist. This is correct: the last writer holding a rotated refresh token wins and no stale token survives (per the double-check above). The concurrency guarantee under test is **persisted-token correctness, not refresh-call count.**
4. Return the fresh access token. Drop the plaintext local binding in a `finally:` (`del`), mirroring CalDAV/Todoist.

(Refresh uses the `google-auth` synchronous `Request` transport; it is the security-critical path and is invoked inside the async coroutine via `asyncio.to_thread(_refresh_sync, …)` so the blocking refresh HTTP call doesn't stall the event loop — then `httpx.AsyncClient` handles the REST calls. `_refresh_sync` performs **only** the network refresh and returns the new token/expiry/rotated-refresh tuple — it must **not** do the ORM write (that happens via the locked `_persist_refreshed_tokens` in step 3); keeping the thread function pure-network avoids a sync ORM `.save()` smuggled into a worker thread.)

**Cache API note (async safety — corrected):** the events cache is read/written from the **async** events view, so it must use the **async** cache API — `await cache.aget` / `await cache.aset` (the `cache.py` helpers are `async def`; see 2A.5). Direct **synchronous** `cache.get` / `cache.set` would be **wrong here**: under `RedisCache` they are blocking network I/O and would stall the ASGI event loop. The `aincr` footgun from MEMORY (`await cache.aincr()` inherits a non-atomic read-modify-write that resets TTL) is **specific to the rate-limiter's counter (RMW)** and does **not** apply to these plain idempotent versioned-key get/set — there is no counter and no RMW here, so `aget`/`aset` are both correct and non-blocking. (Only the rate limiter must stay on sync atomic `incr`; this feature has no rate-limit counter.)

### 2A.4 REST fetch (`service.py`, `httpx.AsyncClient`)
- `async def _fetch_selected_calendars(client, access_token) -> list[dict]`:
  `GET {GOOGLE_CALENDAR_API_BASE}/users/me/calendarList` (paginate on `nextPageToken`), filter `item.get("selected") is True`, return `[{"id", "summary"}]`. Bearer auth header `Authorization: Bearer <token>` — **never logged**.
- `async def _fetch_events_for_calendar(client, access_token, calendar_id, time_min, time_max) -> list[dict]`:
  `GET {base}/calendars/{urllib.parse.quote(calendar_id, safe="")}/events` — **URL-encode `calendar_id` with `urllib.parse.quote(calendar_id, safe="")`** (or an equivalent `httpx.URL` path-safe construction). Google calendar IDs routinely contain `@` (e.g. a shared `…@group.calendar.google.com`), `#`, and `/`; left unescaped they break into extra path segments / a fragment and silently fail shared/group-calendar fetches. Query: `singleEvents=true` (expands recurrence into instances → matches the Apple path's per-occurrence rows), `orderBy=startTime`, `timeMin`/`timeMax` = the date window in RFC3339 UTC, `maxResults` page size, paginate on `nextPageToken` with a `_MAX_PAGES` ceiling (worker protection, raise on overflow — mirror Todoist `_MAX_FILTER_PAGES`).
- `_day_window(target_date)` — `[start, end)` in `settings.TIME_ZONE` then to UTC RFC3339 (reuse the CalDAV `_day_window` logic; Google wants RFC3339 strings, so format with `.isoformat()`).
- `_normalize_gcal_event(raw, calendar_name, account_email) -> NormalizedEvent | None`:
  - **Cancelled-event skip (first check):** if `raw.get("status") == "cancelled"` → return `None`. With `singleEvents=true`, Google emits cancelled instances of recurring events (frequently start/end-less); they must not become ghost rows in the merged list.
  - Timed event: `start.dateTime` / `end.dateTime` carry explicit IANA `start.timeZone`; parse → convert to UTC (Google returns offset-aware RFC3339, so `datetime.fromisoformat` then `.astimezone(UTC)`).
  - All-day event: `start.date` / `end.date` (date-only, exclusive end) → `all_day=True`, synthesize UTC midnight in `settings.TIME_ZONE` (reuse CalDAV `_to_utc` date branch). Google's all-day `end.date` is **exclusive**; keep as-is (matches CalDAV's +1 day all-day convention).
  - `title` = `summary` or `"(no title)"`; `external_uid` = `f"{raw['id']}@google"` (namespaced so it can't collide with a CalDAV UID in the merged list); `calendar_name` = the calendar `summary`; `account_label` = `account_email`.
  - Malformed single event → log a `warning` and return `None` (skip), not crash the whole fetch — narrow the `except` to `(KeyError, ValueError, TypeError)` like CalDAV/Todoist. **Never log token-bearing fields.**

- `async def fetch_events_for_account(account, target_date) -> list[NormalizedEvent]`:
  1. `access_token = await _ensure_access_token(account)`.
  2. `async with httpx.AsyncClient(timeout=settings.GOOGLE_REQUEST_TIMEOUT) as client:` enumerate selected calendars, then `asyncio.gather(*[_fetch_events_for_calendar(...) for cal in selected])` — **per-calendar concurrency within an account**.
  3. Normalise + filter to the window + sort `(start, title, external_uid)` (stable order, same as CalDAV).
  4. Map `httpx.TimeoutException` → `GoogleCalTimeoutError`, `httpx.HTTPStatusError` 401/403 → `GoogleCalAuthError`, other → `GoogleCalProviderError`. `ImproperlyConfigured` propagates.

- **Connect-time path is `exchange_code` only** — there is **no** standalone `verify_and_fetch_account_identity` helper (an earlier draft listed one; it would be orphaned/dead since the callback calls `exchange_code` directly — see 2B step 3). The grant-verification `calendarList` probe and id-token identity both live inside `exchange_code` (2A.2 steps 4–5).

### 2A.5 `cache.py` (clone `calendar_sync/cache.py`)
- `_KEY_PREFIX = "gcal_events"`; key = `f"{_KEY_PREFIX}:{account.user_id}:{account.id}:{account.updated_at.isoformat()}:{target_date.isoformat()}"` — **include `account.id`** because a user has many accounts (the CalDAV key only had `user_id`). Versioned by `updated_at` (token refresh / reconnect rotates it).
- **`async def get_cached_events(account, target_date)`** (`return await cache.aget(key)`) / **`async def set_cached_events(account, target_date, events)`** (`await cache.aset(key, payload, settings.GOOGLE_CACHE_TTL_SECONDS)`) — **async helpers using `aget`/`aset`** (not sync `get`/`set`) because the only caller is the async events view and sync Redis calls would block the event loop (see the 2A.3 cache-API note). No explicit invalidation (version segment handles it).

### 2A.6 `checks.py` (clone `calendar_sync/checks.py`)
- `gcal_sync.W001` — `Warning` when `CACHES["default"]["BACKEND"]` is an ineffective backend (LocMem/FileBased/Dummy) **and** `GoogleCalendarAccount.objects.exists()` **and** not DEBUG. Wrap the DB probe in `except (OperationalError, ProgrammingError)` for pre-migrate safety (same as CalDAV).
- `gcal_sync.E001` — `Error` (blocks startup) when not DEBUG and **either** `GOOGLE_OAUTH_TOKEN_KEY` is unset/malformed (validate with `Fernet(key)`) **or** any of `GOOGLE_OAUTH_CLIENT_ID` / `GOOGLE_OAUTH_CLIENT_SECRET` / `GOOGLE_OAUTH_REDIRECT_URI` is unset. Settings-only, no DB query (safe pre-migrate). Emit a distinct, actionable message per missing var.
  **Prod-ops note (intentional hard requirement, per clarification #1):** unlike CalDAV (whose E001 covers only the encryption key), this fires whenever `DEBUG=False` **even if no user has connected Google** — the four `GOOGLE_OAUTH_*` vars become a hard boot dependency for every production deploy. ## Phase 2B — Backend views & routes (`backend/gcal_sync/views.py`) [parallelizable after Phase 1; coordinates the `NormalizedEvent` contract with 2A]

Envelope + error-map helpers mirror `calendar_sync/views.py`:
`_envelope(detail, status, **extra)` → `{"errors": {"detail": …, **extra}}`;
`_SERVICE_ERROR_STATUS = {GoogleCalAuthError: (401, …), GoogleCalTimeoutError: (504, …), GoogleCalProviderError: (502, …)}`; unknown → logged 500.

### OAuth connect — `GET /api/calendar/google/connect/`  (`@login_required`, `@require_http_methods(["GET"])`)
1. Generate a CSRF `state` (`secrets.token_urlsafe(32)`), store under a session key (e.g. `request.session["gcal_oauth_state"] = state`).
2. `url = service.build_authorization_url(state)`.
3. `return HttpResponseRedirect(url)` (302 full-page redirect to Google consent).

### OAuth callback — `GET /api/calendar/google/callback/?code=&state=`  (`@login_required`, `@require_http_methods(["GET"])`)
1. **State validation:** read `state` from query + `request.session.pop("gcal_oauth_state", None)`. If query `state` is missing, session value is missing, or they mismatch → **reject**: `302 → /settings/?google=error&reason=state` (CSRF guard, security req). Pop so a `state` can't be replayed.
2. If Google returned `?error=...` (e.g. user denied) → `302 → /settings/?google=error&reason=denied`. If **neither `code` nor `error`** is present (malformed/empty callback) → `302 → /settings/?google=error&reason=missing_code`.
3. `info = service.exchange_code(code, state)` (sync call; this view can stay sync — the heavy concurrency is only the events fetch). On `GoogleCalError` → `302 → /settings/?google=error&reason=provider`.
4. **Upsert** under `transaction.atomic()` + `select_for_update`:
   `GoogleCalendarAccount.objects.get_or_create(user=request.user, google_account_id=info["google_account_id"], defaults={"email": …})` then set `email`, `set_refresh_token`, `set_access_token`, `access_token_expiry`, `last_verified_at = timezone.now()`, and **plain `acc.save()`** (auto_now → cache version). Uniqueness constraint `(user, google_account_id)` makes reconnect idempotent.
5. `return HttpResponseRedirect("/settings/?google=connected")`.

**Known V1 limitation (session expiry mid-consent):** `connect` / `callback` are `@login_required`. If the Django session expires while the user is on Google's consent screen, the callback hits the login redirect and the one-time `code` is lost. V1 does **not** persist the in-flight OAuth state across re-login — the popped/absent session `state` makes the callback surface `?google=error&reason=state`, and the user retries Connect after signing in. Surface a Settings message ("Session expired — sign in and connect again"); documented, not auto-recovered.

### Account list / disconnect — `/api/calendar/google/accounts/`  + `/api/calendar/google/accounts/<int:account_id>/`
- `GET /api/calendar/google/accounts/` (`@login_required`) → `{"accounts": [{"id", "email", "last_verified_at"}, …]}` (no token fields, ever) for the Settings list.
- `DELETE /api/calendar/google/accounts/<int:account_id>/` (`@login_required`, `@require_http_methods(["DELETE"])`) → `GoogleCalendarAccount.objects.filter(user=request.user, id=account_id).delete()` — **scoped to `request.user`** (IDOR guard); idempotent; versioned cache keys make prior entries unreachable. Returns the **refreshed account list in the same shape as `GET /accounts/`** — `{"accounts": [{"id", "email", "last_verified_at"}, …]}` (no token fields). (Optionally `POST /accounts/<id>/refresh-token-probe/` is **out of scope**; reconnect = re-run connect.)

### Events (the multi-account async fetch) — `GET /api/calendar/google/events/<str:date>/`
**`async def`** (the deploy is ASGI/uvicorn). `@login_required` + `@require_http_methods(["GET"])` (use the async-compatible decorators / `sync_to_async`-friendly wrappers already proven by the 0009 async AI views — verify decorator order works under async).
1. Parse the `date` route string into **`target_date = datetime.date.fromisoformat(date)`** (module-qualified `datetime.date` to avoid shadowing the `date` route param); 400 on `ValueError`. **Use `target_date` (a `datetime.date`) — never the raw `date` string — in every cache + service call below;** the cache key calls `target_date.isoformat()` (§2A.5) and the service calls `_day_window(target_date)`, so passing the string 500s on every request.
2. **Resolve the user with `user = await request.auser()` FIRST** — touching the sync `request.user` `SimpleLazyObject` proxy inside an `async def` body raises `SynchronousOnlyOperation` (this is exactly why every 0009 async AI view does `user = await request.auser()` before any ORM access; `auser()` is cached on `request._acached_user` by `@login_required`, so this is an `hasattr` short-circuit, not a second DB hit). **Every `request.user` reference in this async handler must be the `auser()` result instead** — including the queryset filter and any `account.email`/`account.id` logging context.
3. Load the user's accounts: `accounts = [a async for a in GoogleCalendarAccount.objects.filter(user=user)]` (the `user` from step 2, **never `request.user`**). **Empty → `503 "No Google Calendar account configured"`** (mirrors CalDAV's `DoesNotExist → 503`; the panel treats 503 as "not connected").
4. **Per-account fetch with cache + partial success:**
   - For each account: check `await cache.get_cached_events(account, target_date)`; on hit use it, else schedule `service.fetch_events_for_account(account, target_date)`.
   - `results = await asyncio.gather(*coros, return_exceptions=True)` — **`return_exceptions=True` is what enables partial success**: one account's `GoogleCalAuthError` doesn't cancel the others.
   - For each result:
     - Success → `await cache.set_cached_events(account, target_date, payload)`, extend the merged `events` list.
     - `GoogleCalAuthError` → append `{"account_id": account.id, "email": account.email, "error": "reconnect_required"}` to `account_errors`.
     - `GoogleCalTimeoutError` / `GoogleCalProviderError` → append `{"account_id", "email", "error": "unavailable"}`.
     - `ImproperlyConfigured` → this is a **server-wide** key-rotation problem, not per-account; short-circuit the whole response to `500 "Google Calendar service is misconfigured. Contact the administrator."` (matches CalDAV/Todoist config-500).
5. **Composite response shape** (extends the existing `{"events":[…]}`):
   `{"events": [ …merged, time-sorted dicts… ], "account_errors": [ {account_id, email, error}, … ]}`.
   `account_errors` is **always present** (empty list when all accounts loaded) so the frontend has a stable shape. Final sort of the merged list by `(start, title, external_uid)` so Apple + Google interleave chronologically (the panel does no sorting).

### Routes — `backend/day_forge/urls.py`
Add `from gcal_sync import views as gcal_views` and a `# API: Google Calendar (feature 0022)` block under `/api/calendar/google/`:

## Cross-cutting invariants (apply throughout)
- **Service-boundary owns secrets:** `service.py` is the only module that calls `get_refresh_token()` / `get_access_token()` / decrypts anything; views pass the `GoogleCalendarAccount` through and never touch plaintext (mirrors CalDAV/Todoist). Makes the never-logged test tractable.
- **Never log** refresh tokens, access tokens, the OAuth `code`, or `Authorization` headers — anywhere (service, views, error wrapping).
- **`redirect_uri` is server-config only**, read from settings; the callback never trusts a client-supplied redirect.
- **auto_now cache-version footgun:** every mutating `save()` on `GoogleCalendarAccount` (token refresh, reconnect, disconnect-is-delete) uses plain `save()` (no `update_fields`) so `updated_at` advances and the versioned cache key rotates.
- **Partial success, never silent fallback:** a per-account failure becomes an `account_errors[]` entry with a reconnect CTA; a whole-request transport failure becomes the mapped HTTP status; a key-rotation failure becomes a config-500. No path silently returns an empty success.
