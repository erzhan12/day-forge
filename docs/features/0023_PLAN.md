# Feature 0023 — Timeline: extend trailing stub to current time when idle after last block

## Context

Feature 0017 compresses the trailing edge gap (last block end → 23:00) to a
fixed `STUB_MINUTES` (30) stub anchored at `lastBlockEnd`. When the current-time
red line is **after** the last block, that compresses the idle interval between
the last block and now, so the user cannot see how long they've been idle since
the schedule ended.

This feature makes the trailing anchor **now-aware on today only**. The idle gap
`lastBlockEnd → now` renders at full scale; only the tail after `now +
STUB_MINUTES` compresses toward 23:00. Past/future days keep exact 0017 behavior.
Frontend only — no backend, API, schema, or `useDrag` px-math changes.

Issue #77. Verbatim desired behavior:

```
trailingAnchor = max(lastBlockEnd, nowMinutes)   // but see inside-block rule
renderEnd = min(DAY_END, trailingAnchor + STUB_MINUTES)
```

- **Now after last block:** idle gap `lastBlockEnd → now` full scale; tail after
  `now + 30` compressed toward 23:00.
- **Now inside last block:** anchor stays at `lastBlockEnd` (not now).
- **Now after 23:00:** clamp to `DAY_END`.
- **Empty day (today, zero blocks):** keep `06:00 → now` full scale; compress
  only after `now + STUB_MINUTES` toward 23:00.
- **Off-today (yesterday/tomorrow):** `renderEnd = lastBlockEnd + STUB_MINUTES`
  only — no now influence (exact 0017).

Unchanged: leading gap logic (shrink before first block), `STUB_MINUTES = 30`,
mid-day gaps between blocks.

## Algorithm

Define a **named** shared helper so `computeRenderBounds` and
`buildBaseDisplayItems` produce **identical** trailing anchors (DRY — the two
must never diverge or flow layout desyncs from the stub height):

```
computeTrailingAnchor(anchorFloor: number, nowMinutes: number | null): number
```

- `anchorFloor` = `lastEnd` when there is ≥1 visible block, or
  `DAY_START_MINUTES` for the empty-day case.
- Off-today / no-now (`nowMinutes === null`): returns `anchorFloor` — exact 0017.
- Today (`nowMinutes` non-null): returns `min(DAY_END_MINUTES, max(anchorFloor,
  nowMinutes))`. The **outer `min(DAY_END_MINUTES, …)` clamp is mandatory** — for
  `now ≥ 23:00` an unclamped `max(anchorFloor, now)` would place the split
  boundary past `DAY_END`, overflowing the idle gap past 23:00 and producing a
  negative-width tail. Both `computeRenderBounds` and `buildBaseDisplayItems`
  call this helper; export it for direct unit tests.

### Trailing anchor (today, with ≥1 visible block)

**`computeTrailingAnchor` is the single source of truth.** Steps 1–3 below are a
readability expansion of the branches it collapses — implement the helper, not
these cases; they must never be hand-inlined separately or the two call sites
risk diverging.

Given `lastEnd = min(timeToMinutes(lastVisible.end_time), DAY_END_MINUTES)`,
`nowMinutes` (int, non-null only on today), and the today flag:

1. If off-today (`nowDate === null` OR now not applicable): `trailingAnchor =
   lastEnd` — 0017 behavior.
2. If today AND `nowMinutes >= lastEnd` (now at/after last block end — the
   half-open `[start,end)` "inside last block" excludes `now === lastEnd`, so at
   exactly `lastEnd` we treat now as *after*): `trailingAnchor = nowMinutes`.
3. If today AND `nowMinutes < lastEnd` (now before/inside last block):
   `trailingAnchor = lastEnd` (0017 — the inside-block rule keeps the anchor at
   the block end, not now).
   - Net of 1–3: `trailingAnchor = computeTrailingAnchor(lastEnd, nowMinutes)`
     (i.e. `min(DAY_END, max(lastEnd, now))` on today, `lastEnd` off-today).
4. `renderEnd = trailingGap > STUB_MINUTES ? min(DAY_END_MINUTES, trailingAnchor
   + STUB_MINUTES) : DAY_END_MINUTES`, where `trailingGap = DAY_END_MINUTES −
   lastEnd` (compression threshold stays keyed off the **real** trailing gap
   from `lastEnd`, matching 0017 — an already-natural trailing gap never
   compresses regardless of now).
   - Because `trailingAnchor` is already clamped, `min(DAY_END_MINUTES, …)`
     covers "now after 23:00" → `renderEnd = DAY_END`.

### Empty day (today, zero visible blocks)

Current `computeRenderBounds` returns full `{DAY_START_MINUTES, DAY_END_MINUTES}`
and `buildBaseDisplayItems` emits one uncompressed 06:00–23:00 gap.

- Off-today empty: unchanged (full-day gap).
- Today empty: `trailingAnchor = computeTrailingAnchor(DAY_START_MINUTES,
  nowMinutes)` (the same helper — `max(DAY_START, now)` floor guards a pre-06:00
  "now" from pulling the anchor below `DAY_START`, and the `min(DAY_END, …)` clamp
  guards now ≥ 23:00). `renderEnd = min(DAY_END_MINUTES, trailingAnchor +
  STUB_MINUTES)`, `renderStart = DAY_START_MINUTES` (keep full morning `06:00 →
  now`). `buildBaseDisplayItems` splits the single full-day gap the same way,
  gated identically (see the empty-day sub-case below).

### `buildBaseDisplayItems` trailing-gap split (today, idle after last block)

Let `trailingAnchor = computeTrailingAnchor(lastEnd, nowMinutes)` (clamped to
`DAY_END`). Today's key visual change: the trailing gap is split into **two**
display items **only when a real compressed tail exists** — the split gate is:

```
split when  trailingAnchor > lastEnd  AND  activeRenderEnd < DAY_END_MINUTES
```

This single gate fixes both edge cases:

- **now ≥ 23:00** → `renderEnd` clamps to `DAY_END`, so `activeRenderEnd ===
  DAY_END_MINUTES` ⇒ gate false ⇒ single gap (no past-23:00 overflow, no negative
  tail).
- **now < 23:00 but trailing gap already ≤ STUB / anchor+STUB reaches DAY_END**
  (e.g. lastEnd 22:00, now 22:50 → `renderEnd = min(1380, 1370+30) = DAY_END`) ⇒
  gate false ⇒ single full-scale gap `[lastEnd, DAY_END)`; no redundant
  zero-benefit second slot. `activeRenderEnd < DAY_END_MINUTES` implies
  `trailingAnchor < DAY_END`, so the idle segment can never cross 23:00.

When the gate is **true**, emit:

1. **Idle gap** `[lastEnd, trailingAnchor)` — `type: "gap"`, full-scale
   (`render_minutes` omitted / `= duration_minutes`, `compact` false). Because
   `trailingAnchor === now` and the range is half-open, this segment **excludes**
   `now` and stays a plain `gap`; `spliceNowMarker` instead upgrades the tail
   (item 2) to `gap-with-now` with the marker at `0%` — the seam pixel at the
   bottom of this full-scale idle gap, i.e. the proportionally correct "now"
   position (issue's pain point solved by the idle gap being full-scale).
2. **Compressed tail** `[trailingAnchor, DAY_END)` — `type: "gap"`,
   `render_minutes = max(0, activeRenderEnd − trailingAnchor)`, `compact: true`.

When the gate is **false** (off-today; today with now before/inside last block;
or the clamp/threshold cases above): emit the **single** trailing gap exactly as
0017 does today (compressed iff `activeRenderEnd < DAY_END_MINUTES`).

Note `activeRenderEnd` is the frozen/live `renderEnd` already computed with the
now-aware anchor, so when the gate is true `activeRenderEnd − trailingAnchor ===
STUB_MINUTES` — the tail stub height is unchanged (30 → 60px), it's just pushed
down past the idle gap.

**Empty-day sub-case:** when zero visible blocks and today, apply the same helper
and gate with `anchorFloor = DAY_START_MINUTES`: `trailingAnchor =
computeTrailingAnchor(DAY_START_MINUTES, nowMinutes)`. Split into idle
`[DAY_START, trailingAnchor)` full-scale (contains the now marker) + compressed
tail `[trailingAnchor, DAY_END)` **only when** `trailingAnchor > DAY_START_MINUTES
AND activeRenderEnd < DAY_END_MINUTES` (so pre-06:00 now and now ≥ 23:00 both
fall to the single-gap path). Off-today or no-now: single full-day gap (current
behavior).

**Accepted GapSlot label edge — empty-day, pre-06:00 now.** When today is empty
and `now < DAY_START` (before 06:00), the gate is false (`trailingAnchor ===
DAY_START_MINUTES`) but `renderEnd = min(DAY_END, DAY_START + STUB) = 390 <
DAY_END`, so the single full-day gap `[06:00, 23:00)` is emitted with `compact:
true`. `GapSlot.vue`'s `edgeHint` heuristic keys the hint purely off
`startTime === DAY_START` → it labels this stub **"earlier"** (`GapSlot.vue:24–27`),
even though it is really the whole remaining day, not a leading stub — i.e. a
`Free 17h earlier` ~60px slot before 06:00. This is **accepted as-is**: it only
occurs on an empty schedule opened before dawn (rare), the click-to-add range is
still correct (`[06:00, 23:00)`), and changing it would require either a new
`ScheduleDisplayItem` shape or a `GapSlot` prop to distinguish "leading stub" from
"whole-day stub" — out of scope for issue #77. Do **not** silently ship it as a
bug; it is a documented cosmetic edge. (Non-empty days never hit this: the idle
segment is full-scale/non-compact and the trailing stub never starts at
`DAY_START`.)

## Threading now into the pure helpers (parameter additions)

Both pure functions gain the now signal. Keep them pure (no `new Date()` inside)
— today-detection stays in `useNowMinutes` and is passed as the existing
`nowDate`/`nowMinutes` pair.

- `computeRenderBounds(blocks, nowMinutes: number | null = null)` — append an
  optional param defaulting to `null` so existing callers/tests that omit it get
  exact 0017 behavior. `null` ⇒ off-today ⇒ trailing anchor `= lastEnd`.
  Non-null ⇒ today ⇒ apply the now-aware anchor. `nowDate` is not needed inside
  the pure fn: the caller passes `nowMinutes` only when on-today (i.e. passes
  `nowDate === null ? null : nowMinutes`), so a non-null `nowMinutes`
  unambiguously means "today". Document this contract in the JSDoc.
- `buildBaseDisplayItems(blocks, activeRenderStart, activeRenderEnd, nowMinutes:
  number | null = null)` — append the same optional param. Do **not**
  reverse-derive the split boundary from `activeRenderEnd`; pass `nowMinutes`
  explicitly and compute `trailingAnchor = computeTrailingAnchor(anchorFloor,
  nowMinutes)` (the same clamped helper `computeRenderBounds` uses) so the split
  boundary is exact. `activeRenderEnd` still drives the compressed-tail
  `render_minutes` and the split gate (`activeRenderEnd < DAY_END_MINUTES`).

## Wiring in `Schedule.vue`

### Declaration-order constraint (must-fix)

`nowMinutes`/`nowDate` are declared at line 347, **after** the `useDrag` call
(line 105) and the `renderBounds` computed (line 111). To feed now into both:

- Move the `const { nowMinutes, nowDate } = useNowMinutes(toRef(props, "date"))`
  declaration **above** the `useDrag(...)` call and the `renderBounds`
  computed. `useNowMinutes` only needs `toRef(props, "date")` (already available
  from `props`), so it can move up with no dependency issues. Update the comment
  block currently attached at lines 349–353 (sound notifications) to stay with
  `useSoundNotifications` — that call can remain at its current location; only
  the `nowMinutes/nowDate` destructure moves up.

### Getter, computed, displayList

- Render-bounds getter (line 108) becomes:
  `() => computeRenderBounds(props.blocks, nowDate.value === null ? null :
  nowMinutes.value)`. This makes the **drag-start snapshot** (`startDrag` →
  `frozenRenderBounds`) capture the now-aware anchor, satisfying the 0017
  frozen-bounds contract (bounds frozen at drag start, no mid-gesture jump).
- `renderBounds` computed (line 111) becomes:
  `computed(() => computeRenderBounds(props.blocks, nowDate.value === null ?
  null : nowMinutes.value))`. It now reactively depends on `nowMinutes`, so it
  recomputes each 60s tick on today — the stub re-anchors as time passes (idle
  gap grows). This is fine outside drag; during drag its value is **ignored** for
  layout (see displayList).
- `displayList` computed (lines 390–408): **read-order matters.** The current
  code reads `const bounds = renderBounds.value` **unconditionally at the top**
  (`Schedule.vue:391`), before the `isDragging` branch — so as written,
  `displayList` would depend on the now-reactive `renderBounds` even during drag.
  Move the `renderBounds.value` read **into the non-drag branch** of each
  `activeRenderStart` / `activeRenderEnd` ternary (i.e. `isDragging.value &&
  frozenRenderBounds.value ? frozenRenderBounds.value.renderStart :
  renderBounds.value.renderStart`), so the frozen path never touches
  `renderBounds`. Then pass now into `buildBaseDisplayItems(effectiveBlocks.value,
  activeRenderStart, activeRenderEnd, <nowArg>)`, where `<nowArg>` is the
  **frozen-aware** expression defined authoritatively in "Freeze now during drag"
  below (`isDragging.value && frozenRenderBounds.value ? frozenNowMinutes.value :
  (nowDate.value === null ? null : nowMinutes.value)`). Do **not** pass the plain
  live-now ternary here — see the drag-freeze rationale next.
  - **Recompute note (accurate framing):** even after moving the `renderBounds`
    read, `displayList` still recomputes each 60s tick **during** drag — because
    `spliceNowMarker(baseItems, nowMinutes.value, nowDate.value)` (line 407) reads
    live `nowMinutes` to move the red now-line, which is intended. The invariant
    that matters is not "displayList never recomputes during drag" but that every
    **geometry** input (`activeRenderStart` / `activeRenderEnd` and the
    `buildBaseDisplayItems` `<nowArg>`) is frozen during drag, so no layout jump —
    only the now-marker position updates. Moving the read simply drops the
    spurious `renderBounds`→`nowMinutes` layout dependency during drag.
  - **Drag-freeze integrity:** while dragging, `activeRenderStart` /
    `activeRenderEnd` already read `frozenRenderBounds` (frozen at drag start
    with the now-aware anchor). If the `nowMinutes` arg to
    `buildBaseDisplayItems` were **live**, a 60s tick mid-drag would move the
    idle/tail split boundary while bounds stay frozen → flow-vs-ghost divergence.
    Fix: freeze the now signal for the split too, via the `frozenNowMinutes`
    snapshot in `useDrag` (see below); `displayList` reads the frozen now during
    drag and the live now otherwise. Drag is rare and short; a stale-by-≤60s
    idle-gap split during a gesture is invisible, so the frozen-now snapshot
    keeps flow/ghost aligned.

### Freeze now during drag (`useDrag.ts`)

- Add a sibling reactive `frozenNowMinutes = ref<number | null>(null)` set in
  `startDrag` from a new getter param (or fold into the existing render-bounds
  getter by returning `{ renderStart, renderEnd, nowMinutes }`). Prefer
  **extending the return shape** of the render-bounds getter to
  `{ renderStart, renderEnd }` unchanged **plus** exporting the frozen now — but
  the getter already returns `RenderBounds`. Minimal-surface option: add a
  second optional getter param `getNow?: () => number | null`; `startDrag` stores
  `frozenNowMinutes.value = getNow?.() ?? null`; `endDrag` / `cancelDrag` /
  `resetState` reset it to `null` (co-located with the existing
  `frozenRenderBounds` reset at line 507). Export `frozenNowMinutes`.
- `Schedule.vue` destructures `frozenNowMinutes` from `useDrag`, passes
  `getNow = () => (nowDate.value === null ? null : nowMinutes.value)`, and in
  `displayList` the now arg to `buildBaseDisplayItems` becomes:
  `isDragging.value && frozenRenderBounds.value ? frozenNowMinutes.value :
  (nowDate.value === null ? null : nowMinutes.value)`.
- This keeps `useDrag` px-math untouched (leading origin still from
  `frozenRenderBounds.renderStart`); the trailing anchor never enters px-math.

**Accepted desync — dragging the last block past frozen-now.** `frozenNowMinutes`
and `activeRenderEnd` are frozen at drag start, but `lastEnd` inside
`buildBaseDisplayItems` comes from the **live** `effectiveBlocks` preview (that is
the point of the drag ghost). If the user drags the last block later than the
frozen now, `trailingAnchor = computeTrailingAnchor(livePreviewLastEnd,
frozenNow)` grows toward `livePreviewLastEnd` while `activeRenderEnd` stays
frozen, so the compressed-tail `render_minutes = max(0, activeRenderEnd −
trailingAnchor)` shrinks (can reach 0) and the idle segment tracks the ghost.
This is **accepted, cosmetic-only** desync of the trailing extent during an
active gesture — the same class of frozen-bounds-vs-live-preview divergence 0017
already accepts for the trailing stub (see 0017 SPEC drag notes). It resolves the
instant the drag ends and bounds recompute. We deliberately do **not** re-freeze
`lastEnd`; freezing the preview would defeat the drag ghost.
