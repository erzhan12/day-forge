# Architecture Decisions

These are non-obvious conventions that can't be inferred from code alone.

**Package manager**: `uv` (not pip/poetry). Run everything via `uv run`.

**Python version**: 3.14 (pinned in `.python-version`).

## Environment Variables
- `DJANGO_SECRET_KEY` ‚Äî Django secret key (required in production, has insecure fallback for dev)
- `DEBUG` ‚Äî `"1"` enables debug mode (default), `"0"` for production
- `ALLOWED_HOSTS` ‚Äî Comma-separated hostnames (only used when `DEBUG=0`)
- `CSRF_TRUSTED_ORIGINS` ‚Äî Comma-separated origins incl. scheme, e.g. `https://app.example.com` (only used when `DEBUG=0`)
- `REDIS_URL` ‚Äî Connection string for `CACHES['default']` (Django's built-in `django.core.cache.backends.redis.RedisCache`), e.g. `redis://localhost:6379/0`. **Required when `LLM_API_KEY` is set** ‚Äî `ai.E001` blocks an ineffective cache backend (LocMem / FileBased / Dummy) once AI is enabled, independent of `DEBUG`. Recommended for any production-shaped deploy (atomic cross-worker rate-limit counters via Redis `INCR` + CalDAV event-cache perf). When unset, `CACHES['default']` falls back to `LocMemCache` ‚Äî fine only when AI is disabled. A `KEY_PREFIX` of `dayforge` namespaces all keys (`ai_cmd_rl:*`, `caldav_events:*`) so a co-tenanted Redis won't collide.
- `LLM_API_KEY` ‚Äî OpenAI-compatible API key for the AI command bar. When empty, AI endpoints return 503 and the frontend shows a degraded-mode indicator; manual editing still works.
- `LLM_BASE_URL` ‚Äî OpenAI-compatible base URL. Default `https://api.openai.com/v1`. Set to OpenRouter or a self-hosted proxy to swap providers.
- `LLM_MODEL` ‚Äî Model name passed to the chat-completions API. Default `gpt-4o-mini`.
- `LLM_REQUEST_TIMEOUT` ‚Äî Hard timeout for the LLM HTTP call in seconds. Default `15`. Prevents a hung provider from holding a worker.
- `LLM_MAX_COMMAND_CHARS` ‚Äî Cap on the user's command string before sending to the LLM. Default `500`.
- `LLM_RATE_LIMIT_PER_HOUR` ‚Äî Per-user fixed-window rate limit on `POST /api/ai/schedules/<date>/command/`. Default `100`. Counter lives in `CACHES['default']` under `ai_cmd_rl:<user_id>`; a 429 is returned when exceeded. **Set `REDIS_URL` when AI is enabled ‚Äî Redis is the production-shaped default** and gives atomic, cross-worker counters (Redis `INCR`). The Django system check `ai.E001` (see `backend/ai/checks.py`) fires whenever `LLM_API_KEY` is set and `CACHES['default']` is an ineffective cache backend for rate limiting (LocMem / Dummy are per-process; FileBased is non-atomic across workers), **independent of `DEBUG`**, so an AI-enabled deploy cannot boot on such a backend.
- `LLM_DRAFT_MODEL` ‚Äî Model used for draft generation (`POST /api/ai/schedules/<date>/generate-draft/`). Default `gpt-4o`. Heavier than `LLM_MODEL` because drafts shape a whole day from history (PRD ¬ß15.3).
- `LLM_DRAFT_RATE_LIMIT_PER_HOUR` ‚Äî Independent fixed-window counter (`ai_draft_rl`) for the draft endpoint. Default `10`. Same `REDIS_URL`-when-AI-enabled requirement as `LLM_RATE_LIMIT_PER_HOUR` ‚Äî covered by the same `ai.E001` system check.
- `LLM_HISTORY_DAYS` ‚Äî Number of past schedules included in the draft context. Default `7` (PRD ¬ß6.2). Only schedules with `status` in `{active, reviewed}` are included ‚Äî `draft`-status days are excluded so the AI doesn't train on its own unreviewed output.
- `LLM_DRAFT_CAPTURE_PROMPT_PATH` ‚Äî Test-only. **Overwrites the target file on every `generate-draft` request when set.** Set to a writable file path (e.g. `/tmp/draft_prompt_test7.txt`) to capture the rendered LLM draft user_message. Used by `frontend/scripts/playwright/draft-prompt-history-suffix.mjs` to verify the prompt content end-to-end without patching `service.py` by hand. Default empty (disables capture). The Django system check `ai.E002` (see `backend/ai/checks.py`) blocks startup when this is set while `DEBUG=False`, so a misconfigured production deploy fails loudly instead of silently spooling the user's full schedule history (PII) to disk.
- `LLM_CHAT_RATE_LIMIT_PER_HOUR` ‚Äî Independent fixed-window counter (`ai_chat_rl`) for the multi-turn chat endpoint (`POST /api/ai/schedules/<date>/chat/`, feature 0007). Default `60`. Same `REDIS_URL`-when-AI-enabled requirement as the other AI buckets ‚Äî covered by the same `ai.E001` system check (references `ai_cmd_rl`, `ai_draft_rl`, `ai_chat_rl`).
- `LLM_CHAT_MAX_TURNS` ‚Äî Hard cap on `len(messages[])` per chat request. Default `20`. Protects against runaway transcript size if the client never resets the thread.
- `LLM_CHAT_MAX_TOTAL_CHARS` ‚Äî Cap on the sum of `content` across all chat messages in one request. Default `LLM_MAX_COMMAND_CHARS √ó 4` (= `2000` with the default 500). Caps prompt cost regardless of how many short turns the user piles on.
- `LLM_CHAT_MAX_ASK_CHARS` ‚Äî Schema cap on the model's clarifying-question field. Default `300`. Truncated/runaway responses surface as a 502 parse error, not a UI overflow.
- `LLM_MAX_EXPLANATION_CHARS` ‚Äî Canonical schema cap on the model's `explanation` field. Default `300`. Shared by both the one-shot `validate_response_envelope` (legacy) and the chat envelope (feature 0007). Tightened from the previous hardcoded 500 so both surfaces stay aligned; the system prompt asks for one-sentence explanations.
- `ANALYTICS_STREAK_THRESHOLD` ‚Äî Per-day completion ratio required for a day to count toward the streak. Default `0.8`. Must be a float in `[0.0, 1.0]`; out-of-range values raise `ValueError` at import time so a misconfigured deploy fails loudly instead of silently producing `streak=0` forever.
- `ANALYTICS_STREAK_WINDOW_DAYS` ‚Äî Hard cap on how far back the streak query walks. Default `30`. Must be a positive int; `<= 0` raises `ValueError` at import time. Prevents `O(account-age)` scans on long-lived accounts.
- `CALDAV_ENCRYPTION_KEY` ‚Äî Fernet symmetric key (URL-safe base64, 32 bytes) used to encrypt CalDAV passwords at rest in `calendar_sync.CalDAVAccount.password_encrypted`. Generate with `python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"`. No insecure default ‚Äî empty value is tolerated only in `DEBUG=True`; `calendar_sync.E001` blocks `DEBUG=False` startup when unset.
- `CALDAV_DEFAULT_BASE_URL` ‚Äî Default base URL for the CalDAV connect form. Default `https://caldav.icloud.com/`. Echoed back by `GET /api/calendar/account/` (`default_base_url` field) so the frontend never hardcodes the value; changing it ships without a frontend rebuild.
- `CALDAV_REQUEST_TIMEOUT` ‚Äî Hard timeout (seconds) on every CalDAV HTTP call. Default `10`. Same risk profile as `LLM_REQUEST_TIMEOUT` ‚Äî prevents a hung iCloud connection from holding a worker.
- `CALDAV_CACHE_TTL_SECONDS` ‚Äî Per-(user, date) event-list cache window (`calendar_sync/cache.py`). Default `300`. The cache key embeds `account.updated_at.isoformat()`, so credential rotation invalidates entries across all dates without any explicit `cache.delete` call. Non-shared cache backends (LocMem / FileBased / Dummy) are flagged by `calendar_sync.W001` as a perf concern (not a correctness one ‚Äî versioned keys keep correctness intact regardless of backend). Setting `REDIS_URL` (required once AI is enabled) moves these keys into shared Redis and silences `calendar_sync.W001`.
- `TODOIST_ENCRYPTION_KEY` ‚Äî Fernet symmetric key (URL-safe base64, 32 bytes) used to encrypt the Todoist personal API token at rest in `todoist_sync.TodoistAccount.token_encrypted`. Generate with `python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"`. No insecure default ‚Äî empty value is tolerated only in `DEBUG=True`; `todoist_sync.E001` blocks `DEBUG=False` startup when unset or malformed. (Feature 0020.)
- `TODOIST_BASE_URL` ‚Äî API host for the Todoist REST client. Default `https://api.todoist.com/api/v1` ‚Äî the **real API host**, not the `developer.todoist.com` docs host (requests there 404/return HTML and surface as a confusing `TodoistProviderError` 502). Analog of `CALDAV_DEFAULT_BASE_URL` ‚Äî `service.py` builds every request URL from this base, so a self-hosted proxy ships without a code change.
- `TODOIST_REQUEST_TIMEOUT` ‚Äî Hard timeout (seconds) on every Todoist HTTP call. Default `10`. Same risk profile as `CALDAV_REQUEST_TIMEOUT` ‚Äî prevents a hung connection from holding a worker. `<= 0` raises `ValueError` at import time.
- `TODOIST_CACHE_TTL_SECONDS` ‚Äî Per-(user, date) task-list cache window (`todoist_sync/cache.py`). Default `300`. The cache key embeds `account.updated_at.isoformat()`, so token rotation invalidates entries across all dates without any explicit `cache.delete` call. Non-shared cache backends are flagged by `todoist_sync.W001` as a perf concern (not correctness ‚Äî versioned keys keep correctness intact). Setting `REDIS_URL` moves the `todoist_tasks:*` keys into shared Redis (they share `CACHES['default']`) and silences `todoist_sync.W001`. `<= 0` raises `ValueError` at import time.

**CalDAV key rotation note (feature 0011)**: rotating `CALDAV_ENCRYPTION_KEY` invalidates all stored `CalDAVAccount.password_encrypted` ciphertext. The next `GET /api/calendar/events/<date>/` per affected user surfaces a `500 {"errors": {"detail": "Calendar service is misconfigured..."}}` (the view's explicit `ImproperlyConfigured` catch ‚Äî see `backend/calendar_sync/views.py:events`). Recovery options: (1) keep the old key accessible and re-encrypt rows with a one-off management command that loads with `Fernet(old)` and saves with `Fernet(new)`; (2) easier for small deployments ‚Äî instruct all users to reconnect via Settings (`DELETE /api/calendar/account/` then `POST` with current credentials). The `calendar_sync.E001` startup check only validates the *current* key ‚Äî it cannot detect mismatched ciphertext from a previous key.

**Todoist token rotation note (features 0020, 0021)**: rotating `TODOIST_ENCRYPTION_KEY` invalidates all stored `TodoistAccount.token_encrypted` ciphertext. The next `GET /api/todoist/tasks/<date>/` (or `POST /api/todoist/tasks/<id>/complete/`) per affected user surfaces a `500 {"errors": {"detail": "Todoist service is misconfigured. Contact the administrator."}}` (the view's explicit `ImproperlyConfigured` catch ‚Äî mirrors the CalDAV one). Recovery: re-encrypt rows with `Fernet(old)`‚Üí`Fernet(new)`, or instruct users to reconnect via Settings (`DELETE /api/todoist/account/` then `POST` with the current token). The `todoist_sync.E001` startup check only validates the *current* key. **Write surface (feature 0021)**: `POST /api/todoist/tasks/<id>/complete/` completes a task ‚Äî it is the **second** token-decryption call site (after `fetch_tasks_for_date`), still service-boundary-owned (`complete_task` lives in `service.py`; views pass the `TodoistAccount` through and never touch plaintext). A successful complete invalidates the user's task cache by bumping `account.updated_at` (`cache.invalidate_tasks` ‚Üí plain `account.save()`; same `auto_now` footgun as `set_token`). `GET /api/todoist/tasks/<date>/?refresh=1` force-refetches bypassing the read cache (still re-warms it). **No AI provider-egress**: task text is never fed into draft/command/chat prompts ‚Äî display + complete only, never sent to the LLM provider.

**Performance note (draft history)**: PR #15 added `select_related("daily_review")` to the draft-history query in `ai_generate_draft` (`backend/ai/views.py`). `prompts.build_draft_user_message` reads `past.daily_review` for each past schedule to embed the per-day completion ratio in the LLM prompt; without the `select_related`, that's an N+1 ‚Äî one extra query per past day. This fix predates the multi-turn chat feature (Phase-6 leftover) and applies regardless of whether chat is enabled.

**Privacy note**: User commands submitted via the AI endpoint are logged verbatim to the `AIInteraction` table for audit purposes (PRD ¬ß6.5). Users should avoid entering sensitive data (passwords, API keys) in command prompts.

**Privacy note (chat ‚Äî feature 0007)**: The multi-turn chat endpoint (`POST /api/ai/schedules/<date>/chat/`) re-sends the **full prior client-supplied transcript** to the LLM provider on every turn ‚Äî not just the latest message. This is a strictly larger provider-egress surface than the one-shot endpoint, even though the per-row DB log only stores the latest user turn (plus a `transcript_sha256` fingerprint of the whole transcript, see `backend/ai/views.py:_build_chat_audit_response`). Users should clear the thread (Clear button or page reload ‚Äî thread state is in-memory per tab, no server-side persistence) before discussing anything sensitive. The thread is also auto-reset on date navigation (see `frontend/src/composables/useChat.ts:setActiveDate`).

**Privacy note (chat ‚Äî wire-format detail)**: prior client-supplied turns are NOT forwarded to the provider under the privileged `assistant` role. They are flattened into a single user-role message prefixed with an "Untrusted prior transcript" header (see `backend/ai/service.py:run_chat` and `backend/ai/prompts.py:serialise_prior_turns`) so a tampered client cannot inject privileged assistant pre-commitments that bias the model toward destructive actions. The security invariant is regression-tested by `backend/tests/test_ai_service_chat.py::TestUntrustedTranscript::test_assistant_role_never_forwarded`.
